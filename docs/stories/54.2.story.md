## Story

As the research-supervisor agent,
I want the Gemini CLI installed via npm, authenticated via OAuth, and wrapped in a research-oriented shell script,
So that I can dispatch research queries with `scripts/gemini-research.sh` and receive structured JSON responses grounded in web search and project context.

## Background

This replaces the original 54.2 which set up the Python-based `agent-deep-research` tool. The new approach:

- **Installation:** `npm install -g @google/gemini-cli` (or `npx` for zero-install)
- **Authentication:** One-time OAuth flow via browser → token cached and auto-refreshed
- **No API key required** — eliminates `GEMINI_DEEP_RESEARCH_API_KEY`
- **No Python dependency** — eliminates `uv`, `pip`, `google-genai` SDK
- **No `_tools/` directory** — CLI is installed globally via npm

The wrapper script (`scripts/gemini-research.sh`) provides:
- Model selection based on `--depth` parameter (Pro vs Flash)
- Context injection via `--include-directories` and stdin piping
- JSON output parsing with `jq`
- Artifact storage in `_bmad-output/research-reports/`

## Acceptance Criteria

**Given** a machine with Node.js/npm installed
**When** a developer follows the setup documentation
**Then** the Gemini CLI is available via `gemini -p "test" --output-format json`

**Given** a first-time setup
**When** the user runs `gemini` interactively
**Then** the browser opens for Google Account OAuth sign-in, and the token is cached locally

**Given** a cached OAuth token
**When** `gemini -p "Hello" --output-format json` is run
**Then** it returns valid JSON with a `.response` field (no API key needed)

**Given** the wrapper script at `scripts/gemini-research.sh`
**When** invoked with `--depth quick --query "test query"`
**Then** it uses `gemini-2.5-flash` model and returns results to stdout

**Given** the wrapper script
**When** invoked with `--depth deep --query "architecture analysis"`
**Then** it uses `gemini-2.5-pro` model and stores artifacts in `_bmad-output/research-reports/YYYYMMDD-HHMMSS-<slug>/`

**Given** the artifact output directory
**When** a research query completes
**Then** it contains: `response.json` (raw CLI output), `report.md` (extracted response text), `stderr.log` (CLI stderr)

**Given** the `_bmad-output/research-reports/` directory
**When** `git status` is run
**Then** individual report directories are gitignored, but `budget.json` is tracked

**Given** the project root
**When** checked for a `GEMINI.md` file
**Then** a minimal research-focused system prompt exists that instructs Gemini to use GoogleSearch grounding, produce structured reports with citations, and follow the three-layer output format

**Given** the setup documentation
**When** a developer reads it
**Then** it covers: npm install, OAuth flow, verification command, wrapper script usage, and troubleshooting (token refresh, model availability fallback)

## Technical Notes

- Pin Gemini CLI version in wrapper script: `npx @google/gemini-cli@0.32.1` to prevent breaking changes.
- The wrapper script should detect if `gemini` is not installed and suggest installation.
- The wrapper script should detect OAuth failure (exit code 1 + auth error in stderr) and provide re-authentication instructions.
- `jq` is required for JSON parsing — script should check for it.
- `GEMINI.md` in project root is loaded automatically by Gemini CLI (like `CLAUDE.md` for Claude Code).
- The `GEMINI.md` file should be concise — just enough to orient Gemini on the project and desired output format.
- Consider `brew install gemini-cli` if a Homebrew formula becomes available in the future.
- No `_tools/` directory needed — remove the `_tools/` gitignore entry if it was added for the old approach.
