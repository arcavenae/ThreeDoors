# Story 42.3: Input Size Limits for YAML and JSONL Readers

## Status: In Review

## Epic

Epic 42: Application Security Hardening

## References

- Security Audit: `_bmad-output/planning-artifacts/security-audit-application.md` (MEDIUM-1, MEDIUM-2, LOW-3)
- Positive example: MCP transport at `internal/mcp/transport.go:37` sets explicit 1MB buffer

## Story

As a user,
I want ThreeDoors to enforce explicit size limits on file reads and scanner buffers,
So that a corrupted or maliciously crafted file cannot cause excessive memory allocation or silent data loss.

## Background

All YAML unmarshaling uses `yaml.Unmarshal()` without file size validation. While `gopkg.in/yaml.v3` has built-in YAML bomb protection, there's no check preventing a multi-GB task file from being read into memory.

Most `bufio.NewScanner()` usages rely on the default 64KB buffer. Lines exceeding this are silently skipped (scanner returns `bufio.ErrTooLong`). The MCP transport correctly sets an explicit 1MB buffer, but other JSONL readers do not.

## Acceptance Criteria

**AC-1: YAML file size limit**
**Given** ThreeDoors reads a YAML file (tasks.yaml, config.yaml, etc.)
**When** the file size exceeds 10MB
**Then** it returns an error before attempting to read the file into memory
**And** the error message indicates the file exceeds the size limit

**AC-2: JSONL scanner buffer limits**
**Given** ThreeDoors reads a JSONL file (sessions.jsonl, sync.log, WAL, metrics)
**When** a `bufio.Scanner` is created
**Then** it has an explicit buffer size set via `scanner.Buffer()`
**And** the buffer max is appropriate for the data type (e.g., 1MB for session entries)

**AC-3: Graceful handling of oversized lines**
**Given** a JSONL file contains a line exceeding the scanner buffer
**When** the scanner encounters it
**Then** the line is skipped with a logged warning
**And** scanning continues for subsequent lines (no crash, no silent stop)

## Technical Notes

### YAML size check locations
- `internal/adapters/textfile/file_manager.go:80` — task YAML

- `internal/dispatch/queue.go:58` — dev queue YAML
- `internal/core/values_config.go:39` — values config
- `internal/core/onboarding.go:28,46` — onboarding config
- `internal/core/dedup_store.go:47` — dedup decisions
- `internal/core/sync_state.go:90` — sync state
- `internal/core/inline_hints_config.go:61,97` — hints config

### JSONL scanner locations (need explicit buffer)
- `internal/core/pattern_analyzer.go:152,185` — sessions JSONL
- `internal/core/sync_log.go:127` — sync log JSONL
- `internal/core/wal_provider.go:345` — WAL JSONL
- `internal/core/metrics/reader.go:76` — metrics JSONL
- `internal/core/completion_counter.go:55` — completed tasks

### Implementation approach
- Create a helper `ReadFileWithLimit(path string, maxBytes int64) ([]byte, error)` in `internal/core/`
- Use `os.Stat()` to check size before `os.ReadFile()`
- For JSONL scanners, add `scanner.Buffer(make([]byte, 0, 64*1024), 1024*1024)` (1MB max, matching MCP)
- Consider a shared constant `const MaxYAMLFileSize = 10 * 1024 * 1024`

## Tasks

### Task 1: Create ReadFileWithLimit helper
- `ReadFileWithLimit(path string, maxBytes int64) ([]byte, error)` in appropriate location
- Returns clear error if file exceeds limit

### Task 2: Apply size limits to YAML reads
- Replace `os.ReadFile()` calls with `ReadFileWithLimit()` for all YAML files
- Use 10MB limit for task files, 1MB for config files

### Task 3: Add explicit buffer limits to JSONL scanners
- Add `scanner.Buffer()` calls to all 6 JSONL scanner locations
- Use 1MB max buffer (consistent with MCP transport)

### Task 4: Tests
- Test ReadFileWithLimit with oversized file (returns error)
- Test ReadFileWithLimit with normal file (succeeds)
- Test scanner buffer handles long lines gracefully
- Run `go test -race ./internal/core/...`
