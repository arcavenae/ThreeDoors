## Story

As the research-supervisor agent,
I want pre-defined context bundles and a prompt template that tailor each research query with relevant project files via Gemini CLI's `--include-directories` and stdin piping,
So that Gemini receives focused, project-specific grounding that improves result relevance within token budgets.

## Background

The context packaging strategy is largely preserved from the original design — the 8 curated bundles, 60KB budget, keyword auto-detection, and priority shedding order all remain. What changes is the delivery mechanism:

| Original (agent-deep-research) | New (Gemini CLI) |
|---|---|
| `--context /tmp/research-context/` flag | `--include-directories docs/architecture,docs/prd` flag |
| `--context-extensions md,go` for file types | Piped stdin: `cat bundle.md \| gemini -p "..."` |
| Files copied to temp directory | Files referenced in-place or assembled to stdin |

The Gemini CLI also supports `GEMINI_SYSTEM_MD` for custom system prompts, and `GEMINI.md` files for project context — both new mechanisms not available in the original design.

## Acceptance Criteria

**Given** the research-supervisor agent definition
**When** it describes context assembly
**Then** it lists all 8 bundle names with their file mappings and estimated sizes

**Given** a research request with `context=architecture,tui`
**When** the research-supervisor assembles context
**Then** it includes core (always) + architecture + tui bundles, passed via `--include-directories` or stdin piping

**Given** a research request with no explicit context
**When** the research-supervisor assembles context
**Then** it auto-detects bundles from query keywords (e.g., "bubbletea" → tui, "provider" → tasks)

**Given** assembled context exceeding 60KB
**When** the budget is exceeded
**Then** the agent follows the priority shedding order: drop code-sample → truncate stories to headers+ACs → drop prd → irreducible minimum is core+architecture+decisions (~31KB)

**Given** any research query
**When** the prompt is formulated
**Then** it wraps the user's question in the standard prompt template with: project context paragraph, grounding instructions (use GoogleSearch), the question, and output requirements (executive summary ≤300 words, detailed findings, relevance section, rejected approaches, ranked recommendations)

**Given** the prompt template
**When** used across multiple queries
**Then** it is stored as a reusable template in the agent definition, not hardcoded per-query

**Given** the `GEMINI.md` file (from Story 54.2)
**When** the Gemini CLI is invoked
**Then** it automatically loads project context from `GEMINI.md`, supplementing the per-query context bundles

## Technical Notes

- Context assembly happens in the research-supervisor agent's reasoning — no Go code needed.
- Two context injection mechanisms available:
  1. `--include-directories path1,path2` — Gemini CLI reads files from specified directories
  2. Stdin piping: `cat assembled-context.md | gemini -p "Given the context above, research: <query>"` — for fine-grained bundle assembly
- Prefer `--include-directories` for directory-aligned bundles (architecture, prd, decisions).
- Prefer stdin piping for assembled bundles that combine files from multiple directories (core, code-sample).
- The `GEMINI_SYSTEM_MD` env var can point to a custom system prompt for research-specific instructions.
- Keyword-to-bundle mapping is a heuristic documented in the agent definition — it doesn't need to be a code implementation.
- The prompt template should be embedded in the agent definition file as a clearly marked section.
