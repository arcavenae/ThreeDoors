# Story 5.1: macOS Binary Signing, Notarization & Distribution

## Story Overview

**Epic:** Epic 5 - macOS Distribution & Packaging
**Status:** Ready for Implementation
**Independence:** This story is independent of the feature story pipeline and can be merged at any time.
**Prerequisites:** Apple Developer Program membership ($99/year) with Developer ID certificates

---

## Problem Statement

The Go binary built in CI is unsigned and untrusted. When macOS users download `threedoors-darwin-*` from GitHub Releases, Gatekeeper quarantines the file:

```
"threedoors-darwin-arm64" cannot be opened because it is from an unidentified developer.
```

Users must manually navigate to System Settings > Privacy & Security to allow execution. This is a terrible first-run experience that:
- Undermines user trust ("Is this safe?")
- Creates unnecessary friction before the user ever sees the product
- Looks unprofessional compared to properly signed macOS software

---

## Acceptance Criteria

### 5.1a: CI Code Signing

1. CI pipeline signs `threedoors-darwin-arm64` and `threedoors-darwin-amd64` with a Developer ID Application certificate
2. Signing uses a macOS runner (codesign requires macOS)
3. Certificate (.p12), certificate password, Apple ID, app-specific password, and Team ID stored as GitHub Actions encrypted secrets
4. `codesign --verify --deep --strict` passes on signed binaries
5. Signing step is skipped gracefully in forks or when secrets are not available (with informative log message)
6. Unsigned linux/amd64 binary is unaffected

### 5.1b: Apple Notarization

1. Signed darwin binaries are submitted to Apple's notarization service via `xcrun notarytool submit`
2. CI waits for notarization to complete (`xcrun notarytool wait`)
3. Notarization ticket is stapled to the binary (`xcrun stapler staple`)
4. `spctl --assess --type execute` passes on notarized binaries
5. Notarization failures produce clear CI error messages with the notarization log URL

### 5.1c: Homebrew Tap

1. A new GitHub repository `arcaven/homebrew-tap` exists with a valid Homebrew formula
2. Formula file: `Formula/threedoors.rb`
3. Formula downloads the correct signed+notarized binary based on architecture:
   - `on_arm`: downloads `threedoors-darwin-arm64`
   - `on_intel`: downloads `threedoors-darwin-amd64`
4. Formula includes SHA256 checksums for both binaries
5. `brew install arcaven/tap/threedoors` works end-to-end
6. `brew upgrade arcaven/tap/threedoors` picks up new versions
7. Formula includes a `test` block that runs `threedoors health` or `threedoors --version`
8. CI automatically updates the formula (SHA256 + version) on each release

### 5.1d: pkg Installer

1. CI generates a signed `.pkg` installer using `pkgbuild`/`productbuild`
2. The `.pkg` installs `threedoors` to `/usr/local/bin/`
3. The `.pkg` is signed with a Developer ID Installer certificate
4. The `.pkg` is notarized with Apple
5. The `.pkg` is uploaded to GitHub Releases alongside raw binaries
6. Double-clicking the `.pkg` launches macOS Installer.app

## Implementation Guidance

### CI Workflow Changes

The existing `alpha-release` job in `.github/workflows/ci.yml` needs to be restructured:

```
quality-gate (ubuntu-latest)
  └── build-binaries (ubuntu-latest) - cross-compile all platforms
       └── sign-and-notarize (macos-latest) - sign darwin binaries, notarize, create pkg
            └── release (ubuntu-latest) - create GitHub release with all artifacts
            └── update-homebrew (ubuntu-latest) - update tap formula
```

**Key decision:** The signing step MUST run on a macOS runner because `codesign` and `xcrun notarytool` are macOS-only tools.

### Required GitHub Actions Secrets

| Secret Name | Description |
|-------------|-------------|
| `APPLE_CERTIFICATE_P12` | Base64-encoded Developer ID Application certificate (.p12) |
| `APPLE_CERTIFICATE_PASSWORD` | Password for the .p12 file |
| `APPLE_INSTALLER_CERTIFICATE_P12` | Base64-encoded Developer ID Installer certificate (.p12) |
| `APPLE_INSTALLER_CERTIFICATE_PASSWORD` | Password for the installer .p12 file |
| `APPLE_ID` | Apple ID email for notarization |
| `APPLE_ID_PASSWORD` | App-specific password for notarization |
| `APPLE_TEAM_ID` | Apple Developer Team ID |
| `HOMEBREW_TAP_TOKEN` | GitHub PAT with repo access to `arcaven/homebrew-tap` |

### Code Signing Steps (in CI)

```bash
# 1. Import certificate to temporary keychain
security create-keychain -p "" build.keychain
security default-keychain -s build.keychain
security unlock-keychain -p "" build.keychain
echo "$APPLE_CERTIFICATE_P12" | base64 --decode > cert.p12
security import cert.p12 -k build.keychain -P "$APPLE_CERTIFICATE_PASSWORD" -T /usr/bin/codesign
security set-key-partition-list -S apple-tool:,apple: -s -k "" build.keychain

# 2. Sign the binaries
codesign --force --options runtime --sign "Developer ID Application: <Team Name> (<Team ID>)" threedoors-darwin-arm64
codesign --force --options runtime --sign "Developer ID Application: <Team Name> (<Team ID>)" threedoors-darwin-amd64

# 3. Verify
codesign --verify --deep --strict threedoors-darwin-arm64
codesign --verify --deep --strict threedoors-darwin-amd64
```

### Notarization Steps (in CI)

```bash
# 1. Zip the binary (notarytool requires zip/dmg/pkg)
zip threedoors-darwin-arm64.zip threedoors-darwin-arm64

# 2. Submit for notarization
xcrun notarytool submit threedoors-darwin-arm64.zip \
  --apple-id "$APPLE_ID" \
  --password "$APPLE_ID_PASSWORD" \
  --team-id "$APPLE_TEAM_ID" \
  --wait

# 3. Staple (for pkg; standalone binaries can't be stapled but are still notarized)
# Note: Standalone Mach-O binaries cannot be stapled. The notarization ticket
# is checked online by Gatekeeper. For the pkg installer, stapling works.
```

### Homebrew Formula Template

```ruby
class Threedoors < Formula
  desc "Three Doors - radical task management that shows only 3 tasks at a time"
  homepage "https://github.com/arcaven/ThreeDoors"
  version "VERSION"
  license "MIT"

  on_arm do
    url "https://github.com/arcaven/ThreeDoors/releases/download/TAG/threedoors-darwin-arm64"
    sha256 "SHA256_ARM64"
  end

  on_intel do
    url "https://github.com/arcaven/ThreeDoors/releases/download/TAG/threedoors-darwin-amd64"
    sha256 "SHA256_AMD64"
  end

  def install
    binary_name = Hardware::CPU.arm? ? "threedoors-darwin-arm64" : "threedoors-darwin-amd64"
    bin.install binary_name => "threedoors"
  end

  test do
    assert_match "ThreeDoors", shell_output("#{bin}/threedoors --version 2>&1", 0)
  end
end
```

### pkg Installer Steps (in CI)

```bash
# 1. Create component package
mkdir -p pkg-root/usr/local/bin
cp threedoors-darwin-arm64 pkg-root/usr/local/bin/threedoors
chmod +x pkg-root/usr/local/bin/threedoors

pkgbuild --root pkg-root \
  --identifier com.arcaven.threedoors \
  --version "$VERSION" \
  --install-location / \
  --sign "Developer ID Installer: <Team Name> (<Team ID>)" \
  threedoors.pkg

# 2. Notarize the pkg
xcrun notarytool submit threedoors.pkg \
  --apple-id "$APPLE_ID" \
  --password "$APPLE_ID_PASSWORD" \
  --team-id "$APPLE_TEAM_ID" \
  --wait

# 3. Staple the notarization ticket
xcrun stapler staple threedoors.pkg
```

- **Apple Developer Program membership** enrolled and active
- **Developer ID Application certificate** and **Developer ID Installer certificate** generated from developer.apple.com
- **App-specific password** generated for the Apple ID (for notarization)
- **GitHub Actions secrets** configured on the upstream repo with all values listed above
- **A `arcaven/homebrew-tap` repository** created (can be empty initially)
- **GitHub PAT** with `repo` scope for pushing to the homebrew-tap repo
- Access to review the existing CI workflow in `.github/workflows/ci.yml`

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Apple Developer Program not enrolled | Must be done before implementation; $99/year cost |
| Certificate expiry (annual) | Document renewal process; CI will fail loudly if expired |
| Notarization service downtime | Add retry logic with backoff; notarization is async |
| macOS runner costs (GitHub Actions) | macOS runners use more minutes; signing step is fast (~2 min) |
| Standalone Mach-O binaries can't be stapled | Gatekeeper checks notarization online; works without stapling for binaries |
| Homebrew formula review delays | Using a tap (not homebrew-core) so no review needed |

---

## Out of Scope

- Windows code signing (Authenticode) - no Windows target currently
- Linux package formats (.deb, .rpm, snap, flatpak) - deferred
- Auto-update mechanism within the binary
- Universal binary (fat binary combining arm64 + amd64) - keep separate for smaller downloads
- Homebrew Cask (for GUI apps; ThreeDoors is a CLI tool, formula is correct)
