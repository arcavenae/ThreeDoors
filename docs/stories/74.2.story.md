## Story

As a project maintainer,
I want a CI workflow that validates story references in PR commits,
So that all code changes are traceable to stories and the commit message format is mechanically enforced.

## Background

CLAUDE.md requires commit messages to reference stories: `feat: <description> (Story X.Y)` or `fix: <description> (#NNN)`. This is currently a prompt-level instruction that agents follow voluntarily. A CI workflow can mechanically verify this, transforming a behavioral expectation into a technical gate.

Per Q-C-012: Start with WARN (CI check reports but doesn't block merge), upgrade to BLOCK after validating the check catches real violations without false positives.

## Acceptance Criteria

**AC1: GitHub Actions workflow validates commit messages**
**Given** a PR is opened or updated
**When** the CI scope-check workflow runs
**Then** it inspects all commit messages in the PR
**And** checks each for a story reference matching `(Story X.Y)` or `(#NNN)`
**And** reports pass/fail per commit

**AC2: Workflow runs as a warning (non-blocking)**
**Given** the scope-check workflow is configured
**When** it detects a commit without a story reference
**Then** it reports a CI check status of "neutral" or "success" with annotations (WARNING, not FAILURE)
**And** the PR can still merge (check does not block)
**And** the check output clearly shows which commits lack references

**AC3: Allowed exceptions documented**
**Given** some commits legitimately lack story references
**When** the workflow validates commits
**Then** it allows exceptions for:
  - Merge commits (automated)
  - Commits with `(#NNN)` format (infrastructure issue references)
  - Commits starting with `Merge` or `Revert`
  - Dependabot/Renovate commits (automated dependency updates)

**AC4: Upgrade path to blocking documented**
**Given** the workflow starts as a warning
**When** it has been running for N PRs without false positives
**Then** documentation explains how to upgrade to blocking:
  - Change the CI status from "neutral" to "failure"
  - Add the check to required status checks in branch protection

**AC5: Story file existence check (optional enhancement)**
**Given** a commit references `(Story X.Y)`
**When** the workflow validates the reference
**Then** it optionally checks that `docs/stories/X.Y.story.md` exists in the repo
**And** warns (not blocks) if the story file is missing

## Tasks

### Task 1: Create `.github/workflows/scope-check.yml`
GitHub Actions workflow that:
- Triggers on `pull_request` events (opened, synchronize)
- Lists all commits in the PR via `gh api`
- Validates each commit message against the story reference pattern
- Reports results as check annotations
- Exits 0 (warn mode, never blocks)

### Task 2: Define commit message regex patterns
Document the accepted patterns:
- `(Story X.Y)` where X and Y are digits
- `(#NNN)` where NNN is one or more digits
- Exception patterns for merge/revert/bot commits

### Task 3: Add story file existence check
Optional check: if `(Story X.Y)` is referenced, verify the file exists.

### Task 4: Document the workflow
Add documentation explaining the scope-check, its warn-vs-block modes, and the upgrade path.

### Task 5: Test with existing PRs
Validate the workflow against recent merged PRs to ensure it would have flagged the right violations.

## Dev Notes

- Use `${{ github.event.pull_request.commits }}` to get commit count, then fetch via API
- For the regex: `\(Story [0-9]+\.[0-9]+\)` or `\(#[0-9]+\)`
- Dependabot commits have author `dependabot[bot]`, Renovate has `renovate[bot]`
- The workflow should be lightweight (< 30s) to avoid slowing CI
- Consider: should the check also validate the commit prefix (feat/fix/docs/chore)?
- Marvel portability: This is a generic CI check pattern — works in any repo with story-driven development

## Upgrade Path (Warn -> Block)

To upgrade this check from warning to blocking after validation:

1. **Change exit code:** In `.github/workflows/scope-check.yml`, replace `exit 0` at the end of the validation step with:
   ```bash
   if [ "$VIOLATIONS" -gt 0 ]; then
     exit 1
   fi
   ```
2. **Add to required status checks:** In GitHub repo Settings > Branches > Branch protection rules for `main`, add "Commit Message Validation" to the required status checks list.
3. **Validation criteria:** Upgrade when the check has run on 20+ PRs with zero false positives. Monitor the `::warning` annotations in CI logs to confirm all flagged commits are genuine violations.
