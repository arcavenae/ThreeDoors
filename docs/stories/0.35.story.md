# Story 0.35: Stabilize Command Mode Input Position

## Status: Done (PR #401)

## Story

As a user typing commands in command mode (: key),
I want the text input to stay in a fixed position at the bottom of the terminal,
So that the input doesn't bounce around as the suggestions list changes height.

## Background

When entering command mode, the suggestions list has variable height depending on how many commands match the current prefix. As the user types and the filter narrows, suggestions disappear and the text input jumps upward. This creates visual instability and makes command mode feel janky.

## Acceptance Criteria

**AC-1: Fixed-height suggestions area**
**Given** the user is in command mode
**When** the suggestions list changes (more or fewer matches)
**Then** the suggestions area maintains a fixed height equal to `len(commandRegistry)` lines, with blank padding for unused slots

**AC-2: Bottom-anchored input**
**Given** the user is in command mode (or any search view state)
**When** the terminal height is known
**Then** the text input is pushed to the bottom of the terminal using vertical padding between the header and content

**AC-3: Graceful fallback without height**
**Given** the terminal height is not yet known (height = 0)
**When** the search view renders
**Then** the view renders compactly without padding (no regression)

**AC-4: Race detector passes**
**Given** the test suite
**When** `go test -race ./internal/tui/...` runs
**Then** all tests pass with zero race conditions

## Technical Notes

- Added `height` field and `SetHeight()` method to `SearchView`
- `View()` calculates header + content + footer lines, inserts padding to fill remaining terminal height
- Command mode suggestion area padded to `len(commandRegistry)` lines regardless of match count
- `newSearchView()` in `main_model.go` sets height from `m.height`
- `WindowSizeMsg` handler updates height on live resize
