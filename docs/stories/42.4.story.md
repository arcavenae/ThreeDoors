## Story

As a user,
I want ThreeDoors to warn me if my config file contains API tokens and is world-readable,
So that I am alerted to credential exposure before it can be exploited.

## Background

The config file `config.yaml` can contain API tokens for Jira, Todoist, and other providers. The sample config generator includes commented-out `api_token` fields. While environment variables are recommended, users may store tokens directly in the config file. Combined with permissive file permissions (addressed in Story 42.1), this creates a credential exposure risk. Some backend configs already use `yaml:"-"` to prevent serialization, but this is not consistent.

## Acceptance Criteria

**AC-1: Startup permission warning**
**Given** ThreeDoors starts and `config.yaml` exists
**When** the file has permissions more permissive than `0o600` AND contains non-empty token/credential fields
**Then** a warning is logged to stderr indicating the config file is world-readable and contains credentials
**And** the warning includes instructions to fix: `chmod 600 ~/.threedoors/config.yaml`

**AC-2: yaml:"-" on sensitive fields**
**Given** any struct field that holds API tokens or credentials
**When** serialized to YAML
**Then** the field uses `yaml:"-"` tag to prevent accidental serialization
**And** credentials are only read from config, never written back

**AC-3: Sample config documentation**
**Given** the sample config generator produces commented-out token fields
**When** a user reads the sample config
**Then** each token field has a comment recommending environment variable usage
**And** the comment includes the specific env var name (e.g., `JIRA_API_TOKEN`)

## Technical Notes

- Check `internal/core/provider_config.go:225-309` for sample config generator
- Check all backend config structs for token fields needing `yaml:"-"`
- `internal/intelligence/llm/backend.go:38` already uses `yaml:"-"` — verify others follow
- The permission warning should fire only when BOTH conditions are true: permissive perms AND non-empty tokens
- Use `os.Stat()` to check file mode bits for world-readable check
- Warning to stderr (not TUI flash) since this is a startup security check

## Tasks

### Task 1: Audit and fix yaml tags on credential fields
- Search all config structs for token/key/secret fields
- Add `yaml:"-"` where missing
- Verify credentials are only loaded, never re-serialized

### Task 2: Add startup credential exposure check
- After loading config, check if file is world-readable
- If world-readable AND contains non-empty credential fields, warn to stderr
- Include remediation instructions in warning

### Task 3: Improve sample config documentation
- Ensure all token fields in sample config have env var recommendation comments
- Verify env var names match what the code actually reads

### Task 4: Tests
- Test warning fires when config is world-readable with tokens
- Test no warning when config is 0o600
- Test no warning when config is world-readable but has no tokens
- Test yaml:"-" prevents token serialization
