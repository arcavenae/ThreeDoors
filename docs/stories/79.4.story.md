# Story 79.4: Task Lifecycle API — completeTask, skipTask, snoozeTask

**Epic:** 79 — ThreeDoors Daemon/Server Mode
**Status:** Not Started
**Priority:** P1
**Estimate:** S (thin wrappers over existing TaskProvider methods)

## Context

Beyond door selection, Stream Deck users need buttons for task lifecycle actions: complete the current task, skip it, or snooze it. These map directly to existing TaskProvider methods and TUI actions.

This story adds the task lifecycle JSON-RPC methods to the daemon.

## Acceptance Criteria

- [ ] `task.complete` method marks the currently selected task as complete via `TaskProvider.MarkComplete()`
- [ ] `task.skip` method returns the selected task to the pool and refreshes doors
- [ ] `task.snooze` method defers the task (sets snooze timestamp) and refreshes doors
- [ ] `task.snooze` accepts optional `{"duration": "30m"}` parameter (default: 1 hour)
- [ ] All three methods return error if no door is currently selected
- [ ] `task.complete` triggers session tracker recording (`RecordDoorSelection`, completion metrics)
- [ ] `task.getActive` returns the currently selected task (or null if none)
- [ ] `task.stats` returns current session statistics (tasks completed, doors viewed, session duration)
- [ ] After `task.complete`, doors auto-refresh to fill the completed slot
- [ ] After `task.skip` or `task.snooze`, doors auto-refresh
- [ ] Tests: complete flow, skip flow, snooze with duration, no selection error, stats accuracy

## Technical Notes

### Session Tracking

The daemon maintains its own `SessionTracker` instance. When a task is completed via the daemon, it records the same metrics as the TUI would — ensuring session analytics are consistent regardless of interaction mode.

### Snooze Implementation

Snooze sets a `SnoozedUntil` timestamp on the task and calls `SaveTask`. The task is excluded from door selection until the snooze expires. This uses existing task model fields — no schema changes needed.

### Auto-Refresh

After lifecycle actions, the daemon automatically calls `doors.refresh` internally and emits a `doors.changed` event (Story 79.5) so connected clients update their displays.
