## Story

As a potential user discovering ThreeDoors,
I want a professional documentation site hosted on GitHub Pages,
So that I can learn about the project without cloning the repository.

## Background

ThreeDoors has a comprehensive 1,173-line user guide (`docs/user-guide.md`) and a 745-line README, but no web-hosted documentation. Publishing via GitHub Pages with MkDocs Material makes the guide discoverable via search engines and provides a professional landing page.

The planning artifact (D-169) established MkDocs + Material for MkDocs as the tool, with a separate `docs-site/` directory to avoid mixing user-facing and internal docs (the existing `docs/` has 262 story files, 33 ADRs, and 17 PRD files).

## Acceptance Criteria

**Given** the project has no GitHub Pages infrastructure
**When** this story is complete
**Then** `docs-site/mkdocs.yml` exists with Material theme, site metadata, and full navigation stub

**Given** the mkdocs.yml configuration
**When** the site is built locally with `mkdocs build --strict`
**Then** it builds successfully with zero warnings

**Given** the docs-site directory
**When** `docs-site/requirements-docs.txt` is examined
**Then** it pins MkDocs and Material for MkDocs versions

**Given** the landing page
**When** `docs-site/docs/index.md` is viewed
**Then** it contains project branding, feature highlights, install quick-reference, and "Get Started" link

**Given** the philosophy page
**When** `docs-site/docs/philosophy.md` is viewed
**Then** it contains content extracted from SOUL.md explaining the Three Doors philosophy

**Given** the GitHub Actions workflow
**When** `.github/workflows/docs.yml` is examined
**Then** it builds and deploys MkDocs on push to `main` when `docs-site/**` files change
**And** it uses path filtering to avoid unnecessary builds

**Given** the Makefile
**When** `make docs` is run
**Then** it builds the MkDocs site locally
**And** `make docs-serve` starts a local preview server

**Given** the deployed site
**When** visiting the site
**Then** dark/light mode toggle works
**And** client-side search functions (even with minimal content)

**Given** the existing `docs/user-guide.md`
**When** this story is implemented
**Then** the existing file is NOT modified or moved

## Technical Notes

- Use `docs-site/` directory (separate from `docs/`) per planning artifact decision
- Material theme with dark-first palette (slate + deep purple/amber)
- Features: navigation.tabs, search.suggest, content.code.copy, etc.
- The `.github/workflows/docs.yml` file will require **manual merge by the project owner** due to OAuth workflow scope limitation (merge-queue's token lacks `workflow` scope)
- See planning artifact for starter `mkdocs.yml` and workflow configuration

## Tasks

### Task 1: Create docs-site directory structure
- Create `docs-site/` with `docs/` subdirectory
- Create `docs-site/requirements-docs.txt` with pinned MkDocs + Material versions

### Task 2: Create mkdocs.yml
- Material theme with dark-first palette (slate scheme, deep purple primary, amber accent)
- Enable: navigation.tabs, navigation.sections, navigation.expand, navigation.top, search.suggest, search.highlight, content.code.copy, content.tabs.link
- Markdown extensions: admonition, pymdownx.details, pymdownx.superfences, pymdownx.tabbed, pymdownx.highlight, pymdownx.keys, toc with permalinks
- Full navigation structure (most pages as stubs initially)

### Task 3: Create landing page (index.md)
- Extract hero content from README.md
- Feature highlights, install quick-reference, "Get Started" link

### Task 4: Create philosophy page
- Extract from SOUL.md — the "why three doors" narrative

### Task 5: Create GitHub Actions workflow
- `.github/workflows/docs.yml` — builds on push to main with path filtering
- Uses `actions/setup-python`, caches pip, runs `mkdocs gh-deploy --force`

### Task 6: Add Makefile targets
- `make docs` — builds MkDocs site
- `make docs-serve` — starts local preview server

### Task 7: Verify build
- Run `mkdocs build --strict` locally
- Verify dark/light toggle, search, navigation
