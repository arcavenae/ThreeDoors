# Story 0.20: CI Churn Reduction — Branch Protection & Merge Queue Optimization

## Status: Done (PR #260)

**Epic:** 0 — Infrastructure & Process
**Depends on:** None (Phase 1-2), cool-platypus research PR (Phase 3)

## User Story

As a development team running multiple parallel agents,
I want CI to run efficiently without cascading reruns,
So that PRs merge quickly without wasting 5-10x CI runs per PR.

## Problem Statement

With branch protection requiring "Require branches to be up to date before merging", every PR merge invalidates all other open PRs. With N open PRs, this creates O(n^2) CI runs. 6 open PRs = ~36 CI runs instead of 6.

## Acceptance Criteria — Phase 1 (Immediate)

- **AC1:** Branch protection rule "Require branches to be up to date before merging" is DISABLED
- **AC2:** Branch protection still requires: CI green, no force push to main
- **AC3:** Document the change and rationale in an ADR at `docs/adr/`
- **AC4:** merge-queue agent updated to stop calling update-branch API (no longer needed)
- **AC5:** pr-shepherd agent updated — rebase-on-main is no longer mandatory before merge

## Acceptance Criteria — Phase 2 (Quick Win)

- **AC6:** CI workflows use path-based triggers — docs-only PRs skip `make test`/`lint`/`build`
- **AC7:** Path filter covers: `*.md`, `docs/**`, `_bmad-output/**`, `ROADMAP.md`, `CHANGELOG.md`
- **AC8:** Code PRs (`*.go`, `go.mod`, `go.sum`, `Makefile`, `.github/**`) still run full CI
- **AC9:** A lightweight "docs-check" job runs for docs PRs (just validates markdown or passes trivially)

### Key Concepts

- **Merge Queue**: GitHub feature that batches PRs, creates temporary merge branches, runs CI once on the combined result, auto-merges if green, ejects failing PRs
- **Branch Protection Cascade**: When "require up-to-date" is enabled, merging PR A invalidates PRs B,C,D,E — each must update and re-run CI, creating O(n^2) runs
- **Path-based CI triggers**: Workflow `paths` filter that only runs jobs when matching files change — docs-only PRs skip code CI entirely
- **Speculative merging**: Merge queue tests PR #3 assuming #1 and #2 pass, running in parallel instead of sequentially
- **Merge trains**: Alternative to merge queue where PRs are tested incrementally on top of each other
