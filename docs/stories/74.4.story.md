## Story

As a platform operator,
I want a machine-readable `.dfcp.yaml` file that declares a repo's Dark Factory Control Pattern profile,
So that tooling can automatically determine governance rules, protected paths, and autonomy levels for any repo.

## Background

The DFCP (Dark Factory Control Pattern) defines two profiles:
- **Golden repo** (e.g., ThreeDoors): Human-gated governance files, AI handles implementation
- **Dark factory repo**: Full AI autonomy, CI-only gates, no human review required

Currently these rules are encoded in CODEOWNERS, branch protection, agent definitions, and documentation — spread across multiple files with no single source of truth. A `.dfcp.yaml` file consolidates this into a machine-readable declaration that tooling (CI, daemon, agents) can read.

## Acceptance Criteria

**AC1: .dfcp.yaml schema defined**
**Given** the DFCP pattern has two profiles
**When** the schema is designed
**Then** it includes:
  ```yaml
  dfcp:
    version: "1.0"
    profile: golden  # golden | factory | custom
    autonomy:
      default: L3    # Default autonomy level for this repo
      max: L3        # Maximum allowed autonomy level
    governance:
      require_human_review:
        - SOUL.md
        - CLAUDE.md
        - .claude/**
        - ROADMAP.md
        - docs/prd/epic-list.md
        - docs/prd/epics-and-stories.md
        - docs/decisions/BOARD.md
        - .github/**
        - agents/**
      require_story_reference: true
      require_provenance: true  # Mandatory provenance tags on AI work
      require_signed_commits: true
    ci:
      scope_check: warn  # warn | block | off
      required_checks:
        - "Quality Gate"
        - "Docker E2E Tests"
    factory:
      enabled: false  # Whether this repo can be used as a dark factory source
      variant_budget: null  # No budget (golden repo)
      repo_visibility: private
  ```

**AC2: .dfcp.yaml created for ThreeDoors**
**Given** the schema is defined
**When** `.dfcp.yaml` is created in the repo root
**Then** it accurately reflects ThreeDoors' current governance rules
**And** all protected paths match what's in CODEOWNERS (Story 74.1)

**AC3: Schema documentation created**
**Given** the schema is defined
**When** documentation is written
**Then** `docs/architecture/dfcp-schema.md` explains:
  - Every field and its purpose
  - The three profiles (golden, factory, custom)
  - How tooling should read and use the file
  - How to create a factory profile (for future dark factory repos)

**AC4: Validation script exists**
**Given** `.dfcp.yaml` declares governance rules
**When** a validation script runs
**Then** it checks:
  - YAML is valid and conforms to schema
  - Protected paths in `.dfcp.yaml` match CODEOWNERS entries
  - Autonomy levels are consistent with profile
  - Required CI checks match actual workflow names

**AC5: Agent awareness documented**
**Given** agents need to respect DFCP rules
**When** the agent integration is designed
**Then** documentation explains how agents should read `.dfcp.yaml`:
  - merge-queue: Check autonomy level, enforce governance gates
  - workers: Read protected paths, avoid modifying without justification
  - arch-watchdog: Validate changes against DFCP profile

## Dev Notes

- Keep the schema extensible — new fields will be added as dark factory features land
- The validation script should be runnable in CI and locally
- Consider: should `.dfcp.yaml` be CODEOWNERS-protected itself? Yes — it controls governance.
- Marvel portability: .dfcp.yaml IS the portable governance declaration — it's what makes repos self-describing for the platform
