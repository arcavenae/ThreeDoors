# Story 79.2: JSON-RPC 2.0 Protocol Layer and API Contract

**Epic:** 79 — ThreeDoors Daemon/Server Mode
**Status:** Not Started
**Priority:** P1
**Estimate:** M (protocol implementation, request routing, error handling)

## Context

The daemon (79.1) needs a well-defined protocol for client communication. JSON-RPC 2.0 is the recommended choice: it's simple, language-agnostic, widely supported, and already used by the MCP server (`internal/mcp/protocol.go`).

This story implements the JSON-RPC 2.0 codec (request parsing, response formatting, error codes) and the method dispatch router. Specific API methods are added in Stories 79.3 and 79.4.

**Research:** [Stream Deck Research](../../_bmad-output/planning-artifacts/streamdeck-research.md) — Section 6 (Unix socket with JSON-RPC 2.0).

## Acceptance Criteria

- [ ] JSON-RPC 2.0 request/response codec implemented in `internal/daemon/` package
- [ ] Support for single requests and batch requests per JSON-RPC 2.0 spec
- [ ] Standard error codes: Parse Error (-32700), Invalid Request (-32600), Method Not Found (-32601), Invalid Params (-32602), Internal Error (-32603)
- [ ] Application-specific error codes defined (e.g., -1 task not found, -2 no active session)
- [ ] Method dispatcher routes requests to registered handlers
- [ ] `system.listMethods` introspection method returns available API methods
- [ ] `system.version` returns daemon version string
- [ ] Request/response logging at debug level (configurable via `--verbose` flag)
- [ ] OpenAPI-style API contract documented in `docs/architecture/daemon-api.md`
- [ ] Connection-level read timeout (30s idle) to prevent resource leaks
- [ ] Tests: codec round-trip, batch requests, error formatting, method dispatch, unknown methods

```
└── server_test.go
```

### Relationship to MCP Protocol

`internal/mcp/protocol.go` already has JSON-RPC types (`Request`, `Response`, `Error`). Evaluate whether to:
- **Reuse** the MCP types directly (risk: coupling daemon to MCP)
- **Copy** the types into `internal/daemon/` (risk: duplication)
- **Extract** shared JSON-RPC types into `internal/jsonrpc/` (cleanest, but may be over-engineering for now)

Recommendation: Start with a lean copy in `internal/daemon/`. If a third consumer appears, extract to shared package.

### Framing

Unix socket connections use newline-delimited JSON (one JSON-RPC message per line). This is simpler than HTTP framing and works well for persistent connections.
