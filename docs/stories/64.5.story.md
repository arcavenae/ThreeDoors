## Refined by Spike (ADR-0034)

With Git-based sync (ADR-0034), the offline queue is largely handled by Git itself — local commits accumulate while offline and push when connectivity returns. This story focuses on the application-level awareness of offline state and the reconciliation logic that runs after `git pull --rebase` brings in remote changes.

## User Story

As a ThreeDoors user who sometimes works offline,
I want my changes to queue locally and sync when connectivity is restored,
So that I can use ThreeDoors on a plane or in areas without internet.

## Acceptance Criteria

- AC1: Local changes commit to the Git sync repo immediately (stage + commit) regardless of connectivity — Git's local commits ARE the offline queue
- AC2: Commits persist across app restarts (Git guarantees this)
- AC3: When connectivity is restored (circuit breaker transitions to half-open → closed), queued commits push automatically via the debounced sync trigger
- AC4: After `git pull --rebase`, reconciliation runs the three-way merge (Story 64.4) to merge remote changes with local state
- AC5: If rebase produces conflicts, the custom merge driver (Story 64.4) resolves them — unresolvable conflicts abort rebase and queue for manual resolution
- AC6: TUI status bar shows: online/offline indicator, unpushed commit count (queue depth), last successful sync time
- AC7: `threedoors sync status` shows: connectivity state, unpushed commits count, oldest unpushed commit timestamp, remote HEAD vs local HEAD divergence
- AC8: Offline mode is fully transparent — all ThreeDoors features work normally; only sync is deferred
- AC9: Circuit breaker (existing from Epic 43) detects offline state — trips after 5 failed push attempts, probes periodically
- AC10: Periodic `git gc` runs after every 100 commits to prevent repo bloat during extended offline periods
- AC11: Unit tests for: offline detection via circuit breaker, commit queuing, reconnect-and-push cycle, reconciliation after pull, gc trigger

## Technical Notes

- No separate offline queue file needed — Git's local commit log serves this purpose (simpler than the WAL approach)
- The WAL (Epic 21) remains unchanged — it handles provider sync failures, not cross-computer sync
- Connectivity detection: attempt `git ls-remote` (lightweight) on circuit breaker probe interval
- Reconciliation flow: `git pull --rebase` → if conflict → run merge driver → if unresolvable → `git rebase --abort` → log conflict → retry on next cycle
- Extended offline periods may accumulate many commits — `git gc` keeps repo size manageable
