## User Story

As a ThreeDoors user who sometimes works offline,
I want my changes to queue locally and sync when connectivity is restored,
So that I can use ThreeDoors on a plane or in areas without internet.

## Acceptance Criteria

- AC1: All local changes are written to an offline queue when sync transport is unavailable
- AC2: Offline queue persists across app restarts (file-based, not in-memory)
- AC3: When connectivity is restored, queued changes sync automatically (FIFO order)
- AC4: Reconciliation merges remote changes received during offline period with local queue
- AC5: Queue size is bounded (configurable, default 1000 operations) with oldest-first eviction
- AC6: TUI status bar shows offline/online indicator and queue depth
- AC7: `threedoors sync status` shows queue depth, oldest queued change, and estimated sync time
- AC8: Offline mode is transparent — all ThreeDoors features work normally without connectivity
- AC9: Reconciliation uses the conflict resolution strategy from Story 64.4
- AC10: Unit tests for queue persistence, bounded eviction, and reconciliation scenarios

## Technical Notes

- Offline queue extends the WAL (Epic 21) pattern — append-only log of pending operations
- Reconciliation is the most complex part — must handle: local creates + remote creates, local edits + remote edits, local deletes + remote restores
- Consider using the existing circuit breaker (Epic 43) to detect offline state
- Queue should use JSONL format for consistency with session logs
