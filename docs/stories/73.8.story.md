## Story

As a multiclaude operator,
I want the git safety hook to only block fetch/pull/rebase/merge in worker worktrees,
So that persistent agents, the supervisor, and the multiclaude CLI itself can perform git operations essential to their roles.

## Background

Story 73.3 (PR #840) introduced a PreToolUse hook (`scripts/hooks/git-safety.sh`) that blocks dangerous git commands to prevent INC-002-style worktree corruption. However, the hook was implemented with universal scope — it blocks ALL Claude agents, not just workers in multiclaude-managed worktrees.

This over-broad scope caused two distinct failures:

### Failure Mode 1: Persistent Agents Blocked
Persistent agents (merge-queue, pr-shepherd, supervisor) that share the main checkout could not run `git fetch`, `git pull`, or `git rebase`. These operations are essential to their core functions:
- **merge-queue** needs `git fetch` to check branch status and update PR branches
- **pr-shepherd** needs `git rebase` to keep PR branches up-to-date with main
- **supervisor** needs `git fetch` to verify repository state

### Failure Mode 2: multiclaude CLI Blocked (Catch-22)
The multiclaude CLI itself internally runs `git fetch` when creating worker worktrees. With the hook active, the supervisor could not even spawn workers to fix the problem — creating a catch-22 where the system could not self-heal.

### Temporary Fix
A worktree path detection block was added to `git-safety.sh` on main that exits 0 (allow) for paths outside the multiclaude worktree directory (`~/.multiclaude/wts/`). This story formalizes and tests that fix.

### Design Principle
The INC-002 protections must remain absolute for workers — they should NEVER run fetch/pull/rebase/merge in their isolated worktrees. But persistent agents operating from the main checkout need these operations for their core responsibilities. The worktree path (`~/.multiclaude/wts/<repo>/<worker-name>/`) is a reliable discriminator.

## Acceptance Criteria

**AC1: Worker worktrees remain fully protected**
**Given** the git safety hook is active in a worker worktree (path matches `~/.multiclaude/wts/`)
**When** a worker attempts `git fetch`, `git pull`, `git rebase`, or `git merge`
**Then** the hook exits with code 2 (blocked) with clear error messages
**And** all other 73.3 protections (unsigned commits, push to main, Co-Authored-By) remain enforced

**AC2: Main checkout agents are exempt from sync restrictions**
**Given** the git safety hook is active in the main checkout (path does NOT match `~/.multiclaude/wts/`)
**When** a persistent agent or supervisor runs `git fetch`, `git pull`, `git rebase`, or `git merge`
**Then** the hook exits with code 0 (allowed)

**AC3: Universal protections remain universal**
**Given** the git safety hook is active in ANY context (worker or main checkout)
**When** an agent attempts unsigned commits (`--no-gpg-sign`), push to main, or Co-Authored-By trailers
**Then** the hook exits with code 2 (blocked)
**And** these protections are NOT affected by worktree detection

**AC4: multiclaude CLI operations unblocked**
**Given** the multiclaude CLI runs `git fetch` internally when creating worktrees
**When** the hook intercepts this command from the main checkout
**Then** the hook allows it (exit 0)
**And** worker creation proceeds normally

**AC5: Tests cover both paths**
**Given** the test suite for git-safety.sh
**When** tests run
**Then** there are explicit test cases for:
  - Worker worktree context: fetch/pull/rebase/merge blocked
  - Main checkout context: fetch/pull/rebase/merge allowed
  - Universal blocks enforced in both contexts

## Dev Notes

- The worktree path pattern is `~/.multiclaude/wts/<repo-name>/<worker-name>/`
- Detection can use `$PWD` or `$CLAUDE_PROJECT_DIR` — prefer `$PWD` as it's always available
- The detection must handle both expanded (`/Users/skippy/.multiclaude/wts/...`) and unexpanded (`~/.multiclaude/wts/...`) paths
- Keep the check simple and fast — this hook runs on EVERY Bash tool call
- Do NOT weaken universal protections (unsigned commits, push to main, Co-Authored-By)

## Implementation Notes

> [decision] 2026-03-30 — Used CLAUDE_PROJECT_DIR (with PWD fallback) for worktree detection.
> Alternative considered: parsing git worktree list output — rejected because it requires a git command (slow, could fail) and the path pattern is a reliable, fast discriminator.

> [decision] 2026-03-30 — Scoped only fetch/pull/rebase/merge to worktree detection. Universal protections (unsigned commits, push to main, Co-Authored-By) remain enforced everywhere.
> Rationale: even persistent agents should never push to main or make unsigned commits.

> [observation] 2026-03-30 — The pattern `*/.multiclaude/wts/*` correctly matches worker paths while excluding `/tmp/wts/` and other false positives.
