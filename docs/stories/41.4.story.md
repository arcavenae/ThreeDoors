## Story

As a user,
I want consistent scrolling behavior across all scrollable views,
So that my muscle memory works everywhere and mouse wheel scrolling is always available.

## Background

After Story 41.3 migrates the help view to viewport, two custom scroll implementations remain: synclog_view.go and keybinding_overlay.go. This story completes the viewport migration and creates a shared factory function for ThreeDoors-specific viewport defaults, ensuring consistency across all three migrated views.

## Acceptance Criteria

**Given** the sync log view is opened
**When** log content exceeds the visible area
**Then** a bubbles/viewport is used for scrollable content display
**And** mouse wheel scrolling works

**Given** the keybinding overlay is opened
**When** content exceeds the visible area
**Then** a bubbles/viewport is used for scrollable content display
**And** mouse wheel scrolling works

**Given** a `NewScrollableView()` factory function exists
**When** any view needs scrollable content
**Then** the factory creates a pre-configured viewport with ThreeDoors defaults

**Given** all three migrated views (help, synclog, keybinding overlay)
**When** the user scrolls in any of them
**Then** scrolling behavior is consistent (same speed, same key bindings)

**Given** golden file tests for synclog and keybinding overlay
**When** tests run after the migration
**Then** all golden file tests pass (updated for viewport rendering differences)

## Technical Notes

- Reuse viewport patterns established in Story 41.3
- Create `NewScrollableView(width, height int) viewport.Model` factory in a shared location
- Factory configures: mouse wheel enabled, consistent key bindings, ThreeDoors style
- Migrate synclog_view.go: replace custom scroll offset with viewport
- Migrate keybinding_overlay.go: replace custom scroll logic with viewport
- Remove all custom scroll state from both views
- Verify mouse wheel works in both views
- Total custom scroll implementations after this story: 0
