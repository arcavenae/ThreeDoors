# Story 0.38: Epic Number Registry Accuracy & Enforcement

## Status: Done (PR #485)

**Epic:** 0 — Infrastructure & Process
**Priority:** P1
**Depends On:** D-112 (epic number reservation registry in BOARD.md)

## Context

Decision D-112 established an Epic Number Registry in BOARD.md to prevent parallel workers from claiming the same epic number (the Epic 39 collision incident). The registry exists but has data accuracy issues and incomplete coverage. This story corrects the registry and strengthens the enforcement rules.

See: `_bmad-output/planning-artifacts/epic-39-governance-sync-investigation.md`

## Problem

The current registry in BOARD.md has:
1. **Wrong epic mapping** — Epic 42 listed as "Door-Like Doors" but should be "Application Security Hardening" (Door-Like Doors is Epic 48)
2. **Missing epics** — Epic 48 (Door-Like Doors) not listed; Epic 39, 40 not listed (recently completed)
3. **Epic 51 listed as "next available"** but is allocated to SLAES
4. **Rules don't reference project-watchdog** as the MUTEX for epic numbers (per standing orders in MEMORY.md)

## Acceptance Criteria

- [ ] All active epics (39-51) are accurately listed in the Epic Number Registry
- [ ] Epic numbers match their feature names as shown in ROADMAP.md
- [ ] "Next available" row shows the correct next unallocated number
- [ ] Rules explicitly name project-watchdog as the MUTEX for epic number allocation
- [ ] Rules state that workers and /plan-work agents NEVER self-assign epic numbers
- [ ] ROADMAP.md updated to include Story 0.38

## Tasks

1. Fix Epic 42 mapping (Security Hardening, not Door-Like Doors)
2. Add missing epics: 39, 40, 48, 51
3. Update "next available" to correct number (52)
4. Update rules to reference project-watchdog as MUTEX
5. Add rule about workers/plan-work not self-assigning
