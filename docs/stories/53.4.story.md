## Story

As a remote Claude Code session,
I want MCP tools to send messages, dispatch workers, and acknowledge messages,
So that I can actively collaborate with the multiclaude supervisor, not just observe.

## Background

Story 53.3 establishes the MCP bridge server with read-only tools. This story adds **write tools** that modify multiclaude state: sending messages, dispatching workers, and acknowledging messages.

Write tools require careful input validation to prevent command injection — tool parameters are passed to shell commands, and a malicious or malformed parameter could execute arbitrary code. The feasibility analysis identified this as the primary security risk for the MCP bridge.

## Acceptance Criteria

**Given** the MCP tool `multiclaude_message_send`
**When** invoked with `recipient` and `body` parameters
**Then** it sends the message via `multiclaude message send` and returns the message ID

**Given** the MCP tool `multiclaude_message_ack`
**When** invoked with a `message_id` parameter
**Then** it acknowledges the message via `multiclaude message ack` and returns success/failure

**Given** the MCP tool `multiclaude_worker_create`
**When** invoked with a `task` description parameter
**Then** it dispatches a new worker via `multiclaude worker create` and returns the worker name

**Given** any write tool
**When** invoked with a parameter containing shell metacharacters (`;`, `|`, `$()`, backticks, `&&`, `||`, `>`, `<`)
**Then** the tool rejects the input with a descriptive error before executing any command

**Given** any write tool
**When** invoked with a `recipient` parameter that is not a known agent name pattern (alphanumeric + hyphens)
**Then** the tool rejects the input with a validation error

**Given** the `multiclaude_worker_create` tool
**When** invoked with a task description longer than 500 characters
**Then** the tool rejects the input with a length validation error

**Given** any write tool
**When** the underlying `multiclaude` command fails
**Then** the tool returns a structured MCP error with the failure reason, without exposing internal paths

## Technical Notes

- Extends the MCP bridge server from Story 53.3. Same binary, new tools.
- **Critical: No shell interpolation.** All parameters must be passed as arguments to `exec.Command()`, never interpolated into shell strings. Use `exec.Command("multiclaude", "message", "send", recipient, body)` — never `exec.Command("sh", "-c", fmt.Sprintf("multiclaude message send %s '%s'", ...))`.
- Input validation layer:
  - `recipient`: Must match `^[a-zA-Z0-9_-]+$` (alphanumeric, hyphens, underscores only)
  - `message_id`: Must match `^msg-[a-f0-9-]+$` (UUID format)
  - `body`/`task`: Max 2000 characters, no null bytes
  - All parameters: Reject shell metacharacters as defense-in-depth (even though exec.Command doesn't use a shell)
- Rate limiting: Optional but recommended — max 10 write operations per minute per tool.
- Write tools should log all invocations to stderr for audit trail.
