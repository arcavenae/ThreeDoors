# Story 39.13: Unified Door Key Indicator Toggle

**Epic:** 39 — Keybinding Display System
**Status:** Done (PR #407)
**Priority:** P1
**Estimate:** M (3-5 hours)
**Depends on:** 39.4 (Done), 39.9 (Done), 39.10 (Done)

## Context

The current implementation has two separate toggle mechanisms for key discoverability: the `h`-toggled keybinding bar (bottom footer) and the `:hints`-toggled inline door hints (rendered on door frames). This creates redundancy — when both are ON, door selection keys appear in two places — and cognitive overhead from having two toggles for one concept.

This story unifies them: `h` toggles inline door key indicators `[a]` `[w]` `[d]` rendered directly on the doors. The bottom bar is removed from the doors view. Non-door views retain the keybinding bar since they lack spatial anchors for inline hints.

**Course correction:** Overrides D-125 (separate mechanisms) and D-126 (auto-fade). See `_bmad-output/planning-artifacts/door-key-indicators-course-correction.md`.

## Acceptance Criteria

### AC1: `h` toggles door key indicators
- [ ] Pressing `h` in doors view toggles inline hints `[a]` `[w]` `[d]` on the door frames
- [ ] Toggle state persisted to config.yaml as `show_key_hints` (new field name)
- [ ] Backward compatibility: reads `show_keybinding_bar` if `show_key_hints` is absent
- [ ] Defaults to ON for new users (preserves D-092)
- [ ] `h` key guard: suppressed during text input (`isTextInputActive()`), suppressed when overlay visible

### AC2: No bottom bar in doors view
- [ ] When `show_key_hints` is ON: door hints visible on doors, no keybinding bar at bottom of doors view
- [ ] When `show_key_hints` is OFF: no hints anywhere — not on doors, not at bottom
- [ ] Non-door views (detail, search, mood, health, etc.) continue to show the keybinding bar as before, controlled by the same `show_key_hints` config

### AC3: Door hint rendering (reuses 39.9/39.10 infrastructure)
- [ ] `[a]` at base of left door, `[w]` at base of middle door, `[d]` at base of right door
- [ ] Selection-state awareness preserved: selected door hint brightens, unselected dims
- [ ] Action hints (`[s] re-roll`, `[n] add task`, `[enter] confirm`) shown below doors when hints ON
- [ ] All theme compatibility maintained (Classic, Modern, Sci-Fi, etc.)

### AC4: Config unification
- [ ] Remove `show_inline_hints`, `inline_hints_session_count`, `inline_hints_fade_threshold` config fields
- [ ] Remove `show_keybinding_bar` field (replaced by `show_key_hints`)
- [ ] Migration: on config load, if `show_key_hints` is absent, read `show_keybinding_bar` value as default
- [ ] `:hints` command now toggles the same `show_key_hints` config (alias behavior)

### AC5: Remove auto-fade mechanism
- [ ] Remove session counter increment logic from `NewMainModel()`
- [ ] Remove `ResolveInlineHintState()` or simplify to just read config bool
- [ ] Remove fade mode rendering (ANSI 240 "extra dim" state)
- [ ] Remove flash message "Key hints hidden — type :hints to bring them back"

### AC6: Tests
- [ ] Table-driven tests for `h` toggle behavior in doors view
- [ ] Table-driven tests for keybinding bar still showing in non-door views
- [ ] Config migration test: old `show_keybinding_bar` field read correctly
- [ ] Config persistence test: `show_key_hints` saved and loaded
- [ ] Race detector passes: `go test -race ./internal/tui/...`
