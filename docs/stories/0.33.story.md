# Story 0.33: Stable Search Result Ordering

**Epic:** 0 — Infrastructure & Process
**Priority:** P1
**Status:** Ready
**Issue:** [#334](https://github.com/arcaven/ThreeDoors/issues/334)

## Context

Search results in the TUI jump/dance on each keystroke because `filterTasks()` calls `GetAllTasks()`, which iterates a Go `map[string]*Task` with randomized iteration order. The same query produces a different result order on every call.

## Acceptance Criteria

- [ ] `filterTasks()` returns results in a stable, deterministic order (sorted alphabetically by `Task.Text`)
- [ ] Typing the same query twice produces the same result list order
- [ ] Adding/removing a character from the query does not reshuffle unaffected results
- [ ] Race detector passes: `go test -race ./internal/tui/...`
- [ ] Unit test verifies stable ordering across multiple `filterTasks()` calls

## Tasks

- [ ] Add `slices.SortFunc` call in `filterTasks()` after match collection, sorting by `Task.Text`
- [ ] Add test in `search_view_test.go` confirming deterministic result order
- [ ] Run `make lint` and `make test` — zero warnings, all passing
- [ ] Run `go test -race ./internal/tui/...` — passes

## Technical Notes

- Party mode artifact: `_bmad-output/planning-artifacts/issue-334-search-jumping-triage.md`
- Decision: Sort in `filterTasks()` (Option A) — minimal blast radius, idiomatic Go
- Rejected: Sorting in `GetAllTasks()` (unnecessary cost for other callers), caching (premature optimization), ordered data structure (over-engineered)
- Decision recorded in `docs/decisions/BOARD.md`
