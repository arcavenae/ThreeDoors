## Story

As a user,
I want doors to cast a visible gradient shadow instead of the current imperceptible 1-character shadow,
So that doors appear to "lift off" the terminal background with proper spatial depth.

## Background

The current shadow system (`shadow.go`) uses a single `▐` character in `#585858`. On dark terminals (bg `#000000` to `#1a1a2e`), the contrast ratio is ~2.7:1 — below the perceptual threshold. The brain needs ~4:1 contrast and ~2-3 chars width to read "shadow cast by object."

The shadow is also a post-processor — `ApplyShadow()` splits rendered output by `\n` and appends characters. This limits it to right/bottom additions. This story refactors shadow rendering into the theme `Render()` pipeline and implements a multi-column gradient shadow with proper contrast.

Width-adaptive behavior ensures graceful degradation:
- Width < minWidth + 2: no shadow
- Width >= minWidth + 2: 1-column shadow
- Width >= minWidth + 4: 2-column gradient shadow
- Width >= minWidth + 6: 3-column gradient shadow

## Acceptance Criteria

**Given** a door rendered at width >= minWidth + 4
**When** the shadow is displayed
**Then** the right-edge shadow uses 2 columns with gradient colors: `ShadowNear` (near) → `ShadowFar` (far)
**And** the gradient creates a visible fade-out effect

**Given** a door rendered at width >= minWidth + 6
**When** the shadow is displayed on a wide terminal
**Then** the right-edge shadow uses 3 columns (▓▒░ or bg-colored spaces) for maximum depth

**Given** a door rendered at width >= minWidth + 2 but < minWidth + 4
**When** the shadow is displayed
**Then** the right-edge shadow uses 1 column (current minimal behavior)

**Given** a door rendered at width < minWidth + 2
**When** the door is displayed
**Then** no shadow is rendered (graceful degradation preserved)

**Given** the bottom shadow row
**When** rendered below the door
**Then** the bottom shadow is offset 1 character to the right
**And** uses the same gradient colors as the right-edge shadow
**And** spans the width of the door

**Given** a selected door with crack-of-light enabled
**When** the shadow is rendered
**Then** the crack-of-light rendering (from `crack.go`) integrates cleanly with the new shadow system
**And** the selected door gets a wider shadow (3-col) for enhanced depth

**Given** each of the 8 themes
**When** shadow colors are applied
**Then** each theme uses its own `ShadowNear` and `ShadowFar` colors from the research artifact palette

**Given** the refactored shadow rendering
**When** `ApplyShadow()` in `shadow.go` is replaced
**Then** shadow rendering is integrated into each theme's `Render()` function
**And** `shadow.go` is simplified or deprecated

**Given** existing golden file tests
**When** the shadow changes are applied
**Then** golden files are regenerated to match the new rendering

## Technical Notes

- **Depends on Story 56.1** — needs `ShadowNear` and `ShadowFar` fields in `ThemeColors`
- Refactor shadow from post-processor (`ApplyShadow`) to inline rendering in each theme's `Render()` row loop
- Each row appends shadow columns after the right border character
- Shadow columns use background-colored spaces (not foreground shade chars) for better visual quality on TrueColor terminals, with `▓▒░` fallback for ANSI
- Bottom shadow: rendered as part of the theme output, not bolted on after
- `ApplyShadowWithCrack` integration: crack-of-light (░ char) becomes the first shadow column for selected doors, gradient continues after it
- Width threshold logic in each `Render()` function — calculate shadow columns based on `width - minWidth`
- ~150 LOC total across themes + shadow.go refactor
- Must pass `go test -race ./internal/tui/...`
