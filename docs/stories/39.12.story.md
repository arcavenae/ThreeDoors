# Story 39.12: Auto-Fade After N Sessions

## Status: Not Started

**Epic:** 39 — Keybinding Display System
**Priority:** P2
**Estimate:** S (1-2 hours)
**Depends On:** 39.9, 39.10

---

## User Story

As a returning user, I want inline key hints to gracefully fade away after I've used the app several times, so that the UI becomes cleaner as I learn the key bindings.

## Context

Inline hints are onboarding scaffolding, not permanent UI chrome. After a configurable number of sessions (default 5), hints should auto-disable with a graceful visual transition. This prevents the "training wheels" from becoming visual clutter for experienced users.

The auto-fade mechanism is built into the infrastructure (Story 39.9) but this story handles the graceful dimming transition and the user-facing notification.

Design reference: [Default Tooltips Party Mode](../../_bmad-output/planning-artifacts/default-tooltips-mode-party-mode.md)

## Acceptance Criteria

### AC-1: Graceful Dimming at Session N-1
- [ ] One session before the fade threshold: all inline hints render in extra-dim style (ANSI 240)
- [ ] The visual dimming is noticeable but not dramatic — hints are still readable
- [ ] This creates a subconscious "these are fading" cue before they disappear

### AC-2: Auto-Disable at Session N
- [ ] At the fade threshold session: `show_inline_hints` set to `false` in config
- [ ] One-time flash message displayed: "Key hints hidden — type :hints to bring them back"
- [ ] Flash message uses the standard flash message system (existing pattern)
- [ ] Session counter stops incrementing once threshold is reached

### AC-3: Re-Enable via `:hints on`
- [ ] `:hints on` re-enables hints AND resets session counter to 0
- [ ] User can re-learn with a fresh fade cycle
- [ ] `:hints off` disables without resetting counter

### AC-4: Configurable Threshold
- [ ] `inline_hints_fade_threshold` in config.yaml, default 5
- [ ] Setting to 0 means hints never auto-fade (always on until manual :hints off)
- [ ] Setting to 1 means hints fade after first session

### AC-5: Tests
- [ ] Session counter increments correctly across simulated launches
- [ ] Fade dimming applied at exactly session N-1
- [ ] Auto-disable triggers at exactly session N
- [ ] Flash message generated on auto-disable
- [ ] Counter reset on `:hints on`
- [ ] Threshold 0 means no auto-fade
- [ ] Table-driven tests following project testing standards

## Technical Notes

- Session counter increment happens in MainModel Init() or wherever app startup is handled
- The fade styling (ANSI 240 vs 245) is already defined in 39.9's renderInlineHint
- This story wires the session counter to the rendering pipeline
- Flash message follows existing pattern in DoorsView
