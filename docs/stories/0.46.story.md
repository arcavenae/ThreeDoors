# Story 0.46: Label Authority & Triage Flow Documentation

## Story

As the ThreeDoors project maintainer,
I want a documented label authority matrix and triage flow that defines which agents can set/remove which labels and the end-to-end triage pipeline,
So that all agents follow consistent labeling practices and the triage process is unambiguous.

**Status:** Not Started

---

## Background

BOARD recommendation P-003 identified the need for a documented triage flow, and P-005's party mode produced a label authority table (Round 3). The P-003 artifact (`issue-labeling-and-triage-strategy.md`) was never committed, but its content was partially captured in the P-005 deliberation. This story consolidates both into a single operational reference document.

---

## Acceptance Criteria

### Label Authority Matrix

- **AC1:** A label authority matrix is documented (in `docs/operations/label-authority.md` or similar) covering:
  - Every label scope (type, priority, triage, scope, status, agent, contrib, resolution, process)
  - Who can SET each label (specific agent name or role)
  - Who can REMOVE each label
  - Whether the action is autonomous or requires approval

- **AC2:** The authority matrix matches the party mode consensus:

  | Label Scope | Set By | Removed By |
  |------------|--------|------------|
  | type.* | Envoy (autonomous) | Envoy, supervisor |
  | priority.* | Envoy (autonomous) | Supervisor (override) |
  | triage.* | Envoy (autonomous) | Envoy |
  | scope.* | Envoy (propose), supervisor (decide) | Supervisor |
  | status.blocked | Any agent | Resolving agent |
  | status.stale | Envoy, PR-shepherd | Any agent on activity |
  | status.needs-human | Any agent | Human |
  | status.do-not-merge | Supervisor, human | Supervisor, human |
  | agent.envoy | Envoy | Envoy |
  | agent.worker | Supervisor | Supervisor, worker |
  | contrib.* | Envoy, supervisor | Envoy, supervisor |
  | resolution.* | Envoy (propose), supervisor (confirm) | Supervisor |
  | process.fast-track | Envoy | Envoy |

### Triage Flow Documentation

- **AC3:** End-to-end triage flow is documented as a sequence:
  1. Issue created → envoy detects on patrol
  2. Envoy applies `triage.new` + `type.*` label
  3. Envoy screens (duplicate/spam/previously rejected → screen out)
  4. If passes screen → envoy applies `triage.in-progress`, messages supervisor
  5. If needs info → envoy applies `triage.needs-info`, asks reporter
  6. Supervisor decides triage approach (BMAD pipeline, fast-track, etc.)
  7. On completion → envoy applies `triage.complete` + `scope.*` + `priority.*`
  8. Story created → issue linked to story file

- **AC4:** Fast-track flow is documented as a shortcut:
  1. Envoy identifies trivial fix
  2. Envoy applies `process.fast-track`
  3. Skip full BMAD pipeline → create story directly
  4. Supervisor dispatches worker

### Scoped Label Mutual Exclusivity

- **AC5:** Document which scopes are mutually exclusive (only one label per scope):
  - `type.*` — mutually exclusive
  - `priority.*` — mutually exclusive
  - `triage.*` — mutually exclusive (represents current state)
  - `scope.*` — mutually exclusive
  - `status.*` — NOT mutually exclusive (can be both `blocked` and `stale`)
  - `agent.*` — NOT mutually exclusive (envoy triages, worker implements)
  - `resolution.*` — mutually exclusive

- **AC6:** Note that GitHub does not enforce mutual exclusivity natively — agents must enforce by convention (remove old label before applying new one within exclusive scopes)
