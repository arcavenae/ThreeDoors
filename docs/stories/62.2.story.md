## Story

As a multiclaude operator,
I want retrospector to write recommendations to a queue file instead of directly to BOARD.md,
So that recommendations are durably persisted without PR conflicts or authority violations.

## Background

The retrospector agent definition says "Cannot: Create PRs (deferred to Phase 2)." This means it has no mechanism to commit BOARD.md changes. Even if it wrote to BOARD.md in the shared checkout, concurrent edits from other agents cause merge conflicts. The BOARD.md file is ~400 lines and growing — even append-only edits at the bottom conflict.

The fix separates detection (retrospector → queue file) from persistence (project-watchdog/supervisor → BOARD.md). This aligns with the existing authority model where project-watchdog handles document updates.

## Acceptance Criteria

**Given** the retrospector detects a pattern worth recommending
**When** it would previously write to BOARD.md
**Then** it appends a structured JSONL entry to `docs/operations/retrospector-recommendations.jsonl`
**And** the entry includes: recommendation ID, text, date, confidence, evidence links, and status ("pending")

**Given** the recommendation queue file exists
**When** project-watchdog or supervisor processes the queue
**Then** pending recommendations are applied to BOARD.md's Pending Recommendations table in a governed PR
**And** the queue entries are updated with status "applied" and the PR number

**Given** the retrospector definition is updated
**When** it references BOARD.md as an output target
**Then** all references are changed to the queue file instead
**And** the BOARD.md recommendation format documentation is preserved for reference (so recommendations match the expected table format)

**Given** the queue file grows over time
**When** applied recommendations exceed 100 entries
**Then** a retention note in the schema documentation specifies that applied entries older than 30 days may be pruned

## Technical Notes

- Queue file: `docs/operations/retrospector-recommendations.jsonl`
- Queue schema:
  ```jsonl
  {"id": "REC-NNN", "recommendation": "...", "date": "YYYY-MM-DD", "confidence": "High|Medium|Low", "evidence": ["PR #NNN", "PR #MMM"], "status": "pending", "timestamp": "ISO8601"}
  {"id": "REC-NNN", "status": "applied", "applied_pr": 700, "applied_timestamp": "ISO8601"}
  ```
- The recommendation ID format (`REC-NNN`) should continue from the highest existing ID in BOARD.md
- Project-watchdog should check this queue file during its periodic governance sync cycle
- Update `agents/project-watchdog.md` to document the queue consumption responsibility
