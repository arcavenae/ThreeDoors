## Story

As a multiclaude operator,
I want to see current token consumption and estimated quota remaining,
So that I can make informed decisions about spawning workers and agent activity levels.

## Background

Claude Max plan has NO programmatic quota API. The only signals are: messages stop being accepted (hard block) or the `/stats` command in Claude Code. The Max plan operates on a rolling 5-hour window with estimated limits of ~88K tokens (Max 5x) or ~220K tokens (Max 20x).

Token consumption can be estimated by reading JSONL session logs at `~/.claude/projects/<project-hash>/<session-id>.jsonl`. Each entry contains input/output token counts per interaction.

Phase 1 is passive monitoring only — visibility before control. Active throttling (Phase 2+) comes later.

## Acceptance Criteria

**AC1: Script reads JSONL session logs and calculates token consumption**
**Given** JSONL session files exist at `~/.claude/`
**When** the monitoring script runs
**Then** it sums input and output tokens from the current 5-hour window
**And** reports total consumption as a number and percentage of estimated limit

**AC2: Script identifies per-agent consumption**
**Given** multiple agents have session logs
**When** the script runs
**Then** it breaks down consumption by agent/session
**And** identifies the highest consumers

**AC3: Script detects window boundaries**
**Given** the 5-hour window resets based on first message timestamp
**When** the script calculates the window
**Then** it correctly identifies the current window start time
**And** estimates when the window will reset

**AC4: Color-coded warning levels**
**Given** consumption is calculated
**When** displayed to the operator
**Then** output uses color coding:
  - Green: < 70% of estimated limit
  - Yellow: 70-85% of estimated limit
  - Red: > 85% of estimated limit

**AC5: Script is runnable from workspace window**
**Given** the operator is in the workspace window
**When** they run the monitoring command
**Then** it executes without requiring Claude agent context (pure shell script)

## Tasks

### Task 1: Create quota monitoring script
Write `scripts/quota-status.sh` that:
- Finds all JSONL session files from the last 5 hours
- Parses token counts from each entry
- Sums total input + output tokens
- Compares against configurable estimated limit (default: 88000 for Max 5x)
- Outputs formatted report with color coding

### Task 2: Add per-agent breakdown
Enhance the script to:
- Map session files to agent names (via PID/directory correlation)
- Show consumption per agent sorted by usage
- Identify the session with highest token burn rate

### Task 3: Add window timing
Calculate:
- Current window start (earliest message in current 5h window)
- Estimated window reset time
- Time remaining in current window

### Task 4: Add to justfile
Add `just quota` command that runs the monitoring script.

### Task 5: Document usage
Add documentation for the quota monitoring script in CLAUDE.md or a dedicated ops doc.

## Dev Notes

- JSONL format: each line is a JSON object with `type`, `message`, and token counts
- Session files are at `~/.claude/projects/-Users-skippy--multiclaude-repos-ThreeDoors/<session-id>.jsonl`
- The script should handle missing/corrupt JSONL files gracefully
- Estimated limits are community-derived, not official — make them configurable via environment variable
- This is a shell script, not Go code — it runs outside the application
- Marvel portability: Quota monitoring applies to any multiclaude deployment. Design the script to work with any project path.
- Consider: the ccusage tool (Go) already does some of this — evaluate whether to use/adapt it vs build from scratch
