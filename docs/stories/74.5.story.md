## Story

As a project maintainer,
I want story file comments to use typed prefixes ([decision], [observation], [blocker]),
So that the audit trail in story files is structured and machine-parseable for dark factory variant evaluation.

## Background

Story files accumulate comments during implementation — decisions made, blockers encountered, observations about the codebase. Currently these are unstructured prose. Chainlink (R-010) uses typed comments that allow tooling to extract specific information:
- `[decision]` — A choice was made between alternatives
- `[observation]` — Something notable was discovered
- `[blocker]` — Work is blocked on something external

Typed comments enable:
1. Dark factory variant evaluation: Which decisions led to which outcomes?
2. Automated decision extraction: Feed into BOARD.md
3. Retrospector analysis: Pattern detection across stories
4. Session handoff: Quickly scan for blockers and decisions

## Acceptance Criteria

**AC1: Comment type taxonomy defined**
**Given** story files need structured comments
**When** the taxonomy is defined
**Then** the following prefixes are supported:
  - `[decision]` — A choice between alternatives with rationale
  - `[observation]` — A notable finding during implementation
  - `[blocker]` — Work is blocked, needs external resolution
  - `[risk]` — A risk or concern identified during implementation
  - `[deviation]` — Implementation differs from the original AC/design

**AC2: Comment format specified**
**Given** typed comments are defined
**When** the format is documented
**Then** each typed comment follows:
  ```markdown
  > [decision] 2026-03-29 — Chose X over Y because Z.
  > Alternative considered: Y — rejected because W.
  ```
  - Prefix in brackets
  - ISO date
  - One-line summary after dash
  - Optional follow-up lines for detail

**AC3: Story file template updated**
**Given** a story file template or convention exists
**When** the template is updated
**Then** it includes a `## Implementation Notes` section with guidance on using typed comments

**AC4: Worker agent definition updated**

**Then** it instructs workers to:
  - Add `[decision]` comments when choosing between alternatives
  - Add `[blocker]` comments when work is blocked
  - Add `[observation]` comments for notable findings
  - Add `[deviation]` comments when implementation differs from ACs

**AC5: Documentation created**
**Given** the typed comment system is defined
**When** documentation is written
**Then** `docs/operations/typed-comments.md` explains:
  - The 5 comment types with examples
  - When to use each type
  - The structured format
  - How tooling will use typed comments (future: extraction, analysis)

## Dev Notes

- This is a convention change, not a code change — no application logic
- The typed prefix format `[type]` is chosen for simplicity and grep-ability: `grep '^\[decision\]' docs/stories/*.story.md`
- Keep the taxonomy small (5 types) to encourage adoption — too many types leads to inconsistency
- Blockquote format (`> [type]`) distinguishes comments from story content
- Marvel portability: Typed comments are a story-driven development enhancement — applicable to any project using story files
