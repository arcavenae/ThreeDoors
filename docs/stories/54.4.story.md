## Story

As the supervisor (or any requesting agent),
I want research results delivered as concise executive summaries with full reports stored on disk,
So that my context window is not overwhelmed by lengthy research output while still having full access to detailed findings on demand.

## Background

The three-layer shielding architecture is preserved from the original design. What changes is the input format — parsing JSON from `gemini -p --output-format json` instead of files from `agent-deep-research --output-dir`.

| Original (agent-deep-research) | New (Gemini CLI) |
|---|---|
| Output in `--output-dir`: report.md, metadata.json, sources.json | Single JSON response: `{ "response": "...", "stats": {...} }` |
| Multiple structured files to parse | Single `.response` field to extract |
| metadata.json provided by tool | Stats from JSON, request metadata added by wrapper |

The three layers remain:
1. **Executive Summary** (≤500 words) — sent via messaging to requesting agent
2. **Detailed Report** (full response) — stored as `report.md` on disk
3. **Raw Data** (response.json, request.json) — stored for auditing

## Acceptance Criteria

**Given** a completed Gemini CLI query
**When** the research-supervisor processes the JSON output
**Then** artifacts are stored at `_bmad-output/research-reports/YYYYMMDD-HHMMSS-<slug>/` containing: report.md, executive-summary.md, response.json, request.json, context-bundle.md

**Given** the raw JSON response from Gemini CLI
**When** the research-supervisor extracts the report
**Then** it reads the `.response` field from `response.json` and saves it as `report.md`

**Given** the full report.md
**When** the research-supervisor extracts a summary
**Then** it produces an executive-summary.md of ≤500 words with: key findings (3-5 bullets), top recommendation, confidence level, and file path to the full report

**Given** the executive summary
**When** delivered to the requesting agent
**Then** it is sent ONLY as a messaging text — the full report is NOT included in the message

**Given** the result message format
**When** the requesting agent receives it
**Then** the message follows the format: `RESEARCH-RESULT for '<query-slug>': <summary text> Full report: <path>`

**Given** the context-bundle.md file
**When** stored alongside the report
**Then** it records which context bundles were assembled and sent with the query (for reproducibility)

**Given** the request.json file
**When** stored alongside the report
**Then** it records: requester, priority, depth, model used, query text, timestamp, `--include-directories` paths used

**Given** research results
**When** the requesting agent decides to act on findings
**Then** the decision to act is made by the requesting agent or supervisor — the research-supervisor never autonomously triggers code changes, story creation, or ROADMAP edits

## Technical Notes

- The three-layer architecture is enforced by convention in the agent definition, not by Go code.
- `response.json` is the raw output from `gemini -p --output-format json`. The wrapper script saves this directly.
- `report.md` is extracted via `jq -r '.response' response.json` — done by the wrapper script.
- `executive-summary.md` extraction: if the report starts with an "Executive Summary" section, extract it. Otherwise, the research-supervisor reads and summarizes to ≤500 words. The research-supervisor can also ask Gemini Flash to summarize a long Pro response (costs 1 Flash query).
- No `sources.json` or `metadata.json` from the CLI — those were agent-deep-research-specific. Stats are embedded in the JSON response.
- The gating flow (research → summary → decision → action) must be clearly documented to prevent autonomous action.
