## Story

As a user,
I want to run `threedoors llm status` (or `:llm-status` in the TUI) to see which LLM backend is active and what services are available,
So that I can debug LLM feature issues without guessing what's configured.

## Background

LLM services depend on external tools being installed and configured. Users need visibility into what's working. This command complements Epic 49 (ThreeDoors Doctor) but is specific to LLM backend status.

**Party mode decision (S5-D7):** `threedoors llm status` from day one. Essential for debugging.

## Acceptance Criteria

**AC-1: CLI `llm status` command**
**Given** the CLI
**When** `threedoors llm status` is run
**Then** it displays:
- Active backend name and type (CLI vs HTTP)
- Command path (for CLI backends)
- Available status (reachable/unreachable)
- Fallback backends and their availability
- Service readiness (extract: ready, enrich: ready, etc.)

**AC-2: TUI `:llm-status` command**
**Given** the TUI
**When** user types `:llm-status`
**Then** an overlay or view shows the same information as the CLI command

**AC-3: JSON output for scripting**
**Given** the `--json` flag
**When** `threedoors llm status --json` is run
**Then** output is structured JSON with backend, fallbacks, and service status

**AC-4: No backend available**
**Given** no LLM backends are available
**When** `llm status` is run
**Then** it shows a helpful message: "No LLM backends available" with installation instructions for claude, gemini, and ollama
