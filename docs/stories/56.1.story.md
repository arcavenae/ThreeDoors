## Story

As a user,
I want door interiors to have a solid background color instead of being transparent wireframes,
So that doors look like physical surfaces with visual mass rather than flat line drawings.

## Background

Current doors are wireframes — box-drawing character borders with transparent (terminal background) interiors. The brain reads them as "line drawings" or "cards," not solid objects. The `Fill` field in `ThemeColors` exists but is never applied to interior rendering.

Adding background color to door interiors is the single highest-impact change for visual quality. It transforms doors from wireframes into solid surfaces. Without this foundation, shadow and bevel improvements look like artifacts on a flat wireframe.

This story also extends `ThemeColors` with the new color fields needed by subsequent stories (bevel, shadow gradient, panel zones). This makes it the foundational story for the entire epic.

## Acceptance Criteria

**Given** the `ThemeColors` struct in `theme.go`
**When** the struct is extended
**Then** the following new fields exist:
- `FillLower lipgloss.TerminalColor` — lower panel background (darker than `Fill`)
- `Highlight lipgloss.TerminalColor` — top/left edge bevel color (lighter)
- `ShadowEdge lipgloss.TerminalColor` — bottom/right edge bevel color (darker)
- `ShadowNear lipgloss.TerminalColor` — near shadow gradient column
- `ShadowFar lipgloss.TerminalColor` — far shadow gradient column
**And** all fields use `lipgloss.TerminalColor` with `CompleteColor` values for graceful degradation

**Given** a door is rendered in any of the 8 themes (classic, modern, scifi, shoji, winter, spring, summer, autumn)
**When** interior rows are displayed (between border characters)
**Then** blank interior space characters have a background color applied via `lipgloss.NewStyle().Background(theme.Colors.Fill)`
**And** content text is rendered over the background color
**And** the background color is theme-specific per the color palette in the research artifact

**Given** a door at minimum width (15 chars)
**When** rendered with background fill
**Then** content area width is unchanged (zero width cost)
**And** the door displays correctly

**Given** the 8 theme files
**When** `Fill` colors are defined
**Then** each theme uses the TrueColor/ANSI256/ANSI values from the research artifact:
- Classic: `#0d0d2a` / `17` / `0`
- Modern: `#0d0d0d` / `233` / `0`
- Sci-Fi: `#0a1a2e` / `17` / `0`
- Shoji: `#1a1508` / `234` / `0`
- Winter: `#0a0f1a` / `233` / `0`
- Spring: `#0a1a0d` / `233` / `0`
- Summer: `#1a1508` / `234` / `0`
- Autumn: `#1a0f08` / `233` / `0`

**Given** existing golden file tests
**When** the rendering changes are applied
**Then** golden files are regenerated to match the new rendering

## Technical Notes

- Extend `ThemeColors` in `internal/tui/themes/theme.go` with 5 new fields
- Update all 8 theme files to populate the new `ThemeColors` fields (all 6 new colors per theme)
- Modify each theme's `Render()` function to apply `Background(theme.Colors.Fill)` to interior space characters
- Content text rows: apply bg color to the entire interior width, render text over it
- Blank rows: apply bg color to `strings.Repeat(" ", innerWidth)`
- ~10 LOC per theme for bg fill rendering, ~80 LOC total + struct extension
- Zero width cost, zero performance impact
- Must pass `go test -race ./internal/tui/...`
