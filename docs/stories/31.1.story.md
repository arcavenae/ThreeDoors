## User Story

As a ThreeDoors developer, I want a native `ParentID` field on the Task struct, so that parent-child relationships are part of the core domain model and can be queried without external dependencies.

## Context

This is the foundational story for Epic 31. It adds the `ParentID *string` field to `core.Task` and creates `TaskPool` methods for querying parent-child relationships. The field is a native domain concept (not enrichment metadata) per the party mode consensus (2026-03-08) and architecture document.

Design Decision H9 specifies Expand (subtask creation) and Fork (variant creation) as detail view actions. This story provides the data model foundation that Stories 31.2 and 31.3 build upon.

**Architecture reference:** `_bmad-output/planning-artifacts/architecture-expand-fork.md` (Section 1)
**Party mode artifact:** `_bmad-output/planning-artifacts/party-mode-expand-fork-2026-03-08.md`
**Requirements:** FR121, FR123, FR124

### Current State

- `ExpandTaskMsg` already exists in `internal/tui/messages.go` and carries `ParentTask *core.Task`
- The main model's `ExpandTaskMsg` handler (`internal/tui/main_model.go:943-953`) creates a standalone task — no parent relationship stored
- `core.Task` struct (`internal/core/task.go`) has no `ParentID` field
- `DependsOn []string` exists for dependency tracking (different concept)

### Design Constraints

- Single-level nesting only — subtasks cannot have their own subtasks (v1)
- No property inheritance — subtasks are independent work items (FR124)
- No auto-completion — parent never auto-completes when all children finish
- ParentID is immutable — once set, a subtask cannot be re-parented
