## Story

As a power user or CI system,
I want to configure data source connections via CLI flags without interactive prompts,
So that I can script and automate connection setup.

## Background

The `threedoors connect <provider>` command supports both interactive (launches wizard from Story 44.1) and non-interactive modes. Non-interactive mode uses flags for all required fields: `--label`, `--token` (or env var), and provider-specific flags like `--repos`, `--server`, `--lists`, `--path`. The command validates inputs, stores credentials, creates the connection, and runs a connection test.

## Acceptance Criteria

**AC1: Todoist connection via flags**
**Given** `threedoors connect todoist --label "Personal" --token $TOKEN`
**When** the command runs
**Then** a Todoist connection is created with the specified label
**And** the token is stored in the keychain
**And** a connection test runs and results are printed
**And** the command exits 0 on success

**AC2: GitHub connection via flags with env var auth**
**Given** `threedoors connect github --label "OSS" --repos owner/repo1,owner/repo2`
**When** the command runs (with GH_TOKEN set)
**Then** a GitHub connection is created for the specified repos
**And** the GH_TOKEN env var is used for authentication

**AC3: Jira connection via flags**
**Given** `threedoors connect jira --label "Work" --server https://jira.company.com --token $TOKEN`
**When** the command runs
**Then** a Jira connection is created with the specified server URL

**AC4: Textfile connection via flags**
**Given** `threedoors connect textfile --label "Notes" --path ~/tasks.yaml`
**When** the command runs
**Then** a textfile connection is created pointing to the specified path

**AC5: Interactive fallback**
**Given** `threedoors connect todoist` with no flags
**When** the command runs in a terminal
**Then** a message indicates interactive mode is not yet available (defers to Story 44.1)

**AC6: Missing required flags error**
**Given** required flags are missing in non-interactive mode
**When** the command runs
**Then** a clear error message lists the missing required flags
**And** exit code is ExitValidation (3)

**AC7: JSON output**
**Given** `--json` flag is set
**When** any connect subcommand runs
**Then** output follows the JSONEnvelope format (schema_version=1, command="connect")

## Technical Notes

### Architecture
- New file: `internal/cli/connect.go` — all connect command logic
- Register `newConnectCmd()` in `root.go` via `cmd.AddCommand()`
- Add `"connect"` to `KnownSubcommands()` return value

### Dependency Wiring
- CLI commands create services on-demand (same pattern as `config.go`)
- `loadConfig()` → get configDir and config
- Create `ConnectionManager`, `CredentialStore`, `ConnectionService` in the command's RunE
- Use `NewChainCredentialStore` with `NewEnvCredentialStore` for reads
- For writes (--token flag), use `ConnectionService.Add()` which calls `CredentialStore.Set()`

### Provider Flag Mapping
| Provider | Required Flags | Optional Flags | Settings Map Keys |
|----------|---------------|----------------|-------------------|
| todoist | `--label`, `--token` (or TODOIST_API_TOKEN) | `--project-ids`, `--filter` | `api_token`, `project_ids`, `filter` |
| github | `--label`, `--repos` | `--token` (or GH_TOKEN/GITHUB_TOKEN) | `token`, `repos` |
| jira | `--label`, `--server`, `--token` (or JIRA_API_TOKEN) | `--jql`, `--email` | `url`, `api_token`, `email`, `jql` |
| textfile | `--label`, `--path` | (none) | path stored as setting |

### Output Format
- Human: tabwriter showing connection ID, provider, label, state, health check result
- JSON: `JSONEnvelope{command: "connect", data: {id, provider, label, state, health_check}}`

### Health Check
- After creating connection, run `ConnectionService.TestConnection(id)` if a HealthChecker is available
- If no HealthChecker, skip test and note in output
- Health check failure does NOT roll back the connection — just reports the result

## Tasks

### Task 1: Define `connect` Cobra command with provider subcommands
- Create `newConnectCmd()` returning parent command
- Create `newConnectTodoistCmd()`, `newConnectGithubCmd()`, `newConnectJiraCmd()`, `newConnectTextfileCmd()`
- Register in `root.go` and `KnownSubcommands()`

### Task 2: Implement provider-specific flag parsing
- Each provider subcommand defines its flags via `cmd.Flags().StringVar()`
- Validate required flags in RunE, return clear error listing missing ones
- Build `settings map[string]string` from parsed flags

### Task 3: Wire to ConnectionService.Add() and CredentialStore
- Create `ConnectionManager` + `CredentialStore` + `ConnectionService` in each RunE
- Call `service.Add(providerName, label, settings, credential)`
- Handle errors with appropriate exit codes

### Task 4: Run connection test and format output (table + JSON)
- After Add, attempt TestConnection if HealthChecker available
- Format output: human-readable table or JSON envelope
- Include: connection ID, provider, label, state, health check results

### Task 5: Tests — each provider's flag combinations, missing flags, JSON output
- Table-driven tests for each provider's flag parsing
- Test missing required flags produce ExitValidation
- Test JSON output format
- Test that connection is created with correct settings
- Use mock CredentialStore (test helper) to avoid keyring dependency in tests

## Dev Notes

### Testing Strategy
- Create a `testConnectionService()` helper that returns a `ConnectionService` with in-memory manager and mock credential store
- Use `bytes.Buffer` as stdout to capture output
- Execute cobra commands via `cmd.SetArgs()` + `cmd.Execute()`
- Table-driven tests per provider covering: valid flags, missing flags, JSON mode

### Edge Cases
- Token provided both via flag AND env var — flag takes precedence
- Empty `--repos` value → validation error
- Invalid Jira server URL → validation error from `jira.ParseConfig()`
- `--path` that doesn't exist → create connection anyway (file may be created later)
