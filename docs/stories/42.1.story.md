# Story 42.1: File Permission Standardization (0o700/0o600)

## Story

As a user,
I want my ThreeDoors data directory and files to use restrictive permissions (0o700 for directories, 0o600 for files),
So that other local users cannot read my task data, session logs, or configuration.

## Background

The `~/.threedoors/` directory is created with `0o755` (world-readable) and all data files with `0o644` (world-readable). Task data may contain sensitive information (personal notes, work-related items). The MCP subsystem and API provider caches already use correct permissions (0o700/0o600), creating an inconsistent security posture. This story standardizes all subsystems to the restrictive model.

Additionally, atomic write `.tmp` files are created with `os.Create()` which defaults to `0o666` (before umask), creating a brief window where temp files are world-readable. Using `os.OpenFile(..., 0o600)` eliminates this window.

## Acceptance Criteria

**AC-1: Directory permissions**
**Given** a fresh installation (no `~/.threedoors/` exists)
**When** ThreeDoors creates the data directory
**Then** `~/.threedoors/` is created with `0o700` permissions

**AC-2: Data file permissions**
**Given** ThreeDoors writes any data file (tasks.yaml, completed.txt, sync.log, sessions.jsonl, config.yaml, pattern cache, dispatch audit log)
**When** the file is created or overwritten
**Then** the file has `0o600` permissions

**AC-3: Temp file permissions**
**Given** ThreeDoors performs an atomic write operation
**When** the `.tmp` file is created
**Then** it uses `os.OpenFile(path, os.O_WRONLY|os.O_CREATE|os.O_TRUNC, 0o600)` instead of `os.Create()`

**AC-4: Existing directory migration**
**Given** a user upgrades from a version with permissive permissions
**When** ThreeDoors starts and detects `~/.threedoors/` with permissions more permissive than `0o700`
**Then** it calls `os.Chmod()` to tighten the directory to `0o700`
**And** logs a message indicating the migration occurred

**AC-5: Permission consistency**
**Given** any subsystem in the codebase
**When** creating directories or files for user data
**Then** it uses `0o700` for directories and `0o600` for files consistently
