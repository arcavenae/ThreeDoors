## Story

As the supervisor,
I want the research-supervisor to track daily query counts per model tier, prioritize requests, deduplicate similar queries, and select the appropriate model,
So that the free-tier limits (50 Pro/day, 1,000 Flash/day) are used efficiently and high-priority research is never blocked.

## Background

The budget model is significantly simplified — no cost tracking (queries are free), no dry-run gate (no per-query cost), no cost estimation. Instead, budget management focuses on **request count limits** per model tier.

| Original (paid API) | New (free tier) |
|---|---|
| 50 queries/day (paid, $2-5 each) | 50 Pro/day + 1,000 Flash/day (free) |
| Cost tracking in budget.json | Count tracking in budget.json |
| `--dry-run` gate for cost estimation | No dry-run needed (free queries) |
| Reserve 5 queries after 6pm UTC | Reserve 5 Pro queries after 6pm UTC |
| Dollar-based budget alerts | Count-based budget alerts |

Features preserved: priority queue, batch optimization, deduplication, cooldown, end-of-day reporting.

## Acceptance Criteria

**Given** the budget tracking file at `_bmad-output/research-reports/budget.json`
**When** the research-supervisor dispatches a query
**Then** the budget file is updated with: query ID, timestamp, depth, priority, requester, model used, status

**Given** the start of a new UTC day
**When** the research-supervisor checks the budget
**Then** `pro_used` resets to 0, `pro_remaining` resets to 50, `flash_used` resets to 0, `flash_remaining` resets to 1000

**Given** a high-priority request and Pro budget is exhausted
**When** the research-supervisor processes the request
**Then** it escalates to the supervisor via `RESEARCH-BUDGET` message rather than silently dropping the request

**Given** the time is after 6pm UTC
**When** the research-supervisor has ≤5 Pro queries remaining
**Then** only high-priority requests use Pro; normal and low-priority are downgraded to Flash or queued until the next day

**Given** a `depth=quick` request
**When** the research-supervisor selects a model
**Then** it always uses Flash (preserving Pro budget for deeper queries)

**Given** a `depth=deep` request and Pro budget is exhausted
**When** the research-supervisor evaluates the request
**Then** it informs the requester that Pro is exhausted, offers Flash as a fallback, or queues for next day

**Given** 3+ queued normal-priority requests with overlapping keywords
**When** the research-supervisor evaluates the queue
**Then** it combines them into a single deeper query with a merged prompt, saving budget

**Given** a new research request
**When** the research-supervisor checks for duplicates
**Then** it searches `request.json` files from the last 7 days for similar queries and notifies the requester of existing results if found

**Given** two queries dispatched in sequence
**When** timing is checked
**Then** at least 2 minutes elapsed between Pro dispatches (cooldown to respect rate limits)

**Given** the budget at 80% daily Pro usage (40/50 used)
**When** the research-supervisor checks budget status
**Then** it sends a `RESEARCH-BUDGET` warning to the supervisor

## Technical Notes

- Budget tracking is convention-based — the agent maintains `budget.json` through its own reasoning, not Go code.
- `budget.json` schema: `{ date, pro_limit: 50, pro_used, pro_remaining, flash_limit: 1000, flash_used, flash_remaining, queries: [...] }`
- No `--dry-run` gate — queries are free, so the dry-run step is eliminated.
- No cost fields in query records — replaced with `model` field (Pro vs Flash).
- Batch detection is heuristic: keyword overlap between queued request texts.
- Deduplication searches `request.json` files in recent report directories.
- The 2-minute cooldown applies to Pro queries only — Flash queries can be dispatched more frequently (60 RPM limit).
- End-of-day budget report: the agent sends a summary to the supervisor at the last poll cycle of the day.
- Model fallback: if Pro returns a rate limit error (429), automatically retry with Flash and log the downgrade.
