# Story 69.1: Extract View Transition & Navigation Logic

## Status: Not Started

## Epic

Epic 69: TUI MainModel Decomposition

## References

- `internal/tui/main_model.go` — 2991 lines, 32 ViewModes
- CLAUDE.md: "One primary type per file"

## Story

As a ThreeDoors developer,
I want view transition and navigation logic extracted from main_model.go into a dedicated file,
So that navigation behavior is isolated, testable, and doesn't contribute to merge conflicts in the main model.

## Background

`main_model.go` is the largest file in the codebase at 2991 lines. It contains 32 `ViewMode` constants, the `MainModel` struct with ~50+ fields, and all `Update()`/`View()` logic for every view. This is a classic god object. Every TUI PR risks merge conflicts because all changes touch this one file.

This story extracts the foundation — view transition logic — that other decomposition stories depend on. Go methods on the same receiver type can live in different files within the same package, making this a safe, idiomatic refactoring.

## Acceptance Criteria
