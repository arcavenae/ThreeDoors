# Story 38.5: Alpha Release Retention Cleanup

**Epic:** 38 — Dual Homebrew Distribution
**Status:** Done (PR #294)
**Priority:** P2
**Estimate:** S (1-2 hours)

---

## User Story

As a repository maintainer, I want old alpha releases to be automatically cleaned up so that the GitHub releases page does not accumulate hundreds of pre-releases.

## Context

Every push to main creates a GitHub pre-release with the tag `alpha-YYYYMMDD-SHA7`. With active development, this can produce 5-20 releases per day. Without cleanup, the releases page becomes unusable and git tags accumulate. This story adds a retention step that keeps the last 30 alpha releases and deletes older ones.

Party mode artifact: [Homebrew Dual Publish Course Correction](../../_bmad-output/planning-artifacts/homebrew-dual-publish-course-correction.md) (Decision CC-6)

## Acceptance Criteria

### AC-1: Automatic Cleanup After Release
- [ ] After creating a new alpha release, a cleanup step runs that deletes alpha releases beyond the retention limit
- [ ] The cleanup step is part of the `release` job in `ci.yml` (runs after the release is created)

### AC-2: Retention Policy
- [ ] The last 30 alpha releases are preserved
- [ ] Releases older than the 30th most recent alpha release are deleted
- [ ] Only releases matching the `alpha-*` tag pattern are targeted
- [ ] Stable releases (`v*` tags) are NEVER deleted by this cleanup

### AC-3: Tag Cleanup
- [ ] When an alpha release is deleted, its associated git tag is also deleted (`--cleanup-tag`)
- [ ] This prevents orphaned `alpha-*` tags from accumulating

### AC-4: Formula Safety
- [ ] The alpha formula always points to the latest release BEFORE cleanup runs
- [ ] The release referenced by the current `threedoors-a.rb` formula is never deleted (it's always the most recent, within the kept 30)

### AC-5: Idempotent and Safe
- [ ] If there are fewer than 30 alpha releases, the cleanup step is a no-op
- [ ] If cleanup fails, the new release is not affected (cleanup is best-effort)
- [ ] The cleanup step uses `continue-on-error: true` to prevent blocking the workflow

## Tasks

1. **Add cleanup step to `ci.yml`** — After release creation, run cleanup
2. **Implement cleanup logic** — `gh release list` + filter + delete
3. **Add `continue-on-error: true`** — Ensure cleanup failures don't block the workflow
4. **Verify stable releases are safe** — Confirm `v*` tags are not matched by the filter
5. **Update story status** — Mark as Done with PR number

## Technical Notes

- Implementation:
  ```bash
  gh release list --limit 200 \
    | grep 'alpha-' \
    | tail -n +31 \
    | awk '{print $1}' \
    | xargs -I{} gh release delete {} --yes --cleanup-tag
  ```
- `--cleanup-tag` deletes the git tag along with the release
- `gh release list` returns releases sorted by creation date (newest first)
- The `grep 'alpha-'` filter ensures only alpha releases are targeted
- `tail -n +31` skips the first 30 matches (the ones to keep)
- 30 releases provides 2-6 days of history at typical development velocity
- `GH_TOKEN` is already available in the workflow
