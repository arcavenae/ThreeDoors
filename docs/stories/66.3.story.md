# Story 66.3: Provider Spec Parity & Validation

**Epic:** 66 — CLI/TUI Adapter Wiring Parity
**Status:** Not Started
**Priority:** P1
**Estimate:** M (Medium)

## User Story

**As a** user connecting any supported provider via CLI,
**I want** the `threedoors connect` command to validate required flags for my provider,
**so that** I get clear error messages instead of silently incomplete configurations.

## Context

The CLI connect command's `knownProviderSpecs` map covers only 4 of 9 registered providers (todoist, github, jira, textfile). The remaining 5 (applenotes, obsidian, reminders, linear, clickup) are accepted as valid args but get no flag validation — all values pass through as raw settings without enforcing required fields. For example, `threedoors connect obsidian` doesn't enforce `--path` for `vault_path`.

**Sprint change proposal:** `_bmad-output/planning-artifacts/sprint-change-proposal-2026-03-13-unwired-features-gaps.md`
**Audit artifact:** `_bmad-output/planning-artifacts/unwired-features-audit.md` (Gap 4)

## Acceptance Criteria

1. `knownProviderSpecs` has entries for all 9 registered providers: textfile, applenotes, jira, github, obsidian, reminders, todoist, linear, clickup
2. Each provider spec correctly maps CLI flags to adapter settings keys
3. Required flags are enforced — `threedoors connect obsidian --label "Vault"` without `--path` returns a clear validation error
4. A parity test exists that verifies every adapter name in `registerBuiltinAdapters()` has a corresponding entry in `knownProviderSpecs`
5. The CLI `ValidArgs` list matches the `knownProviderSpecs` keys exactly
6. The connect command `Long` description is updated to list all supported providers
7. `make test`, `make lint`, `make fmt` all pass
