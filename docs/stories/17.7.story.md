# Story 17.7: Fix Width Calculation and Shoji ANSI Escape Leak

**Status:** Done (PR #181)
**Epic:** 17 — Door Theme System
**Depends on:** None (hotfix priority)
**Blocks:** Stories 17.8, 17.9

## User Story

As a user,
I want all door themes to render without visual artifacts,
So that I can read my tasks regardless of which theme I choose.

## Acceptance Criteria

**Given** a theme renders a door at any supported width
**When** the output is displayed in the terminal
**Then:**

- AC1: No raw ANSI escape codes are visible as literal text in any theme at any width
- AC2: `countRunes()` in `themes/modern.go` is replaced with `ansi.StringWidth()` from `charmbracelet/x/ansi` (or equivalent visual-width function)
- AC3: All width calculations in `modern.go`, `scifi.go`, and `shoji.go` use visual width measurement instead of rune counting
- AC4: Shoji `buildContentRow` produces lines whose visual width matches `totalW` exactly (no overflow, no underflow)
- AC5: Shoji `buildEmptyRow` produces lines whose visual width matches `totalW` exactly
- AC6: Every rendered line of every theme (at widths 28, 40, 60, 80) has consistent visual width (new test)
- AC7: Golden file tests updated to reflect corrected rendering output
- AC8: Existing theme integration tests (`doors_view_theme_test.go`) continue to pass

## Technical Notes

- `countRunes()` at `internal/tui/themes/modern.go:112-118` counts raw runes but does not account for ANSI escape sequences emitted by `lipgloss.Style.Render()`
- `ansi.StringWidth()` from `charmbracelet/x/ansi` (already a dependency) properly handles escape sequences and wide Unicode
- The Shoji bug manifests because `buildContentRow` mixes styled segments (`style.Render(ch.v)`) with plain text, and width math uses rune count of the plain text only
- Lipgloss or the terminal truncates overflowing lines mid-escape-sequence, exposing raw `[38;5;180m` text
- New regression test: iterate all rendered lines, assert `ansi.StringWidth(line) == expectedWidth` for each
- Golden files in `internal/tui/themes/testdata/` will need regeneration via `go test -update`
