# Story 0.51: TUI View Rendering Benchmarks

## Status: Done (PR #623)

**Epic:** 0 — Infrastructure & Process
**Priority:** P2
**Depends On:** None

## User Story

As a developer,
I want benchmarks for TUI `View()` rendering in complex views,
So that I can detect responsiveness regressions before they affect users.

## Context

The state-of-testing audit (R-001, 2026-03-09) identified that benchmark coverage is limited to `internal/core` and `internal/adapters/textfile`. No benchmarks exist for TUI view rendering, which is critical for perceived responsiveness. Complex views (doors, dashboard, stats, sources) render on every keypress — performance degradation would be immediately felt.

## Acceptance Criteria

- AC1: Benchmark functions exist for `DoorsView.View()`, `DashboardView.View()`, `StatsView.View()`, and `SourcesView.View()`
- AC2: Each benchmark uses realistic model state (populated task pool, active theme, multiple sources)
- AC3: Benchmarks run with `go test -bench=. ./internal/tui/...` and produce stable results
- AC4: Baseline results captured in `internal/tui/testdata/benchmarks-baseline.txt` for future comparison
- AC5: All benchmarks pass with `-race` flag
- AC6: No new dependencies added — use stdlib `testing.B` only

## Technical Notes

- Benchmark pattern: `func BenchmarkDoorsView(b *testing.B)` with `b.ResetTimer()` after setup
- Use `testkit` factories to build realistic model state
- Focus on `View()` output — the string rendering path, not `Update()`
- Consider `b.ReportAllocs()` to track allocation pressure
- Existing benchmarks in `internal/core/` provide good reference patterns
