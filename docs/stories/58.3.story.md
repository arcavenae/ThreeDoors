## Story

As a multiclaude daemon,
I want to orchestrate the full handover sequence from outgoing to incoming supervisor,
So that authority transfers cleanly without message loss, split-brain, or worker disruption.

## Background

The handover orchestrator is the core coordination logic. When the shift clock (Story 58.1) writes a handover signal, the orchestrator executes a 5-step protocol:

1. Send handover request to outgoing supervisor
2. Wait for outgoing to write its delta and signal completion
3. Spawn incoming supervisor with state file
4. Wait for incoming to confirm ready
5. Kill outgoing supervisor

**Key invariant:** At no point are zero supervisors watching workers. At no point are two supervisors dispatching work. The daemon is the mutex for authority.

**Party mode consensus (Sessions 2 & 5):** Phased authority transfer (outgoing → transition → incoming). File-based messaging survives supervisor lifecycle transitions by design.

## Acceptance Criteria

**AC-1: Signal file detection**
**Given** a handover signal file exists at `~/.multiclaude/handover/<repo>/handover-requested`
**When** the daemon refresh loop runs
**Then** the orchestrator initiates the handover sequence
**And** removes the signal file to prevent re-triggering

**AC-2: Outgoing supervisor notification**
**Given** handover is initiated
**When** the orchestrator notifies the outgoing supervisor
**Then** it sends a message via `multiclaude message send supervisor "HANDOVER_REQUESTED: Write your delta to shift-state.yaml and signal HANDOVER_COMPLETE"`

**AC-3: Delta wait with timeout**
**Given** the outgoing supervisor was notified
**When** waiting for completion
**Then** the orchestrator waits up to 120 seconds for a "HANDOVER_COMPLETE" message or updated shift-state.yaml delta
**And** proceeds with emergency protocol if timeout expires (see Story 58.5)

**AC-4: Incoming supervisor spawn**
**Given** the outgoing supervisor signaled completion (or timeout expired)
**When** spawning the incoming supervisor
**Then** it uses `multiclaude agents spawn --name supervisor --class persistent --prompt-file agents/supervisor.md --task "SHIFT_HANDOVER: Read ~/.multiclaude/handover/<repo>/shift-state.yaml and assume control"`

**AC-5: Ready confirmation**
**Given** the incoming supervisor was spawned
**When** waiting for readiness
**Then** the orchestrator waits up to 180 seconds for the incoming supervisor to send a "READY" message
**And** logs an error if timeout expires (incoming failed to start — see Story 58.5)

**AC-6: Outgoing termination**
**Given** the incoming supervisor confirmed ready
**When** terminating the outgoing
**Then** the orchestrator kills the outgoing supervisor's tmux session
**And** logs the handover completion with timing metrics

**AC-7: No split-brain**
**Given** the handover sequence
**When** at any point during the sequence
**Then** at most one supervisor has dispatch authority (can run `multiclaude work`)

**AC-8: Handover metadata**
**Given** a completed handover
**When** logging
**Then** the orchestrator records: outgoing name, incoming name, handover duration, whether delta was received, and any anomalies

## Technical Notes

- Implementation: Shell script (or extension of daemon's existing bash logic)
- Authority transfer is implicit — workers address the "supervisor" role, not an instance name. The daemon's spawn replaces the supervisor tmux window.
- For MVP, the "HANDOVER_COMPLETE" detection can be polling `multiclaude message list` for the keyword
- The emergency protocol (AC-3 timeout path) is stubbed in this story and fully implemented in Story 58.5
- The orchestrator must handle the case where the outgoing supervisor is already dead (crashed before handover request)

## Tasks

### Task 1: Signal file detection and consumption
- Check for `handover-requested` file in daemon loop
- Read contents (metrics, timestamp) and remove file atomically

### Task 2: Outgoing supervisor notification
- Send handover request message
- Log notification with timestamp

### Task 3: Delta wait loop
- Poll for HANDOVER_COMPLETE message (every 5 seconds, up to 120s)
- Alternatively detect shift-state.yaml modification time change
- On timeout: log warning, proceed to spawn (emergency path stubbed)

### Task 4: Incoming supervisor spawn
- Construct spawn command with task parameter including state file path
- Execute spawn, verify tmux session created

### Task 5: Ready confirmation wait
- Poll for READY message from incoming supervisor (every 5 seconds, up to 180s)
- On success: proceed to termination
- On timeout: log error, alert user

### Task 6: Outgoing termination
- Kill outgoing supervisor's tmux window/session
- Verify process is gone

### Task 7: Handover logging
- Record all timing metrics
- Log completion or failure with diagnostics

### Task 8: Integration tests
- Test full happy-path sequence with mock commands
- Test timeout paths (outgoing unresponsive, incoming fails to start)
- Test signal file consumption (no re-trigger)
