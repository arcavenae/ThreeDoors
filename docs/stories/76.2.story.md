## Story

As a multiclaude operator,
I want to see which agents consume the most tokens,
So that I can identify high-cost agents and optimize the system's resource usage.

## Background

multiclaude runs 6+ persistent agents plus ephemeral workers, all sharing the same Max plan quota. Without per-agent attribution, it's impossible to know which agents are burning through the budget. R-004 defines agent priority tiers (P0-P3) and identifies heartbeat polling as the primary waste category.

## Acceptance Criteria

**AC1: Agent-to-session mapping**
**Given** JSONL session files and multiclaude state data
**When** the attribution system runs
**Then** it maps each session to its multiclaude agent name (using worktree paths, state.json mappings, or session metadata)

**AC2: Per-agent usage breakdown**
**Given** mapped session data
**When** aggregated per agent
**Then** produces: agent name, total tokens (input + output), percentage of total consumption, active session count

**AC3: Priority tier classification**
**Given** per-agent usage data
**When** displayed
**Then** agents are classified by tier: P0 (active workers), P1 (merge-queue, pr-shepherd), P2 (envoy, watchdogs), P3 (retrospector)

**AC4: Productive vs overhead detection**
**Given** JSONL entries for an agent
**When** analyzed
**Then** distinguishes productive messages (containing tool_use entries) from overhead (heartbeat-like responses with no tool calls) where detectable

**AC5: Structured output for CLI display**
**Given** attribution data
**When** requested
**Then** returns structured data suitable for rendering as a CLI table (sorted by usage descending)

**AC6: Multi-agent test fixtures**
**Given** the attribution system
**When** tested
**Then** tests cover: multiple agents with different usage levels, unmapped sessions (fallback to "unknown"), agents with zero usage

## Tasks

### Task 1: Investigate agent-to-session mapping sources
Read multiclaude `state.json` structure. Check if agent name → JSONL session path mapping exists. Identify worktree path patterns as fallback.

### Task 2: Implement agent attribution logic
Build the mapping layer that connects JSONL files to agent identities.

### Task 3: Implement productive vs overhead classifier
Scan JSONL entries for tool_use blocks. Entries with tool calls are "productive"; entries without are "overhead/polling."

### Task 4: Build per-agent aggregation on top of 76.1
Consume 76.1's parser output, group by agent, calculate per-agent totals and percentages.

### Task 5: Write tests with multi-agent fixture data
Table-driven tests with multiple agents at different usage levels.

## Dev Notes

- Agent-to-session mapping is the hardest part of this story
- multiclaude state file: `~/.multiclaude/state.json` — check for agent session tracking fields
- Worktree-based inference: `~/.multiclaude/wts/<repo>/<agent-name>/` may appear in JSONL file paths
- If exact mapping is unreliable, support manual mapping via config file as fallback
- The productive/overhead classifier doesn't need to be perfect — heuristic is fine for Phase 1
