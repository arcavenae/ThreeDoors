## User Story

As a ThreeDoors user with multiple computers,
I want each device to have a stable identity,
So that the sync system can track which device made which changes.

## Acceptance Criteria

- AC1: Each ThreeDoors installation generates a unique device ID on first run (UUID v4)
- AC2: Device ID is persisted in `~/.threedoors/device.yaml` (or equivalent per spike findings)
- AC3: Device ID survives app updates and reinstalls (tied to install location, not binary)
- AC4: Device metadata includes: ID, human-readable name (defaults to hostname), first-seen timestamp, last-sync timestamp
- AC5: `threedoors devices` CLI command lists known devices with sync status
- AC6: `threedoors devices rename <id> <name>` allows setting human-readable device names
- AC7: Device registration is automatic — no manual setup required for first device
- AC8: Second device registration triggers sync setup wizard
- AC9: Unit tests for device ID generation, persistence, and collision avoidance

## Technical Notes

- Device ID must be deterministic enough to survive reinstalls but unique across machines
- Consider using a combination of machine-id + install-path hash as seed
- Device metadata is local-first — synced to other devices during sync handshake
- This story establishes the identity layer; transport (64.3) and conflict resolution (64.4) build on it
