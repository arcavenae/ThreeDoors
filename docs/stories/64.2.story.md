## Refined by Spike (ADR-0034)

Acceptance criteria refined based on Story 64.1 findings. Device identity uses UUID v5 seeded from machine ID + install path, with registration via the shared Git sync repository.

## User Story

As a ThreeDoors user with multiple computers,
I want each device to have a stable identity,
So that the sync system can track which device made which changes.

## Scope Decision (Party Mode — Phase 2 Enrichment)

AC4 (sync repo registration), AC5 (list from sync repo), AC7 (auto-registration on first run), and AC8 (sync setup wizard) reference sync repo infrastructure that doesn't exist until Story 64.3. To avoid circular dependencies:

- **AC4 scoped to local:** Device registration writes to a local `DeviceRegistry` interface. This story implements `LocalDeviceRegistry` (writes to `~/.threedoors/devices/`). Story 64.3 adds `GitDeviceRegistry` that writes to the sync repo.
- **AC5 scoped to local:** CLI lists devices from local registry. Story 64.3 extends to read from sync repo.
- **AC7 scoped to local:** First-run auto-creates local device identity and registers locally. No sync repo interaction.
- **AC8 deferred to 64.3:** Sync setup wizard requires transport infrastructure.

## Acceptance Criteria

- AC1: Device ID generated as UUID v5 (namespace UUID + machine-id + config dir path hash) for deterministic stability across reinstalls
- AC2: Fallback to UUID v4 (random) when machine-id is unavailable (e.g., some Linux containers); still persisted for stability
- AC3: Device ID persisted in `~/.threedoors/device.yaml` with metadata: ID, human-readable name (defaults to `os.Hostname()`), first-seen timestamp (UTC), last-sync timestamp (UTC)
- AC4: Device registration writes entry to local `DeviceRegistry` (`~/.threedoors/devices/<device-id>.yaml`). Registry is behind `DeviceRegistry` interface for future Git sync repo implementation.
- AC5: `threedoors devices` CLI command lists known devices from local registry with name, ID, first-seen, and last-sync time. Supports `--json` flag for machine-readable output.
- AC6: `threedoors devices rename <id> <name>` updates the human-readable name in both local `device.yaml` and local registry
- AC7: First device identity creation is automatic on first run — no manual setup required. Uses existing `core.EnsureConfigDir()` for path resolution.
- AC8: **Deferred to Story 64.3** — sync setup wizard requires transport infrastructure
- AC9: `DeviceID` type is a named string type with validation (must be valid UUID v5 or v4). Constructor: `NewDeviceID(s string) (DeviceID, error)`. Zero value `DeviceID("")` is invalid.
- AC10: Unit tests for: ID generation determinism (same inputs → same ID), persistence round-trip, collision avoidance (different machines → different IDs), fallback to UUID v4

### MachineIDReader Interface (DI for testability)

```go
type MachineIDReader interface {
    ReadMachineID() (string, error)
}
```

Platform implementations:
- **macOS:** `ioreg -rd1 -c IOPlatformExpertDevice` → parse `IOPlatformUUID`
- **Linux:** Read `/etc/machine-id`
- **Other/fallback:** Return `ErrMachineIDUnavailable` → triggers UUID v4 fallback

### ThreeDoors Namespace UUID

Generate a stable namespace UUID for the project. Use UUID v5 with DNS namespace to derive it:

```go
// ThreeDoorsNamespace is the UUID v5 namespace for device ID generation.
// Generated via uuid.NewSHA1(uuid.NameSpaceDNS, []byte("threedoors.app"))
var ThreeDoorsNamespace = uuid.NewSHA1(uuid.NameSpaceDNS, []byte("threedoors.app"))
```

### Config Path Integration

Use `core.GetConfigDirPath()` and `core.EnsureConfigDir()` from `internal/core/config_paths.go`. Do NOT hardcode `~/.threedoors/`.

### DeviceRegistry Interface

```go
type DeviceRegistry interface {
    Register(device Device) error
    Get(id DeviceID) (Device, error)
    List() ([]Device, error)
    Update(device Device) error
    Remove(id DeviceID) error
}
```

`LocalDeviceRegistry` writes individual YAML files to `~/.threedoors/devices/<device-id>.yaml`. Uses atomic writes (write to `.tmp`, sync, rename) per project coding standards.

### Sentinel Errors

```go
var ErrMachineIDUnavailable = errors.New("machine-id unavailable")

var ErrDeviceAlreadyExists = errors.New("device already exists")
var ErrDeviceYAMLCorrupted = errors.New("device.yaml corrupted")
```

### Error Handling

- **Corrupted `device.yaml`:** Log warning, regenerate device identity (preserving ID if parseable), back up corrupted file to `device.yaml.bak`
- **Permission errors:** Return wrapped error with clear message ("cannot read/write device.yaml: permission denied")
- **Unsupported platform:** Fall back to UUID v4 (AC2), log that machine-id is unavailable
- **Concurrent access:** Use atomic writes (tmp + rename) for both `device.yaml` and registry files
- **Existing `~/.threedoors/` without device identity:** Create device identity on first access without disrupting existing task data (upgrade path)

## Test Strategy Notes

### Test Fixtures (`internal/device/testdata/`)

- `valid_device.yaml` — complete, well-formed device metadata
- `corrupted_device.yaml` — malformed YAML syntax
- `partial_device.yaml` — missing required fields (e.g., no ID)

### Mock Machine IDs for Determinism

Use concrete mock values in table-driven tests:
- Input `"test-machine-id-alpha"` + config dir `/tmp/test/.threedoors` → expected UUID v5 must be deterministic (compute once, hardcode in test)
- Input `"test-machine-id-beta"` + same config dir → must produce DIFFERENT UUID v5
- Same input + different config dir → must produce DIFFERENT UUID v5

### Registry Test Scenarios

- List empty registry → returns empty slice (not nil)
- Register then Get → round-trip succeeds
- Register same device twice → returns `ErrDeviceAlreadyExists`
- Get non-existent → returns `ErrDeviceNotFound`
- Remove non-existent → returns `ErrDeviceNotFound`
- Registry directory doesn't exist → auto-creates on first Register
- Concurrent Register calls → no data corruption (atomic writes)

### Test Infrastructure

- Use `t.TempDir()` for all filesystem tests — never hit real `~/.threedoors/`
- Use `core.SetHomeDir(t.TempDir())` for config path integration tests
- CLI tests follow existing pattern: `cobra.Command.Execute()` with captured stdout
- Build-tagged files (`_darwin.go`, `_linux.go`) are thin wrappers — core logic in platform-independent `generator.go` testable on any OS
- Mark independent tests with `t.Parallel()`
