## Story

As a multiclaude operator,
I want the retrospector agent to verify its messaging identity on startup and fall back to a file-based inbox when multiclaude messaging is broken,
So that the supervisor can reliably communicate with the retrospector regardless of daemon messaging state.

## Background

Persistent agents spawned via `multiclaude agents spawn` cannot reliably receive messages via `multiclaude message list`. The retrospector gets "No messages" even when the supervisor sends it messages. The agent identity registered at spawn time may not match the identity the daemon routes messages to.

This is a multiclaude daemon bug, but we can work around it in agent definitions and operational files. The fix must be in THIS repo (ThreeDoors), not in multiclaude itself.

## Acceptance Criteria

**Given** the retrospector starts up (fresh spawn or restart)
**When** it runs its startup sequence
**Then** it sends itself a test message via `multiclaude message send retrospector "IDENTITY_PROBE"`
**And** waits up to 30 seconds for the probe to appear in `multiclaude message list`
**And** logs the result (success or failure) to its findings log

**Given** the identity probe fails (message not received within 30 seconds)
**When** the retrospector enters its polling loop
**Then** it logs a warning: "Messaging identity not registered. Falling back to file-based inbox."
**And** polls `docs/operations/retrospector-inbox.jsonl` for messages in addition to `multiclaude message list`
**And** continues operating — does NOT block or prompt the user

**Given** the file-based inbox exists
**When** the supervisor appends a message to `docs/operations/retrospector-inbox.jsonl`
**Then** the retrospector reads and processes the message on its next polling cycle (≤15 minutes)
**And** marks the message as processed (appends an ack entry to the same file)

**Given** the retrospector definition is updated
**When** the supervisor needs to communicate with retrospector
**Then** the definition includes instructions for the supervisor on how to use the fallback inbox
**And** the fallback format is documented in the Communication section

## Technical Notes

- File-based inbox uses JSONL format (consistent with `retrospector-findings.jsonl`)
- Inbox schema: `{"id": "msg-001", "from": "supervisor", "content": "...", "timestamp": "...", "processed": false}`
- Ack schema: `{"id": "msg-001", "acked": true, "timestamp": "..."}`
- The retrospector should always try `multiclaude message list` first, then check the file inbox as fallback
- Create `docs/operations/retrospector-inbox.jsonl` as an empty file if it doesn't exist
- Update supervisor agent definition (`agents/supervisor.md`) to document the fallback path
