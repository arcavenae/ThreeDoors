## Story

As a project maintainer,
I want CI to run ShellCheck on shell scripts when they change,
So that shell bugs in release-critical scripts (create-pkg.sh, create-dmg.sh, create-app.sh) are caught before merge.

## Background

There are 30 shell scripts in `scripts/` with no CI linting. Several are release-critical: `create-pkg.sh`, `create-dmg.sh`, and `create-app.sh` are used in the CI signing/notarization pipeline. Shell bugs there could break the release process. ShellCheck catches real bugs: unquoted variables, missing error handling, bashisms in `#!/bin/sh` scripts, and more.

The job should start as informational (non-blocking) since existing scripts likely have ShellCheck warnings. It can be promoted to required after cleanup.

## Acceptance Criteria

**AC1: Scripts filter added to path-filter**
**Given** the `ci.yml` path filter configuration
**When** a PR modifies files matching `scripts/**` or `justfile`
**Then** a `scripts` output is set to `true`

**AC2: ShellCheck job runs on script changes**
**Given** the `scripts` filter output is `true` or the event is a push to main
**When** CI runs
**Then** a `ShellCheck` job executes `shellcheck` against `scripts/*.sh`
**And** the job reports results as annotations

**AC3: ShellCheck is informational (non-blocking)**
**Given** ShellCheck finds warnings in existing scripts
**When** the job completes
**Then** it does NOT block PR merge (uses `continue-on-error: true` or similar)
**And** the output clearly shows which scripts have issues

**AC4: Hooks scripts excluded from ShellCheck**
**Given** scripts in `scripts/hooks/` are safety-critical hooks
**When** ShellCheck runs
**Then** it lints `scripts/*.sh` (top-level scripts only)
**And** `scripts/hooks/*.sh` may be included separately if desired, but should not block

**AC5: Push-to-main runs ShellCheck unconditionally**
**Given** a push event to `main`
**When** CI runs
**Then** the ShellCheck job executes regardless of which files changed

## Tasks

### Task 1: Add `scripts` filter to `dorny/paths-filter`
Add a new filter output:
```yaml
scripts:
  - 'scripts/**'
  - 'justfile'
```

### Task 2: Add ShellCheck job to `ci.yml`
Create a new job that:
- Depends on the `changes` job
- Runs when `scripts` output is `true` or on push-to-main
- Installs `shellcheck` via apt
- Runs `shellcheck scripts/*.sh`
- Uses `continue-on-error: true` to avoid blocking

### Task 3: Document upgrade path
Add a comment in the workflow explaining how to promote ShellCheck to required once existing warnings are cleaned up.

## Dev Notes

- ShellCheck runs in <10 seconds — negligible CI cost
- `shellcheck` is available via `apt-get install -y shellcheck` on Ubuntu runners
- Scripts use a mix of `#!/usr/bin/env bash` and `#!/bin/bash` shebangs — ShellCheck handles both
- Do NOT add ShellCheck to the branch ruleset's required status checks (informational only for now)
- Future story: clean up existing ShellCheck warnings and promote to required
