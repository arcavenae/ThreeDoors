## Story

As a user facing an overwhelming task,
I want to press B (or type `:breakdown`) to decompose it into smaller subtasks via an LLM,
So that large tasks become approachable bite-sized actions.

## Background

Task Breakdown is a P0 service that extends the existing `LLMTaskDecomposer` from Epic 14. The decomposer already works with HTTP backends — this story ensures it also works with CLI backends (which it should, since CLIProvider implements the same `LLMBackend` interface).

The main new work is the TUI integration: a `:breakdown` command in the detail view that shows proposed subtasks for selection and import.

**Party mode decision (S1-D2):** Task Breakdown is P0. Extends existing Epic 14; reduces task paralysis.

## Acceptance Criteria

**AC-1: Existing decomposer works with CLI backends**
**Given** the `LLMTaskDecomposer` from `internal/intelligence/llm/decomposer.go`
**When** initialized with a CLIProvider backend
**Then** decomposition produces the same structured output as with HTTP backends

**AC-2: `:breakdown` command in detail view**
**Given** the user is viewing a task in detail view
**When** they press B or type `:breakdown`
**Then** the breakdown flow begins with a loading spinner

**AC-3: Subtask review screen**
**Given** breakdown completes
**When** results are shown
**Then** proposed subtasks are listed with: index, toggle checkbox, text, effort estimate
**And** the parent task is shown at the top for context

**AC-4: Subtask import**
**Given** subtasks are selected
**When** user presses Enter
**Then** selected subtasks are added to the task pool
**And** they are linked to the parent task (if the data model supports it)
**And** a flash message confirms import

**AC-5: Cancel support**
**Given** breakdown is in progress
**When** user presses Esc
**Then** the LLM call is cancelled and user returns to detail view

**AC-6: Latency target**
**Given** a single task
**When** breakdown runs
**Then** results appear within 8 seconds (S4-D7)
