# Story 61.2: Content Split — Getting Started & Core Guide

## Status: Done (PR #643)

## Epic

Epic 61: GitHub Pages User Guide

## References

- Research: PRs #481 and #500
- Planning: `_bmad-output/planning-artifacts/gh-pages-user-guide-plan.md`

## Story

As a new ThreeDoors user,
I want getting-started guides and core usage documentation on the docs site,
So that I can learn to use the app through well-organized, searchable pages.

## Background

The monolithic `docs/user-guide.md` (1,173 lines) contains all documentation in one file. This story extracts the getting-started and core guide sections into dedicated MkDocs pages with proper structure, navigation, and MkDocs-specific enhancements (admonitions, keyboard key rendering, code copy).

## Acceptance Criteria

**Given** the docs site infrastructure from Story 61.1
**When** getting-started pages are created
**Then** `docs-site/docs/getting-started/installation.md` covers all installation methods (Homebrew stable + alpha, binary, go install, source build) with prerequisites
**And** `docs-site/docs/getting-started/quickstart.md` walks through first launch, onboarding wizard, and completing a first task in 5 minutes
**And** `docs-site/docs/getting-started/concepts.md` explains Three Doors philosophy, selection algorithm, behavioral science foundation

**Given** the docs site infrastructure
**When** core guide pages are created
**Then** `docs-site/docs/guide/task-management.md` covers task statuses, transitions, quick add, categorization, undo completion
**And** `docs-site/docs/guide/search-and-commands.md` covers search (`/`), command palette (`:`), available commands
**And** `docs-site/docs/guide/doors-interaction.md` covers door selection, refresh, feedback options (blocked/not now/needs breakdown), mood logging
**And** `docs-site/docs/guide/keybindings.md` has complete key binding tables for all views plus keybinding overlay
**And** `docs-site/docs/guide/sessions.md` covers session metrics tracking, mood correlation, pattern insights

**Given** all new pages
**When** content is verified
**Then** all content accurately reflects the current implementation

**Given** the mkdocs.yml navigation
**When** updated with new pages
**Then** all pages appear in the correct navigation sections

**Given** the complete site
**When** `mkdocs build --strict` is run
**Then** there are no broken internal links and no warnings
