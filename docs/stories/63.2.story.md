# Story 63.2: Read-Only ClickUp Provider with Field Mapping

## Status: Done (PR #719)

## User Story

As a ThreeDoors user who uses ClickUp for project management,
I want my ClickUp tasks to appear as ThreeDoors tasks,
So that I can manage my ClickUp work through the Three Doors interface.

## Acceptance Criteria

- AC1: `ClickUpProvider` struct in `internal/adapters/clickup/clickup_provider.go` implements `TaskProvider` interface
- AC2: `NewClickUpProvider(config *ClickUpConfig) *ClickUpProvider` constructor
- AC3: `LoadTasks(ctx)` fetches tasks from configured lists/spaces and maps to ThreeDoors `Task` structs
- AC4: Field mapping: ClickUp `name` → Task `Title`, ClickUp `description` → Task `Body`, ClickUp `status` → Task `Status` (with configurable status mapping)
- AC5: Priority mapping: ClickUp priority (1=Urgent, 2=High, 3=Normal, 4=Low) → ThreeDoors priority levels
- AC6: `SourceRef` populated with ClickUp task ID, list ID, and workspace URL for back-linking
- AC7: Due date mapping: ClickUp `due_date` (Unix ms) → Task `DueDate` (UTC)
- AC8: Tag mapping: ClickUp tags → Task labels
- AC9: `HealthCheck(ctx)` verifies API connectivity via `GetAuthorizedUser()`
- AC10: Poll-based sync at configurable interval (default 5m) using `tea.Tick`
- AC11: Graceful handling of ClickUp tasks without required fields (skip with warning log, don't fail batch)
- AC12: Integration tests with `httptest.NewServer` serving canned ClickUp responses through full provider flow

## Technical Notes

- Follow the exact provider pattern from `internal/adapters/todoist/todoist_provider.go`
- ClickUp statuses are workspace-specific (custom per space) — use configurable mapping in `ClickUpConfig.StatusMapping map[string]Status`
- Default status mapping: "to do"/"open" → StatusTodo, "in progress" → StatusActive, "complete"/"closed"/"done" → StatusDone
- ClickUp `due_date` is Unix milliseconds (not seconds) — divide by 1000 for Go time.Unix()
- Accept interfaces, return concrete types
- Context as first parameter on all methods
