# Story 79.3: Door Operations API — getDoors, selectDoor, refresh

**Epic:** 79 — ThreeDoors Daemon/Server Mode
**Status:** Not Started
**Priority:** P1
**Estimate:** M (API methods, door state management, event emission)

## Context

The daemon's primary value is exposing ThreeDoors' core "three doors" concept to external clients. This story implements the door-related JSON-RPC methods that let a Stream Deck plugin (or any client) see the current doors, select one, and request a refresh.

The door selection logic already exists in `internal/core/door_selector.go` (`SelectDoors`, `SelectDoorsWithFocus`, `SelectDoorsWithEnergy`). This story wraps it in daemon API methods and manages the shared door state across concurrent clients.

## Acceptance Criteria

- [ ] `doors.get` method returns current three doors with task metadata (id, text, type, effort, status, source provider)
- [ ] `doors.select` method accepts `{"door": 1|2|3}` and marks the selected task as in-progress
- [ ] `doors.refresh` method generates a new set of three doors (re-roll)
- [ ] `doors.get` returns empty array with descriptive message when no tasks are available
- [ ] Door state is shared across all connected clients (one canonical set of doors)
- [ ] Door state persists across client reconnects (doors don't reset when a client disconnects)
- [ ] `doors.select` returns the selected task's full details in the response
- [ ] `doors.select` with invalid door number returns JSON-RPC error (Invalid Params)
- [ ] `doors.refresh` uses the same diversity-aware selection as the TUI (`SelectDoors`)
- [ ] After `doors.select`, the selected door is replaced on next `doors.refresh` (not immediately)
- [ ] Tests: get doors, select door, refresh, concurrent access, empty pool, invalid door number

## Technical Notes

### Daemon Door State

The daemon maintains a `DoorState` struct:

```go
type DoorState struct {
    mu    sync.RWMutex
    doors []*core.Task  // current 3 doors
    pool  *core.TaskPool
}
```

This is separate from the TUI's door state — the daemon and TUI can run simultaneously without conflict since they share the same TaskProvider (file-based YAML).

### Concurrency

Multiple Stream Deck buttons may fire `doors.select` nearly simultaneously. Use `sync.Mutex` on DoorState to serialize writes. Reads (`doors.get`) use `sync.RWMutex` for concurrent access.

### Task Metadata in Response

```json
{
  "doors": [
    {
      "position": 1,
      "task": {
        "id": "abc123",
        "text": "Review PR #456",
        "type": "technical",
        "effort": "medium",
        "status": "todo",
        "provider": "github"
      }
    }
  ]
}
```
