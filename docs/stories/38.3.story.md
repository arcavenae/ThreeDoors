# Story 38.3: Stable Release Signing & Notarization

## Status: Done (PR #288)

**Epic:** 38 ‚Äî Dual Homebrew Distribution
**Priority:** P1
**Estimate:** L (5-8 hours)

---

## User Story

As a ThreeDoors user, I want stable releases to be code-signed and notarized so that macOS Gatekeeper does not quarantine the binary, providing the same security assurance as alpha builds.

## Context

Stable releases via GoReleaser run on `ubuntu-latest` and produce unsigned binaries. Alpha builds run signing and notarization on `macos-latest`. This creates an ironic gap where the "stable" channel is less secure than the alpha channel. This story adds a post-GoReleaser signing job to `release.yml` that mirrors the proven alpha signing pipeline.

Party mode artifact: [Homebrew Dual Publish Course Correction](../../_bmad-output/planning-artifacts/homebrew-dual-publish-course-correction.md) (Decision CC-1, CC-2)

## Acceptance Criteria

### AC-1: Post-GoReleaser Signing Job
- [ ] `release.yml` includes a `sign-stable` job that runs after the `goreleaser` job completes
- [ ] The `sign-stable` job runs on `macos-latest`
- [ ] The job uses the `release` environment (same as alpha signing)
- [ ] The job is gated by `vars.SIGNING_ENABLED == 'true'`

### AC-2: Darwin Binary Signing
- [ ] Darwin binaries (arm64, amd64) are extracted from GoReleaser's tar.gz archives
- [ ] Binaries are signed with the Apple Developer ID certificate using `codesign --force --options runtime --sign`
- [ ] Bundle identifier is `com.arcaven.threedoors`
- [ ] Signatures are verified with `codesign --verify --deep --strict`

### AC-3: Notarization
- [ ] Signed binaries are packaged for notarization submission
- [ ] Notarization uses the same Apple credentials as the alpha pipeline (`APPLE_NOTARIZATION_APPLE_ID`, `APPLE_NOTARIZATION_PASSWORD`, `APPLE_NOTARIZATION_TEAM_ID`)
- [ ] Notarization completes successfully (with timeout)

### AC-4: Re-Upload Signed Archives
- [ ] Signed binaries are re-archived in the same tar.gz format GoReleaser uses
- [ ] SHA256 checksums are recalculated for the new archives
- [ ] The `checksums.txt` release asset is updated with new checksums
- [ ] Signed archives replace the unsigned ones in the GitHub release

### AC-5: Fail-Open Behavior
- [ ] If `SIGNING_ENABLED` is not set or `'false'`, the signing job is skipped entirely
- [ ] If the signing job fails, the GoReleaser release remains with unsigned binaries (not a regression from current state)
- [ ] The Homebrew formula (pushed by GoReleaser) continues to work with either signed or unsigned binaries

### AC-6: Shared Signing Identity
- [ ] The same Apple Developer ID certificate is used for both stable and alpha signing
- [ ] No separate certificate or identity is required for stable releases

### AC-7: Keychain Cleanup
- [ ] The temporary keychain is deleted in an `always` cleanup step (same pattern as alpha)

## Tasks

1. **Add `sign-stable` job to `release.yml`** ‚Äî New job running on `macos-latest` after `goreleaser`
2. **Import certificates** ‚Äî Same keychain import pattern as `ci.yml`'s `sign-and-notarize`
3. **Download and extract darwin archives** ‚Äî Fetch tar.gz archives from the GoReleaser release, extract the `threedoors` binary
4. **Sign darwin binaries** ‚Äî `codesign` with Developer ID, hardened runtime, timestamp
5. **Notarize** ‚Äî Submit for notarization, wait for completion
6. **Re-archive and re-upload** ‚Äî Re-create tar.gz, update checksums, replace release assets
7. **Add keychain cleanup** ‚Äî `security delete-keychain` in `always` block
8. **Test with a test tag** ‚Äî Verify end-to-end with a test `v*` tag (can be deleted after)
9. **Update story status** ‚Äî Mark as Done with PR number

## Technical Notes

- GoReleaser creates archives named `threedoors_VERSION_OS_ARCH.tar.gz` containing the `threedoors` binary plus `LICENSE` and `README.md`
- The signing job must preserve the archive structure (not just the binary)
- GoReleaser's `checksums.txt` must be regenerated after re-archiving with signed binaries
- The `gh release upload --clobber` flag allows replacing existing release assets
- Certificate import uses the same secrets: `APPLE_CERTIFICATE_P12`, `APPLE_CERTIFICATE_PASSWORD`, `APPLE_SIGNING_IDENTITY`
- No changes to GoReleaser config needed ‚Äî it runs on ubuntu as before
- Linux binaries do not need signing (no Gatekeeper equivalent)
- DRY concern: codesign steps appear in both `ci.yml` and `release.yml`. Acceptable for now; extracting to a composite action is a future optimization

## Dependencies

- GoReleaser release workflow (`release.yml`) ‚Äî already exists and working
- Apple Developer ID certificate secrets ‚Äî already configured for alpha signing
- `release` environment ‚Äî already configured
