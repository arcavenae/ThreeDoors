# Story 0.32: Help Display UX — Dedicated Help View

## Status: Done (PR #309)

## Story

As a ThreeDoors user,
I want `:help` to show a persistent, readable, categorized help screen,
So that I can learn keybindings and commands without content disappearing or running off-screen.


---

## Background

The current `:help` implementation returns a `FlashMsg` — a single ~300-character line that auto-clears after 3 seconds. This has two critical UX failures:

1. **Content runs off-screen:** The single line exceeds 80 columns with no wrapping. Most content is invisible.
2. **Content disappears:** `FlashMsg` auto-clears via `ClearFlashCmd()` after `flashDuration = 3 * time.Second`. Users cannot read it.

Every other informational command in ThreeDoors (`:synclog`, `:theme`, `:health`, `:insights`, `:devqueue`) uses a dedicated ViewMode with persistent display until dismissed. Help is the only command that uses flash for informational content.

## Acceptance Criteria

### AC1: Dedicated Help View

- [ ] New `ViewHelp` value added to `ViewMode` enum
- [ ] New `HelpView` struct with `Update()`, `View()`, and `SetWidth()` methods
- [ ] `:help` command transitions to `ViewHelp` (sends `ShowHelpMsg`, not `FlashMsg`)
- [ ] Help view persists until explicitly dismissed — no auto-clear timer

### AC2: Width-Aware Layout

- [ ] Help content renders correctly in 80-column terminals with no horizontal overflow
- [ ] Two-column layout: key/command on left, description on right
- [ ] Description column word-wraps at terminal width boundary
- [ ] Layout adapts when terminal is resized (responds to `tea.WindowSizeMsg`)

### AC3: Categorized Content

- [ ] Help content organized into named sections: Navigation, Task Actions, Commands, Search
- [ ] Each section has a styled header
- [ ] Sections separated by visual whitespace
- [ ] All current keybindings and commands are listed (no missing entries vs current `:help` output)

### AC4: Scrollable Navigation

- [ ] j/k (and arrow keys) scroll line-by-line
- [ ] PgUp/PgDn (and Space) scroll by page
- [ ] Scroll position indicator shown (e.g., "Showing lines X-Y of Z" or percentage)
- [ ] Cannot scroll past content boundaries

### AC5: Standard Dismissal

- [ ] Esc returns to previous view
- [ ] q returns to previous view
- [ ] Dismissal follows same pattern as SyncLogView (`ReturnToDoorsMsg`)

### AC6: Global `?` Keybinding

- [ ] `?` (Shift+/) opens help view from any non-text-input view
- [ ] `?` is handled in `MainModel.Update()` with `isTextInputActive()` guard
- [ ] `?` does not conflict with any existing keybinding

### AC7: Tests

- [ ] Table-driven unit tests for HelpView.Update() (key handling, scroll bounds)
- [ ] View rendering test verifying content fits within 80-column width
- [ ] Golden file test for help view at standard width
- [ ] Integration test: `:help` command produces `ShowHelpMsg` (not `FlashMsg`)
- [ ] `go test -race ./internal/tui/...` passes

## Technical Notes

### Pattern to Follow

`SyncLogView` (`internal/tui/synclog_view.go`) is the closest analog:
- Scrollable content with offset-based pagination
- j/k/PgUp/PgDn navigation
- q/Esc dismissal via `ReturnToDoorsMsg`
- `SetWidth()` for terminal width awareness

### Files to Create

- `internal/tui/help_view.go` — HelpView struct
- `internal/tui/help_view_test.go` — tests

### Files to Modify

- `internal/tui/main_model.go` — ViewHelp enum value, helpView field, routing in Update()/View(), `?` key handler
- `internal/tui/search_view.go` — Change `:help` case from `FlashMsg` to `ShowHelpMsg`
- `internal/tui/messages.go` — Add `ShowHelpMsg` type

### Content Structure

Help content should be defined as structured data (slices of section/entry structs), not a raw string. This makes content maintainable and allows width-adaptive rendering.

### Decisions

- D-086: Dedicated ViewHelp view mode (not flash, not overlay, not pager)
- D-087: `?` global keybinding for help access
