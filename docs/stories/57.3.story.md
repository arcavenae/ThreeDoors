## Story

As a user,
I want to extract actionable tasks from unstructured text (meeting notes, Obsidian pages, transcripts, clipboard),
So that tasks hiding in prose are surfaced into my ThreeDoors pool without manual entry.

## Background

Task Extraction is the highest-priority LLM service (P0). It converts natural language text into structured tasks. Unlike existing pattern-matching adapters (which look for checkboxes), LLM extraction understands intent: "I should probably email Sarah about the budget" becomes a task.

All sources reduce to `extractFromText(ctx, text) → []ExtractedTask` (S3-D1). Source-specific methods just handle getting the text blob. The extraction prompt requests JSON-only output for reliable parsing (S3-D3).

**Party mode decision (S1-D1):** Task Extraction is P0. LLMs understand intent; regex only catches format.
**Party mode decision (S3-D2):** User review required before import. LLMs hallucinate.
**Party mode decision (S3-D5):** 32KB input size limit for MVP. Chunking deferred.

## Acceptance Criteria

**AC-1: TaskExtractor service created**
**Given** the `internal/intelligence/services/` package (new)
**When** `TaskExtractor` is initialized with an `LLMBackend`
**Then** it exposes `ExtractFromText(ctx, text) ([]ExtractedTask, error)`

**AC-2: ExtractedTask struct defined**
**Given** the services package
**When** extraction completes
**Then** each `ExtractedTask` has: Text (string), Effort (int 1-5), Tags ([]string), Source (string), Confidence (float64)

**AC-3: Extraction prompt produces parseable output**
**Given** a text blob with embedded tasks
**When** the extraction prompt is sent to an LLM backend
**Then** the response is a JSON array of task objects
**And** each task has imperative-form text, effort estimate, and tags

**AC-4: Source-specific extraction helpers**
**Given** the TaskExtractor
**Then** `ExtractFromFile(ctx, path)` reads a file and calls extractFromText
**And** `ExtractFromClipboard(ctx)` reads clipboard via pbpaste and calls extractFromText

**AC-5: Input validation**
**Given** an extraction request
**When** input is empty
**Then** a descriptive error is returned
**When** input exceeds 32KB
**Then** an error with the size limit is returned

**AC-6: Response parsing with retry**
**Given** an LLM response
**When** JSON parsing fails
**Then** one retry with a stricter prompt is attempted (S3-D8)
**And** if the retry also fails, a parse error is returned

**AC-7: Deduplication check**
**Given** extracted tasks
**When** compared against the existing task pool
**Then** potential duplicates are flagged (not silently dropped)
