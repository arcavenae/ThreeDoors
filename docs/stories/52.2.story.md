## Story

As the supervisor or envoy agent,
I want the operations guide to include decision flowcharts for each firewall layer,
So that the screening process is unambiguous and produces consistent results across agent restarts.

## Background

The current `docs/envoy-operations.md` documents the patrol cycle, communication protocols, and authority matrix. After Story 52.1 restructures the agent definition into three layers, the operations guide needs corresponding decision flowcharts that map each issue through the layer pipeline.

## Acceptance Criteria

**Given** the operations guide at `docs/envoy-operations.md`
**When** a reader looks for how issues flow through the firewall
**Then** there is a top-level flowchart showing: Issue → Layer 1 → (resolved? stop : Layer 2) → (resolved? stop : Layer 3 recommendation)

**Given** the Layer 1 section of the operations guide
**When** the envoy processes a new issue
**Then** there is a decision tree for each gate:
- Spam gate: body length check → pattern match → action (close + notify)
- Duplicate gate: title match → keyword match → symptom match → action (flag, don't close)
- Already-fixed gate: PR cross-reference → action (comment + recommend closure)
- Previously-decided gate: BOARD.md search → SOUL.md pattern match → action (cite decision)

**Given** the Layer 2 section
**When** an issue reaches Layer 2
**Then** there is a decision tree covering: alignment classification → authority routing → classification/labeling → scope assessment → relay/decline

**Given** the Layer 3 section
**When** the envoy determines an issue warrants BMAD deliberation
**Then** there is a checklist of criteria and a template for the supervisor escalation message that includes:
- Which criteria were met
- Which agents should participate
- What question(s) the party mode should address

**Given** all flowcharts
**When** compared against the updated agent definition (Story 52.1)
**Then** the flowcharts are consistent — no undocumented decisions, no missing branches

## Technical Notes

- Decision flowcharts use ASCII/text art (mermaid-style or box-drawing) — no external diagram tools
- Each gate/screen should have a concrete example showing how a real issue would flow through it
- Reference existing templates from the current operations guide (milestone updates, staleness alerts)
- This story depends on 52.1 because the operations guide must reflect the restructured agent definition
