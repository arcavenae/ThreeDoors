## Story

As a persistent agent operator,
I want agents to persist critical state before shutdown and restore it on restart,
So that agent restarts don't lose accumulated context and in-progress work tracking.

## Background

Persistent agents (merge-queue, pr-shepherd, envoy, project-watchdog, arch-watchdog, retrospector) lose all context when restarted. This means:
- merge-queue forgets which PRs it was tracking
- pr-shepherd loses rebase state
- envoy forgets triage progress on open issues
- All agents lose understanding of recent decisions and conversations

Chainlink solves this with structured session management: `session start`, `session action "breadcrumb"`, `session end --notes "handoff"`. These breadcrumbs survive context compression and session restarts.

Per Q-C-010, this should be a daemon feature — the daemon already manages agent lifecycle and is the natural place for state persistence. This extends Epic 58's shift-handover work.

## Acceptance Criteria

**AC1: Per-agent state directory structure defined**
**Given** the multiclaude state directory exists
**When** the handoff protocol is designed
**Then** each agent has a state directory at `~/.multiclaude/agent-state/<repo>/<agent-name>/` containing:
  - `handoff.md` — structured handoff notes from last session
  - `session.jsonl` — breadcrumb actions from current/recent sessions
  - `context.json` — machine-readable state (tracked PRs, pending work, etc.)

**AC2: Agent definitions include handoff instructions**
**Given** persistent agent definitions exist in `agents/*.md`
**When** handoff protocol sections are added
**Then** each agent definition specifies:
  - What state to persist before shutdown (agent-specific)
  - How to read handoff notes on startup
  - What breadcrumb actions to log during operation

**AC3: Handoff note format specified**
**Given** the handoff protocol is defined
**When** an agent writes handoff notes
**Then** the notes follow a structured format:
  - `## In Progress` — work currently being done
  - `## Recently Completed` — work finished this session
  - `## Blocked/Waiting` — items awaiting external action
  - `## Key Decisions` — decisions made this session that affect future work
  - `## Warnings` — issues or risks the next session should know about

**AC4: Startup reads handoff notes**
**Given** an agent restarts and handoff notes exist
**When** the daemon spawns the agent
**Then** the initial prompt includes the contents of `handoff.md` so the agent starts with context

**AC5: Design document for daemon integration**
**Given** the full handoff protocol is specified
**When** the design is documented
**Then** it includes:
  - How the daemon triggers handoff note writing before agent shutdown
  - How the daemon injects handoff notes into agent startup
  - How breadcrumb logging works (agent writes to session.jsonl)
  - State file cleanup/rotation policy

## Tasks

### Task 1: Design the state directory structure
Define the per-agent state directory layout, file formats, and rotation policy.

### Task 2: Define agent-specific state requirements
For each persistent agent, document what state needs to be persisted:
- merge-queue: tracked PRs, merge status, CI check results
- pr-shepherd: rebase queue, conflict state, branch tracking
- envoy: triage progress, issue assignments, pending responses
- project-watchdog: epic/story sync state, pending updates
- arch-watchdog: review queue, flagged patterns
- retrospector: analysis state, pending reports

### Task 3: Create handoff note template
Write the structured handoff note format with sections for in-progress, completed, blocked, decisions, and warnings.

### Task 4: Design daemon integration
Document how the daemon:
- Sends a "prepare for shutdown" signal before killing an agent
- Reads handoff.md and injects it into the startup prompt
- Manages state file lifecycle (cleanup, rotation)

### Task 5: Update agent definitions with handoff sections
Add handoff protocol sections to all 6 persistent agent definitions specifying what each agent should persist.

### Task 6: Create prototype handoff for one agent
Implement the handoff protocol manually for merge-queue as a proof of concept, using the current startup prompt mechanism (before daemon integration).
