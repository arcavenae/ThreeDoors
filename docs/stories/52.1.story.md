## Story

As the supervisor,
I want the envoy agent definition to formalize a three-layer firewall architecture,
So that issue screening follows predictable, layered decision paths with clear exit criteria at each stage.

## Background

The current `agents/envoy.md` describes a flat "screen and relay" model. The party mode research (8 sessions, P-002) recommended structuring this into three explicit layers:

1. **Layer 1 (Non-LLM Gates):** Deterministic checks — spam, duplicates, already-fixed, previously-decided. Fast-path exits that don't need AI reasoning.
2. **Layer 2 (Lightweight AI Screening):** SOUL.md alignment, authority tier routing, classification, labeling, scope assessment. The envoy's core reasoning step.
3. **Layer 3 (BMAD Deliberation):** Recommendation to supervisor for party mode when issues are architecturally complex, direction-changing, or systemic.

The agent definition needs restructuring so the envoy processes issues through these layers in sequence, with clear exit criteria at each layer.

## Acceptance Criteria

**Given** the envoy agent definition at `agents/envoy.md`
**When** a developer reads the Screening section
**Then** it is structured into three clearly labeled layers (Layer 1, Layer 2, Layer 3) with entry/exit criteria for each

**Given** the Layer 1 section
**When** the envoy encounters a new issue
**Then** it has documented criteria for four deterministic gates: spam detection, duplicate detection, already-fixed detection, and previously-decided detection

**Given** the Layer 2 section
**When** an issue passes Layer 1 without resolution
**Then** the definition specifies four screening steps: SOUL.md alignment, authority tier routing, classification/labeling, and scope assessment

**Given** the Layer 3 section
**When** an issue is complex enough to warrant multi-agent review
**Then** the definition specifies criteria for recommending BMAD party mode to the supervisor, including which agent perspectives to suggest

**Given** any layer
**When** processing completes
**Then** the definition specifies exactly what action the envoy takes (close, flag, decline, relay, or recommend escalation) and what messages are sent

**Given** the updated agent definition
**When** the envoy is restarted with the new definition
**Then** the definition is backwards-compatible — existing patrol cycle, tracker file format, and communication protocols remain unchanged

## Technical Notes

- This is a documentation/agent-definition change only. No Go code.
- The three-layer structure wraps the existing authority matrix and screening rules — it doesn't replace them.
- Layer 1 gates correspond to the existing "Screen OUT" capabilities in the current definition.
- Layer 2 corresponds to the existing screening + SOUL.md alignment work.
- Layer 3 is new — formalizing when the envoy should recommend party mode.
- After updating the definition, the envoy agent MUST be restarted (Claude cannot hot-reload system prompts).
