# Story 5.3: DMG/pkg Installer for macOS

## Status: Not Started

**Epic:** 5 — macOS Distribution & Packaging
**Priority:** P2
**Story Points:** 3
**Depends On:** Story 5.1 (complete — code signing & Homebrew)
**Blocks:** None
**FRs:** FR25

## User Story

As a macOS user who prefers graphical installation,
I want a .pkg installer available for download,
So that I can install ThreeDoors without using the terminal or Homebrew.

## Acceptance Criteria

- AC1: CI generates a signed .pkg installer containing the signed+notarized binary
- AC2: The .pkg installs `threedoors` to `/usr/local/bin/`
- AC3: The .pkg is also notarized with Apple's notarization service
- AC4: The installer is uploaded to GitHub Releases alongside the raw binaries
- AC5: Double-clicking the .pkg on macOS launches the standard macOS installer UI
- AC6: The .pkg works on both Apple Silicon and Intel Macs (universal binary or architecture-specific packages)
- AC7: Uninstall path documented (manual removal of `/usr/local/bin/threedoors`)
- AC8: CI workflow step is idempotent — re-running release doesn't produce duplicate assets

## Technical Notes

- Use `pkgbuild` and `productbuild` (available on macOS CI runners) to create the .pkg
- Sign the .pkg with Developer ID Installer certificate (separate from Developer ID Application used for the binary)
- Notarize the .pkg separately from the binary using `xcrun notarytool`
- .pkg is preferred over .dmg for CLI tools — no drag-to-install UX needed
- The existing release workflow (`.github/workflows/release.yml`) already handles binary signing and notarization — extend it with a .pkg build step
- Consider producing architecture-specific .pkg files (arm64, amd64) rather than a universal binary .pkg to keep download size small
