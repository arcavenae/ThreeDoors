## Story

As a multiclaude operator,
I want the retrospector to checkpoint its analytical state periodically and restore from checkpoints on restart,
So that pattern detection survives across restarts and context exhaustion doesn't lose analytical progress.

## Background

The retrospector runs out of context after processing ~45 PRs. The current "Context Exhaustion Protocol" says to message the supervisor and request restart, then rebuild state from the JSONL findings log. But:
- The JSONL log only stores raw data points (per-PR observations), not derived analytical state
- Rolling windows (e.g., "CI failure rate over last 10 PRs") must be recomputed from scratch
- Mode rotation position is lost — the agent doesn't know which deep analysis mode it ran last
- The agent definition itself is ~350 lines, consuming significant context budget before any work begins

The fix adds a checkpoint file for derived state and optimizes the definition to reduce context consumption.

## Acceptance Criteria

**Given** the retrospector has processed 5 PRs since last checkpoint (or 2 hours have elapsed)
**When** the checkpoint trigger fires
**Then** it writes analytical state to `docs/operations/retrospector-checkpoint.json`
**And** the checkpoint includes: last PR number, mode rotation index, rolling window values, hours since restart, PRs since restart

**Given** the retrospector starts up (fresh spawn or restart)
**When** it runs its startup sequence
**Then** it reads `docs/operations/retrospector-checkpoint.json` FIRST
**And** restores mode rotation position, rolling window state, and last-processed-PR pointer
**And** then reads the JSONL findings log only for entries AFTER the checkpoint's last PR
**And** catches up on any new merges since the checkpoint

**Given** the checkpoint file doesn't exist (first-ever run)
**When** the retrospector starts
**Then** it falls back to full JSONL rebuild (existing behavior)
**And** creates the checkpoint file after processing its first batch

**Given** the retrospector definition is optimized for context budget
**When** compared to the pre-optimization version
**Then** verbose inline documentation is moved to referenced files where possible
**And** the JSONL schema example is reduced to 2 lines (was potentially more)
**And** the Dual-Loop Architecture section is condensed to essential directives
**And** the total definition size is reduced by ≥15%

## Technical Notes

- Checkpoint file: `docs/operations/retrospector-checkpoint.json`
- Checkpoint schema:
  ```json
  {
    "version": 1,
    "last_pr": 608,
    "last_timestamp": "2026-03-12T14:30:00Z",
    "mode_rotation_index": 2,
    "hours_since_restart": 3.5,
    "prs_since_restart": 12,
    "rolling_windows": {
      "ci_failure_rate_10pr": 0.3,
      "conflict_rate_10pr": 0.1,
      "rebase_avg_10pr": 1.5
    },
    "checkpoint_timestamp": "2026-03-12T14:30:00Z"
  }
  ```
- The checkpoint is NOT a replacement for the JSONL log — it's a cache of derived state
- On restart: checkpoint first (fast), then JSONL for delta (incremental), then new merges (catch-up)
- Checkpoint writes should use atomic write pattern (write to .tmp, rename) — but since this is an agent definition instruction (not Go code), describe the intent and let the agent implement appropriately
- Context budget optimization should NOT remove safety-critical content (Watchmen safeguards, anti-prompting guardrail, authority table)
