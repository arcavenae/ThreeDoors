## Story

As a multiclaude system,
I want a library that reads Claude Code JSONL session files and extracts token usage data,
So that downstream tools can estimate consumption without depending on unavailable API endpoints.

## Background

Claude Code on Max plan has NO programmatic quota API. The only reliable local data source is JSONL session files at `~/.claude/projects/<project-hash>/<session-id>.jsonl`. Each entry contains input/output token counts per interaction. This story builds the foundational parser that all other quota monitoring stories depend on.

Key constraint from R-004: Max plan provides no intermediate throttle signal — it's binary (messages accepted or blocked). All monitoring must be self-imposed and heuristic-based.

## Acceptance Criteria

**AC1: JSONL file discovery and reading**
**Given** Claude Code session files exist at `~/.claude/projects/`
**When** the parser is invoked with a config directory path
**Then** it discovers all JSONL session files, reads them, and extracts token counts (input + output) per interaction

**AC2: Time-windowed aggregation**
**Given** parsed token data with timestamps
**When** aggregated over a configurable time window
**Then** returns total tokens consumed in the current rolling 5-hour window, plus per-session breakdown

**AC3: Graceful error handling**
**Given** malformed, incomplete, or empty JSONL entries
**When** the parser encounters them
**Then** it skips with a warning log (not crash), continues processing remaining entries

**AC4: Plan budget estimation**
**Given** total tokens consumed in the current window
**When** compared against configurable plan budgets
**Then** returns usage as percentage of estimated budget (default: Max 5x ~88K, Max 20x ~220K)

**AC5: Structured output**
**Given** parsed and aggregated data
**When** returned to callers
**Then** provides Go structs with: total tokens, input/output breakdown, usage percentage, window start time, sessions included

**AC6: Test coverage**
**Given** the parser library
**When** tested
**Then** unit tests cover: normal multi-entry files, empty files, malformed entries, multi-session aggregation, window boundary cases

**AC7: CLAUDE_CONFIG_DIR support**
**Given** a non-default `CLAUDE_CONFIG_DIR` is set
**When** the parser is invoked
**Then** it reads session files from the overridden config directory (Marvel portability)

## Tasks

### Task 1: Define Go structs for token usage data
Design `UsageSnapshot`, `SessionUsage`, `WindowUsage` structs. Include input/output token breakdown, timestamps, percentages.

### Task 2: Implement JSONL file discovery
Walk `<config-dir>/projects/` to find all `.jsonl` files. Support both default `~/.claude/` and `CLAUDE_CONFIG_DIR` override.

### Task 3: Implement JSONL parser
Read entries, extract token counts and timestamps. Handle malformed entries gracefully.

### Task 4: Implement time-windowed aggregation
Rolling 5-hour window calculation. Sum tokens within window, calculate percentage of plan budget.

### Task 5: Write unit tests with fixture data
Create `testdata/` fixtures for: normal, empty, malformed, multi-session cases. Table-driven tests.

## Dev Notes

- Package location: `internal/quota/` or `scripts/quota/` — prefer `internal/quota/` for Go library reuse
- JSONL format: each line is a JSON object. Token fields vary by Claude Code version — inspect real files for schema
- The Claude-Code-Usage-Monitor uses P90 percentile analysis over 192 hours (8 days) — we can start simpler with direct summation
- Consider making the window duration configurable (default 5h) for future flexibility
- This is a read-only library — it never modifies JSONL files
