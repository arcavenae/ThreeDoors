# Story 39.9: Inline Hint Rendering Infrastructure

## Status: Done (PR #374)

**Epic:** 39 — Keybinding Display System
**Priority:** P1
**Estimate:** S (1-2 hours)
**Depends On:** 39.1

---

## User Story

As a developer, I want a reusable inline hint rendering system that integrates with the keybinding registry and theme system, so that all views can show contextual key labels on interactive elements.

## Context

Epic 39 stories 39.1-39.3 built the keybinding bar (bottom strip) and overlay (full reference). These are *external reference panels*. This story creates the infrastructure for *inline hints* — key labels rendered directly on/near the interactive elements themselves. Think of `[a]` rendered on a door frame like a doorknob label.

Design reference: [Default Tooltips Party Mode](../../_bmad-output/planning-artifacts/default-tooltips-mode-party-mode.md)

## Acceptance Criteria

### AC-1: Inline Hint Renderer
- [ ] `renderInlineHint(key string, enabled bool) string` function in a new file `internal/tui/inline_hints.go`
- [ ] When enabled: returns Lipgloss-styled `[key]` string (dim foreground, subtle bracket styling)
- [ ] When disabled: returns empty string
- [ ] Hint style uses ANSI 245 (dim) by default, ANSI 240 (extra dim) for fade mode

### AC-2: Theme Render Signature Extension
- [ ] `DoorTheme.Render()` signature extended to accept `hint string` parameter
- [ ] All existing theme implementations updated: Classic, Modern, Sci-Fi, Shoji, Gothic, Minimal (and any others)
- [ ] When `hint` is empty string, door renders exactly as before (backward compatible)
- [ ] When `hint` is non-empty, it renders in the bottom-right area of the door frame (doorknob position)

### AC-3: Config Model Extension
- [ ] `show_inline_hints` field added to config model, default `true`
- [ ] `inline_hints_session_count` field added to config model, default `0`
- [ ] `inline_hints_fade_threshold` field added to config model, default `5`
- [ ] Fields read on startup, persisted on change via existing atomic write pattern

### AC-4: Session Counter Logic
- [ ] Session count incremented on each app launch (in MainModel init)
- [ ] When `inline_hints_session_count >= inline_hints_fade_threshold`: `show_inline_hints` set to `false` automatically
- [ ] When auto-disabled: one-time flash message "Key hints hidden — type :hints to bring them back"
- [ ] Session N-1 (one before threshold): hints render with extra-dim style (ANSI 240) as visual fade cue

### AC-5: `:hints` Command Registration
- [ ] `:hints on` enables inline hints and resets session counter
- [ ] `:hints off` disables inline hints
- [ ] `:hints` (no argument) toggles current state
- [ ] Command registered in existing command dispatch system

### AC-6: Tests
- [ ] `renderInlineHint` returns correct output for enabled/disabled states
- [ ] Theme Render() backward compatibility: empty hint produces identical output
- [ ] Config round-trip: write and read inline hint fields
- [ ] Session counter logic: threshold triggers auto-disable
- [ ] Fade styling applied at correct session count
- [ ] Table-driven tests following project testing standards

## Technical Notes

- New file: `internal/tui/inline_hints.go`
- New file: `internal/tui/inline_hints_test.go`
- Theme signature change touches `internal/tui/themes/` — each theme's `Render()` method
- Config changes in whatever file manages config.yaml (follow existing pattern from 39.4)
- The `renderInlineHint` function is intentionally simple — it's a styled string builder, not a component
- Inline hints are completely independent of the keybinding bar (`show_keybinding_bar`). See D-125.
