# Story 63.4: ClickUp Contract Tests & Integration Testing

## Status: Not Started

**Priority:** P2
**Story Points:** 3
**Depends On:** Story 63.2
**Blocks:** None
**FRs:** product-scope.md Phase 5 (ClickUp integration)

## User Story

As a developer,
I want contract tests that verify the ClickUp adapter against the TaskProvider interface,
So that I can catch integration regressions without hitting the live ClickUp API.

## Acceptance Criteria

- AC1: Contract test suite in `internal/adapters/clickup/clickup_contract_test.go` validates `TaskProvider` interface compliance
- AC2: Tests cover: LoadTasks, SaveTask (status update), HealthCheck, and error paths
- AC3: Golden file tests for ClickUp API response parsing (canned JSON → expected Task structs)
- AC4: Fuzz tests for malformed ClickUp API responses (invalid JSON, missing fields, unexpected types)
- AC5: `httptest.NewServer` integration tests covering the full provider lifecycle (auth → load → sync → error)
- AC6: Test fixtures in `internal/adapters/clickup/testdata/` with representative ClickUp API responses
- AC7: Race detector passes: `go test -race ./internal/adapters/clickup/...`
- AC8: All tests are table-driven where applicable (>2 test cases)
- AC9: Benchmark test for field mapping performance with 100+ tasks

## Technical Notes

- Follow the contract test pattern from `internal/adapters/todoist/todoist_contract_test.go`
- Golden files for API response parsing prevent silent field mapping regressions
- Fuzz testing catches edge cases in ClickUp's occasionally inconsistent API responses
- Benchmark test establishes baseline for field mapping — important for workspaces with large task counts
