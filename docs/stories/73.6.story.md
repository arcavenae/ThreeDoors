## Story

As a multiclaude platform maintainer,
I want heartbeats to be a daemon-managed feature with configurable per-agent intervals,
So that heartbeats survive session restarts, don't inject into the supervisor, and can be tuned per agent type.

## Background

The current heartbeat mechanism has two layers:
1. **Daemon wake loop** (`wakeAgents()`, daemon.go:432): Nudges every agent every 2 minutes with role-specific prompts
2. **CronCreate jobs**: Created by the supervisor to send HEARTBEAT messages to specific agents on varying intervals (7-23 min)

Story 73.2 removes the CronCreate layer. This story enhances the daemon wake loop to be the single, authoritative heartbeat mechanism with:
- Configurable per-agent intervals (persistent agents need different cadences than watchdogs)
- Message-based delivery (write to message queue) rather than tmux paste-buffer injection
- Survives all restarts (daemon state, not session state)

## Acceptance Criteria

**AC1: Design document for daemon-native heartbeats**
**Given** the current wake loop implementation is understood
**When** the design is documented
**Then** it specifies:
  - Per-agent interval configuration (in `multiclaude config` or per-repo config)
  - Recommended intervals by agent type (e.g., merge-queue: 5 min, retrospector: 15 min)
  - Delivery mechanism (message queue vs tmux paste-buffer)
  - Deduplication logic (don't nudge an agent that's actively working)

**AC2: Agent activity detection specified**
**Given** the daemon tracks agent state
**When** the heartbeat fires
**Then** the design includes logic to skip nudging agents that are actively working
  - Check: agent has processed a message in the last N minutes
  - Check: agent's tmux pane shows recent output (activity heuristic)
  - Avoids: interrupting an agent mid-task with a redundant nudge

**AC3: Configuration schema defined**
**Given** the daemon needs per-agent intervals
**When** the configuration is designed
**Then** it supports:
  ```yaml
  heartbeats:
    default_interval: 5m
    agents:
      merge-queue: 5m
      pr-shepherd: 5m
      envoy: 10m
      project-watchdog: 15m
      arch-watchdog: 20m
      retrospector: 15m
  ```

**AC4: Migration path from CronCreate documented**
**Given** Story 73.2 removes CronCreate heartbeats
**When** daemon-native heartbeats are designed
**Then** the migration path is clear:
  - Phase 1 (73.2): Remove CronCreate, rely on existing 2-min wake loop
  - Phase 2 (73.6): Enhance wake loop with per-agent intervals and smart delivery
  - No gap period where agents receive no heartbeats

**AC5: SYNC_OPERATIONAL_DATA handled**
**Given** the SYNC_OPERATIONAL_DATA cron served a specific purpose
**When** daemon-native heartbeats are designed
**Then** operational data sync triggers are included in the design (either as a heartbeat type or separate daemon feature)

## Dev Notes

- This story produces a DESIGN DOCUMENT — the actual daemon code changes happen in the multiclaude codebase
- The current 2-min interval is likely too aggressive for some agents (retrospector, arch-watchdog) and about right for others (merge-queue, pr-shepherd)
- Prime-number intervals (7, 11, 13, 23 min) were used in CronCreate to prevent heartbeat storms — consider whether this is still needed with daemon-managed scheduling
- The daemon already has `LastNudge` tracking per agent — this is the foundation for smart delivery
- Marvel portability: Daemon-native heartbeats are a core platform feature — this design should be platform-agnostic
