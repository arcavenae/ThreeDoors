## Story

As a user,
I want ThreeDoors to verify that its data directory and files are not symlinks before reading or writing,
So that a malicious local user cannot redirect my file operations to arbitrary locations via symlink attacks.

## Background

The codebase has zero calls to `os.Lstat()`, `filepath.EvalSymlinks()`, or any symlink detection for the core data directory. An attacker who creates `~/.threedoors/` before the victim's first run could plant symlinks pointing to sensitive files. The Obsidian adapter has excellent path traversal protection in `sanitizeDailyNotePath()`, but this is not applied to the main data directory.

**Attack scenario:** Attacker creates `~/.threedoors/tasks.yaml` as a symlink to `/etc/some-file`. Victim runs ThreeDoors, which writes task data through the symlink to the attacker-controlled target.

## Acceptance Criteria

**AC-1: Directory symlink check on startup**
**Given** ThreeDoors starts
**When** it resolves the data directory path
**Then** it uses `os.Lstat()` to verify `~/.threedoors/` is not a symlink
**And** if it is a symlink, it returns an error and refuses to start

**AC-2: File symlink check before writes**
**Given** ThreeDoors is about to write a data file (tasks.yaml, config.yaml, sync.log, etc.)
**When** the target path already exists
**Then** it uses `os.Lstat()` to verify the file is not a symlink before writing
**And** if it is a symlink, it returns an error instead of writing

**AC-3: Directory ownership check**
**Given** ThreeDoors starts
**When** it verifies the data directory
**Then** it checks that the directory owner matches the current user (via `os.Stat().Sys()`)
**And** if ownership doesn't match, it returns a warning (not a hard error, for portability)

**AC-4: Reusable validation helper**
**Given** the validation logic
**When** implemented
**Then** it is encapsulated in a reusable function (e.g., `ValidatePath(path string) error`)
**And** it checks: not a symlink, ownership matches (warning), permissions are restrictive

## Technical Notes

- Add `internal/core/path_validation.go` with `ValidatePath()` and `ValidateDir()` functions
- Call `ValidateDir()` from `config_paths.go` during directory resolution
- Call `ValidatePath()` before atomic writes in file_manager.go, provider_config.go, wal_provider.go, sync_log.go
- Use `os.Lstat()` (not `os.Stat()`) to detect symlinks — `os.Stat()` follows symlinks
- Check `fi.Mode()&os.ModeSymlink != 0` on Lstat result
- On macOS/Linux, use `fi.Sys().(*syscall.Stat_t).Uid` for ownership check
- Consider `O_NOFOLLOW` flag for file opens (platform-dependent, may need build tags)
- Model implementation on `sanitizeDailyNotePath()` from Obsidian adapter
