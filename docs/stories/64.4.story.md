# Story 64.4: Cross-Machine Conflict Resolution

## Status: In Review

**Epic:** 64 — Cross-Computer Sync
**Priority:** P2
**Story Points:** 5
**Depends On:** Story 64.1 (research spike), Story 64.2
**Blocks:** Story 64.6
**FRs:** product-scope.md Phase 5 (Cross-computer sync)

## Refined by Spike (ADR-0034)

Conflict resolution strategy determined: extend ADR-0012 property-level LWW with per-device vector clocks. No CRDTs needed at this scale. Git custom merge driver handles file-level conflicts before application-level resolution.

## User Story

As a ThreeDoors user who edits tasks on multiple computers,
I want conflicts to be resolved automatically (with manual override available),
So that I don't lose work when the same task is modified on two machines.

## Acceptance Criteria

- AC1: Extend `FieldVersion` (ADR-0012) with `DeviceID` field — each field change tracks which device made it
- AC2: Per-task vector clock (`map[DeviceID]uint64`) tracks logical timestamps per device for causal ordering
- AC3: Non-overlapping changes (different fields on same task, different devices) merge automatically — both edits preserved
- AC4: Overlapping changes (same field on same task) resolved by: (a) vector clock causal ordering if one happened-before the other, (b) UTC timestamp comparison if concurrent, (c) device ID lexicographic tiebreaker if timestamps equal
- AC5: Custom Git merge driver registered for `tasks.yaml` — performs task-level three-way merge (common ancestor + device A + device B) using application logic, not Git's line-based diff
- AC6: `sessions.jsonl` uses Git's built-in `union` merge strategy (append-only, no conflicts possible)
- AC7: Rejected versions logged to `~/.threedoors/sync/conflicts.jsonl` with: conflict ID, timestamp, task ID, device IDs, conflicting field values, resolution outcome
- AC8: `threedoors sync conflicts` CLI command lists recent conflicts with resolution details
- AC9: `threedoors sync resolve <conflict-id> --keep <device>` allows manual override — replays the rejected version's field values
- AC10: TUI notification when conflicts are auto-resolved (non-blocking, informational — reuse existing notification infrastructure via `FlashMsg`)
- AC11: No data loss — rejected version is always preserved in conflict log for recovery
- AC12: Unit tests for: non-overlapping merge, overlapping LWW, vector clock ordering, three-way merge, concurrent create, concurrent delete, manual override
- AC13: Stress test: 50+ simultaneous conflicting changes from 3 simulated devices resolve correctly
- AC14: Backward compatibility — graceful handling of legacy task files without `FieldVersions`/`VectorClock` fields (nil/missing = version 0, unknown device)

## Technical Notes

### Architecture Decisions

- **FieldVersion location:** New type in `internal/core/field_version.go` alongside existing `field_conflict_resolver.go`
- **Resolver strategy:** Create `CrossMachineConflictResolver` implementing the existing `ConflictResolver` interface. The existing `DefaultConflictResolver` (metadata→remote, ThreeDoors→local) is for provider sync (Epic 47). The new resolver uses vector clock → timestamp → device ID tiebreaker for cross-machine sync. `SyncEngine` selects resolver based on context.
- **VectorClock + UpdatedAt coexistence:** Both are updated on every field change. Vector clock is primary for cross-machine conflict resolution. `UpdatedAt` remains for provider sync compatibility and UI display.
- **Concurrent delete semantics:** Modify-wins — if Device A deletes task X and Device B modifies task X, the modification is preserved (consistent with AC11 no-data-loss guarantee). Delete only wins if both devices delete.

### Git Merge Driver

- The custom Git merge driver is a ThreeDoors binary mode (`threedoors merge-driver %O %A %B`) registered in `.gitattributes` and `.git/config`
- Three-way merge: parse base (%O), local (%A), remote (%B) as YAML task lists, merge per-task using property-level strategy, write result to %A
- The merge driver operates on file content only — it does NOT access `~/.threedoors/` or the `DeviceRegistry`. Device attribution is embedded in the YAML per-task (via `FieldVersions` and `VectorClock` fields on each task)
- Vector clock increment: each local change increments `vectorClock[localDeviceID]++`

### Conflict Logging

- Conflict log path: `~/.threedoors/sync/conflicts.jsonl` (create `sync/` subdirectory if not exists)
- Append-only JSONL — rotates at 1MB (reuse `SyncLog` rotation pattern from `internal/core/sync_log.go`)
- Cross-machine conflicts are peer-to-peer (no authority), unlike provider sync (Epic 47) where the provider is authoritative

### Manual Override (AC9) Detail

- `threedoors sync resolve <conflict-id> --keep <device>` overwrites ONLY the conflicting fields from the rejected version (not the entire task)
- Creates a new conflict log entry recording the manual override
- Increments the local vector clock to ensure the override propagates on next sync
- The conflict ID references an entry in `conflicts.jsonl`

### Zero-Value / Migration Behavior

- Tasks without `FieldVersions` or `VectorClock` fields are treated as version 0, unknown device
- No backfill migration needed — `omitempty` YAML tags handle serialization
- The merge algorithm treats nil vector clocks as `{}` (empty map) — any non-empty clock happened-after

### Vector Clock Comparison Algorithm

- **Happened-before:** Clock A happened-before Clock B if every entry in A is ≤ the corresponding entry in B, AND at least one is strictly less
- **Concurrent:** Neither happened-before the other (some entries in A > B, some < B)
- **Equal:** All entries match
- When concurrent: fall through to UTC timestamp comparison (AC4b), then device ID lexicographic tiebreaker (AC4c)

### Test Expected Outcomes (AC12)

| Scenario | Input | Expected Output |
|---|---|---|
| Non-overlapping merge | A changes `text`, B changes `effort` | Merged task has both A's text and B's effort |
| Overlapping LWW (causal) | A: clock `{A:3,B:2}`, B: clock `{A:2,B:2}` | A happened-after B → A's value wins |
| Overlapping LWW (concurrent) | A: clock `{A:3,B:2}`, B: clock `{A:2,B:4}` | Concurrent → compare timestamps → later wins |
| Overlapping LWW (timestamp tie) | Same timestamp, A: device `aaa`, B: device `bbb` | `bbb` wins (lexicographic, higher) |
| Three-way merge | Base has task X, A modifies text, B modifies status | Merged has A's text + B's status |
| Concurrent create | A creates task Y, B creates task Z | Both tasks present in merged result |
| Concurrent delete vs modify | A deletes task X, B modifies task X | Modify-wins: task X preserved with B's changes |
| Manual override | Resolve conflict-123 --keep deviceA | Only conflicting fields overwritten from A's version |
| Vector clock merge | A: `{A:3,B:2}`, B: `{A:1,B:5}` | Merged: `{A:3,B:5}` (max of each entry) |
| FieldVersion tracking | Modify task.Text on device A | `FieldVersions["text"]` has DeviceID=A, updated timestamp |
| Empty task list merge | Base has 3 tasks, local=[], remote=[] | Result is empty (both sides deleted) |
| Concurrent adds to empty | Base=[], A adds task Y, B adds task Z | Both Y and Z present |
| Manual override idempotency | Resolve same conflict-id twice | Second call returns "already resolved" error, no state change |
| Malformed YAML in merge driver | %O valid, %A invalid YAML | Exit non-zero, %A file unchanged, no data corruption |
| Conflict log rotation | Write >1MB of entries | Rotation keeps newest half, file size < 1MB |

### Stress Test Specification (AC13)

- Use a deterministic test clock (injectable `time.Time` provider) and fixed seed for reproducibility
- "Simultaneous" means concurrent from a vector clock perspective (no causal ordering between devices)
- 3 simulated devices each make 17+ conflicting changes to overlapping task fields
- Verify: all conflicts resolved, no panics, no data loss, conflict log has all entries

### Task 2: VectorClock Operations (AC2)
Create `internal/core/vector_clock.go`:
- `VectorClock` type alias `map[string]uint64`
- `Increment(deviceID string)` — increments clock for given device
- `Merge(other VectorClock)` — takes max of each entry
- `Compare(other VectorClock) Ordering` — returns `HappenedBefore`, `HappenedAfter`, `Concurrent`, or `Equal`
- `Copy() VectorClock` — deep copy
- **Depends on:** nothing (standalone type)

### Task 3: Three-Way Task Merge Algorithm (AC3, AC4, AC14)
Create `internal/core/cross_machine_merge.go`:
- **3a:** `CrossMachineConflictResolver` implementing `ConflictResolver` interface
- **3b:** Non-overlapping field merge: for each field, compare base→local and base→remote diffs; if only one side changed, take that change
- **3c:** Overlapping field resolution: vector clock comparison → timestamp → device ID tiebreaker
- **3d:** Concurrent create/delete handling: modify-wins, concurrent creates both preserved
- **3e:** Legacy compatibility: nil FieldVersions/VectorClock treated as version 0
- **Depends on:** Tasks 1, 2

### Task 4: Custom Git Merge Driver (AC5)
- **4a:** Create `internal/cli/merge_driver.go` — `threedoors merge-driver %O %A %B` subcommand
- **4b:** Parse three YAML files (base, local, remote) into `[]*Task` lists
- **4c:** Match tasks by ID across all three files, invoke three-way merge per task
- **4d:** Write merged result to `%A` (local file) as YAML
- **4e:** Exit code 0 on success (Git treats as clean merge), non-zero on failure
- **Depends on:** Task 3
