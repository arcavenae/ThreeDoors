## User Story

As a ThreeDoors user who edits tasks on multiple computers,
I want conflicts to be resolved automatically (with manual override available),
So that I don't lose work when the same task is modified on two machines.

## Acceptance Criteria

- AC1: Automatic conflict resolution for non-overlapping changes (different fields on same task)
- AC2: Automatic conflict resolution strategy for overlapping changes (per spike: CRDT, LWW, or hybrid)
- AC3: Conflict metadata logged: which devices conflicted, which fields, which version won, timestamp
- AC4: Conflict log viewable via `threedoors sync conflicts` CLI command
- AC5: Manual override: `threedoors sync resolve <conflict-id> --keep <device>` for unresolved conflicts
- AC6: TUI notification when conflicts are auto-resolved (non-blocking, informational)
- AC7: Conflict resolution integrates with existing field-level strategy from Epic 47
- AC8: No data loss — rejected version stored in conflict log for recovery
- AC9: Unit tests for all conflict scenarios: field-level, task-level, concurrent create, concurrent delete
- AC10: Stress test: 50+ simultaneous conflicting changes resolve correctly

## Technical Notes

- Cross-machine conflicts are fundamentally different from provider sync conflicts (Epic 47)
  - Provider sync: one authoritative source (the provider)
  - Cross-machine sync: no single authority — both devices are equal peers
- Vector clocks or Lamport timestamps may be needed to establish causal ordering
- Consider three-way merge: common ancestor + device A version + device B version
- Conflict log should be append-only JSONL (reuse session log pattern)
