## Story

As a developer,
I want a generic CLI-based LLM backend that implements the existing `LLMBackend` interface via subprocess execution,
So that ThreeDoors can invoke any LLM CLI tool (Claude CLI, Gemini CLI, Ollama CLI, custom) through a uniform abstraction.

## Background

ThreeDoors already has an `LLMBackend` interface with `Complete(ctx, prompt) (string, error)` used by HTTP-based backends (Claude API, Ollama API). This story extends that interface with CLI-based implementations that invoke LLM tools as subprocesses via `os/exec`.

The key abstraction is `CLISpec` — a declarative struct describing how to invoke a specific CLI tool (command, args, input method, output format, timeout). A generic `CLIProvider` executor handles all specs uniformly. Adding a new CLI provider means defining a spec, not writing a class.

The existing `CommandRunner` interface in `internal/dispatch/` should be extended (with stdin support) and shared, not duplicated.

**Party mode decision (S2-D1):** Extend `LLMBackend` with CLI implementations (Option A). Rejected: new CLIService interface (Option B) and capability negotiation (Option C) — `Complete(ctx, prompt)` is sufficient for P0 services.

**Party mode decision (S5-D1):** Declarative CLISpec struct. Rejected: per-provider classes with duplicated exec logic.

## Acceptance Criteria

**AC-1: CLISpec struct defined**
**Given** the `internal/intelligence/llm/` package
**When** the CLISpec type is defined
**Then** it includes: Name, Command, BaseArgs, SystemPrompt (ArgTemplate), OutputFormat (ArgTemplate), InputMethod (stdin/arg/file), Timeout, ResponseParser
**And** ArgTemplate has Flag, Value, and Enabled fields

**AC-2: CLIProvider implements LLMBackend**
**Given** a CLIProvider initialized with a CLISpec
**When** `Complete(ctx, prompt)` is called
**Then** it executes the CLI tool as a subprocess with the correct args
**And** pipes the prompt via the configured InputMethod (stdin or positional arg)
**And** returns the parsed stdout as the response
**And** returns an error wrapping stderr on non-zero exit

**AC-3: CLIProvider.Available checks binary existence**
**Given** a CLIProvider
**When** `Available(ctx)` is called
**Then** it returns true if `exec.LookPath(spec.Command)` succeeds
**And** false otherwise

**AC-4: Pre-built CLISpec factories for known providers**
**Given** the llm package
**When** factory functions are called
**Then** `ClaudeCLISpec()` returns a spec with command "claude", BaseArgs ["--print"], stdin input, --system-prompt and --output-format json support
**And** `GeminiCLISpec()` returns a spec with command "gemini", stdin input, --output-format json support
**And** `OllamaCLISpec(model)` returns a spec with command "ollama", BaseArgs ["run", model], arg input, --system support
**And** `CustomCLISpec(cmd, args)` returns a spec with the given command and args, stdin input

**AC-5: CommandRunner with stdin support**
**Given** the need to pipe prompts to CLI tools
**When** a CommandRunner is used
**Then** it supports `RunWithStdin(ctx, stdin, name, args...) (stdout, stderr, error)`
**And** the pattern is shared with (or derived from) `internal/dispatch/command_runner.go`

**AC-6: Timeout enforcement**
**Given** a CLIProvider with a configured timeout
**When** the CLI tool exceeds the timeout
**Then** the context is cancelled and an error is returned

**AC-7: Error handling**
**Given** a CLI invocation
**When** the tool returns a non-zero exit code
**Then** the error includes the CLI name, stderr content, and the underlying error
**When** the tool returns empty stdout
**Then** `ErrEmptyResponse` is returned
**When** an empty prompt is passed
**Then** `ErrEmptyPrompt` is returned
