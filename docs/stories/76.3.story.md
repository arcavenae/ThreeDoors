## Story

As a multiclaude supervisor,
I want automatic warnings when token consumption approaches plan limits,
So that I can proactively manage agent activity before hitting hard blocks.

## Background

The Max plan has no intermediate throttle — it's binary (messages accepted or blocked until window resets). R-004 proposes four warning tiers at 70%/80%/90%/95% with graduated recommended actions. This story implements the warning engine. **CRITICAL: This engine is advisory-only. It NEVER blocks, throttles, or kills agents.**

## Acceptance Criteria

**AC1: Configurable warning thresholds**
**Given** the warning engine configuration
**When** thresholds are set
**Then** supports at least 4 configurable tiers (default: 70%, 80%, 90%, 95%) with per-tier labels (green/yellow/orange/red)

**AC2: Warning message generation**
**Given** current usage exceeds a threshold
**When** a warning is generated
**Then** it includes: current usage %, estimated remaining tokens, time until window reset, recommended action (suggestion only)

**AC3: Supervisor notification delivery**
**Given** a warning is generated
**When** delivered
**Then** sent via `multiclaude message send supervisor "QUOTA_WARNING: ..."` (or stdout for CLI mode)

**AC4: NEVER blocking**
**Given** usage reaches any threshold, including 95%+
**When** the engine runs
**Then** it ONLY sends advisory messages — no blocking, throttling, killing, or pausing of agents

**AC5: Graduated recommended actions (suggestions only)**
**Given** different threshold tiers
**When** warnings are generated
**Then** suggestions are appropriate: 70% → "Consider monitoring closely", 80% → "Consider reducing heartbeat frequency", 90% → "Consider pausing non-critical agents", 95% → "Critical — consider pausing all but P0 work"

**AC6: Time-of-day awareness**
**Given** current time is during Anthropic peak hours (05:00-11:00 PT)
**When** thresholds are evaluated
**Then** they shift ~20% more aggressive (e.g., 70% threshold triggers at ~56% during peak)

**AC7: Configuration via YAML or environment**
**Given** the engine needs configuration
**When** configured
**Then** accepts plan type, custom thresholds, peak hours, and notification method via YAML config or env vars

**AC8: Threshold tier tests**
**Given** the warning engine
**When** tested
**Then** tests cover each threshold tier, peak vs off-peak behavior, and edge cases (exactly at threshold, just below, just above)

## Tasks

### Task 1: Define threshold configuration schema
YAML config struct: plan type, tiers (percentage + label + recommended action), peak hours, notification method.

### Task 2: Implement threshold evaluator
Takes a `WindowUsage` from 76.1, evaluates against configured thresholds, returns the active tier.

### Task 3: Implement time-of-day awareness
PT timezone conversion, peak hour detection, threshold adjustment during peak.

### Task 4: Implement warning message formatter
Generate human-readable warning messages with all required fields (usage %, remaining, reset time, suggestion).

### Task 5: Implement notification delivery
`multiclaude message send` integration for supervisor delivery. Stdout fallback for CLI mode.

### Task 6: Write tests for all tiers and time-of-day scenarios
Table-driven tests covering each tier, peak/off-peak, boundary conditions.

## Dev Notes

- The "recommended actions" are SUGGESTIONS ONLY — they describe what a human might do, not what the system will do
- Future Phase 2 may add automated responses (adaptive heartbeats) but Phase 1 is strictly advisory
- Peak hour awareness: Anthropic reduced session limits during 05:00-11:00 PT in March 2026
- Consider making the peak hour window configurable in case Anthropic changes it
- PT timezone = America/Los_Angeles
