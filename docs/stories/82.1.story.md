# Story 82.1: Fix Envoy Scope Misstatements in Agent and Governance Docs

**Epic:** 82 — Agent Definition Accuracy
**Status:** Not Started
**Priority:** P1
**Estimate:** S (< 1 hour)

## User Story

As a multiclaude operator or agent,
I want agent definitions and governance docs to accurately describe envoy's scope,
So that agents don't attempt actions outside their authority and operators don't misroute work to envoy.

## Context

Two documents incorrectly describe envoy's capabilities:

1. **`agents/supervisor.md:69`** — Says "Envoy runs the BMAD pipeline — you provide scope decisions." This is wrong. Envoy is issues-board ONLY — it triages, labels, and relays issues. It does NOT run BMAD pipelines, party mode, or any implementation pipeline. The supervisor runs BMAD pipelines (or delegates to workers).

2. **`docs/architecture/governance-model.md:38`** — Says envoy can "Welcome, triage, create stories; cannot make scope decisions." The "create stories" part is wrong. Envoy's own definition (`agents/envoy.md:287`) explicitly lists "Create stories or PRs" under "What You Do NOT Do" and "Create stories or docs PRs" under "CANNOT (Forbidden)".

These misstatements can cause:
- Supervisor routing BMAD pipeline work to envoy (which envoy can't execute)
- Other agents expecting envoy to create story files (which it won't do)
- Governance audits showing inconsistencies between docs

## Acceptance Criteria

- **AC1:** `agents/supervisor.md` line 69 interaction protocol with envoy is corrected: envoy screens and relays issues, supervisor decides triage approach (not "envoy runs BMAD pipeline")
- **AC2:** `docs/architecture/governance-model.md` line 38 envoy row is corrected: "Welcome, triage, label, relay; cannot make scope decisions or create stories"
- **AC3:** No other documents contain the misstatement that envoy creates stories or runs BMAD pipelines (grep verification)
- **AC4:** Changes are consistent with envoy.md's existing "What You Do NOT Do" section and authority table

## Technical Notes

- `agents/envoy.md` is already correct — it clearly states envoy cannot create stories, run BMAD, or authorize work
- The fix is to make supervisor.md and governance-model.md consistent with envoy.md (the authoritative definition)
- These are CODEOWNERS-protected files (`agents/`, `docs/architecture/`) — PR will need @skippy review

## Out of Scope

- Changes to envoy.md itself (it's already correct on scope)
- Changes to envoy's actual behavior or authority
- Other agent definition inconsistencies not related to envoy scope
