# Story 34.4: Retroactive Story DRY & Spec Alignment

**Status:** Done (PR #230)
**Epic:** 34 — SOUL.md + Custom Development Skills
**Depends On:** 34.2 (Custom Slash Commands), 34.3 (Story Template & Integration Notes)

## User Story

As a developer (human or AI agent),
I want existing story files cleaned up to follow the DRY principles established in Stories 34.2-34.3,
So that story files contain only story-specific content, reducing noise and ensuring the layered information model (SOUL.md → CLAUDE.md → slash commands → stories) works retroactively.

## Acceptance Criteria

**Given** that CLAUDE.md now contains project-level coding standards, pre-PR checklist, and design patterns, and the DRY story approach is documented in `docs/architecture/ai-tooling-integration.md`
**When** existing story files are retroactively cleaned up
**Then:**

- AC1: All story files that embed pre-PR submission checklists have those checklists removed and replaced with a reference to CLAUDE.md or `/pre-pr`
- AC2: Quality Gate sections in story files are simplified to contain only story-specific quality items, with a single reference to CLAUDE.md for standard checks (formatting, linting, testing, race detection)
- AC3: Embedded coding standards reminders (error wrapping, timestamp UTC, no init(), TUI rules) are removed from story files — CLAUDE.md is the single source of truth
- AC4: Design pattern reminders (atomic writes, factory pattern, provider pattern, MVU) are removed from story files — CLAUDE.md is the single source of truth
- AC5: Testing standards reminders (table-driven tests, t.Helper(), t.Cleanup(), t.Parallel()) are removed from story files — CLAUDE.md is the single source of truth
- AC6: No story-specific content is lost — only content duplicated from CLAUDE.md is removed
- AC7: NFR-DX6 compliance verified — specs reflect code reality (living documentation)

## NOT In Scope

- Changes to CLAUDE.md, SOUL.md, or slash command files
- Changes to Go source code
- Creating new stories or modifying story acceptance criteria
- Modifying the `/new-story` template (done in 34.3)
- Full spec-vs-code audit for every completed story (NFR-DX6 is established as policy; deep per-story alignment is ongoing)

## Quality Gate

- All standard checks pass (see CLAUDE.md)
- No story-specific content accidentally removed
- Completed story statuses preserved
- Valid markdown with no broken links
- Scope-checked against this story — no extras

## Architecture & Design

This is a documentation-only story. The approach:

1. **Batch cleanup by violation type** — process all files for each violation category in one pass
2. **Preserve story-specific QG items** — only remove items that duplicate CLAUDE.md
3. **Use consistent replacement pattern** — "All standard checks pass (see CLAUDE.md)" for Quality Gate baseline
4. **Leave completed story statuses untouched** — only modify content sections, never status/metadata

## Testing

No code changes — documentation only. Validation:
- Grep for known DRY violation patterns post-cleanup to confirm removal
- Spot-check 5+ stories to verify no story-specific content was lost

## Dependencies

- Story 34.2 (Custom Slash Commands) — slash commands must exist for references
- Story 34.3 (Integration Notes) — DRY approach must be documented

3. Remove embedded coding standards reminders
4. Remove design pattern reminders
5. Remove testing standards reminders
6. Verify no story-specific content was lost (spot-check)
7. Grep-validate that violations are eliminated
