## Provisional Notice

## User Story

As a ThreeDoors user with multiple computers,
I want my task data to transfer reliably between machines,
So that I can work on any computer and see my latest tasks.

## Acceptance Criteria

- AC1: `SyncTransport` interface defined with `Push(ctx, changeset) error` and `Pull(ctx, since) (changeset, error)` methods
- AC2: At least one concrete transport implementation (per spike findings: git-based, cloud, or P2P)
- AC3: Transport handles network failures gracefully — queues changes for retry, doesn't lose data
- AC4: Transport respects bandwidth constraints — delta sync (only changed data), not full snapshot
- AC5: Transport supports encryption in transit (TLS or equivalent)
- AC6: Sync trigger: automatic on task change (debounced, 30s default) + manual via `threedoors sync` command
- AC7: `threedoors sync` CLI command shows sync status, last sync time, and pending changes
- AC8: Sync progress surfaced in TUI status bar (reuse Epic 44 status bar infrastructure)
- AC9: Transport is pluggable — additional transports can be added without modifying core sync logic
- AC10: Integration tests with simulated two-device scenario

## Technical Notes

- The transport layer is the "how" of sync — it moves changesets between devices
- Changesets should be opaque to the transport — just bytes with metadata
- Debounced auto-sync prevents thrashing during rapid task edits
- Manual sync command provides user control when auto-sync is disabled
- Consider reusing Connection Manager (Epic 43) patterns for transport lifecycle management
