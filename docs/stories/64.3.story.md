## Refined by Spike (ADR-0034)

Transport mechanism determined: Git-based sync via shared bare repository. The `SyncTransport` interface allows future alternatives, but the initial implementation uses Git.

## User Story

As a ThreeDoors user with multiple computers,
I want my task data to transfer reliably between machines,
So that I can work on any computer and see my latest tasks.

## Acceptance Criteria

- AC1: `SyncTransport` interface defined in `internal/sync/` with `Push(ctx context.Context, changeset Changeset) error` and `Pull(ctx context.Context, since time.Time) (Changeset, error)` methods
- AC2: `Changeset` type defined with `DeviceID device.DeviceID`, `Timestamp time.Time`, `Files []SyncFile` where `SyncFile` has `Path string`, `Content []byte`, `Op SyncOp` (Add/Modify/Delete)
- AC3: `GitSyncTransport` implements `SyncTransport` — initializes Git repo in `GetConfigDirPath()/sync/`, manages the stage → commit → pull --rebase → push cycle
- AC4: Git binary wrapper (`GitExecutor`) provides reusable command execution with timeout (30s default), stderr capture, and structured error types
- AC5: Git remote configured via `threedoors sync init <remote-url>` — validates remote is reachable (via `git ls-remote`), clones or initializes, persists remote URL to config
- AC6: Git availability check — `threedoors doctor` reports `git` binary presence and version; `sync init` fails with clear error if `git` is not in PATH
- AC7: Delta sync via Git — only changed files are transferred (Git's native behavior)
- AC8: Encryption in transit delegated to Git transport (SSH or HTTPS) — ThreeDoors validates the remote URL scheme (`ssh://`, `git@`, `https://`)
- AC9: Sync trigger: debounced 30s after last task change (file watcher via `fsnotify` on all synced files: `tasks.yaml`, `sessions.jsonl`, theme portion of `config.yaml`) + manual via `threedoors sync push` command. Debounce timer uses injectable clock for testability.
- AC10: CLI subcommands: `threedoors sync init <remote-url>` (setup), `threedoors sync push` (manual trigger), `threedoors sync status` (display: idle/syncing/error, last sync time, unpushed commits count, remote URL). Supports `--json` flag for machine-readable output.
- AC11: Sync progress surfaced in TUI status bar as a new "git-sync" provider entry — registers with `SyncStatusTracker`, updates phase during operations (syncing/synced/pending/error)
- AC12: `GitSyncConnection` registered as a connection in Connection Manager — reuses state machine, health checking, and circuit breaker (ADR-0034). Registered during `sync init`, auto-registered on app startup from persisted config.
- AC13: Network failures queue as unpushed Git commits — push retries on next sync cycle with exponential backoff via circuit breaker
- AC14: `.gitattributes` in sync repo configures merge strategies: `tasks.yaml merge=threedoors-task-merge`, `sessions.jsonl merge=union`
- AC15: Error handling: remote unreachable → circuit breaker tracks failure; auth failure → connection transitions to Error with descriptive message; rebase conflict → logged to `conflicts.jsonl`, sync continues with local version winning
- AC16: Integration tests with two `t.TempDir()` directories simulating two devices syncing via a bare Git repo — includes at least one conflict scenario (same file modified on both "devices")
- AC17: `git gc` explicitly deferred to future story (64.x) — not implemented here

## Technical Notes

- **Package:** `internal/sync/` — new package for all sync transport code. Do NOT add to `internal/core/`.
- Git operations via `os/exec` calling `git` binary (avoid CGo dependency on libgit2)
- Sync repo path: `GetConfigDirPath()/sync/` (respects `$XDG_CONFIG_HOME/ThreeDoors`) — NOT hardcoded `~/.threedoors/sync/`
- Sync config (remote URL, enabled flag) persisted in `config.yaml` under a `sync:` key
- Debounce uses `time.AfterFunc` reset on each file change — only triggers sync after 30s of quiet
- Connection Manager integration: `GitSyncConnection` state transitions: Disconnected → Connecting (clone/fetch) → Connected → Syncing (commit+push) → Connected
- Circuit breaker trips after 5 consecutive push/pull failures; probe interval 30s–30m
- File watcher dependency: `github.com/fsnotify/fsnotify` — watches all synced files, not just `tasks.yaml`
- All tests use real Git in `t.TempDir()` — no mocking of `os/exec`. Git is available in CI.
- Git commit messages include device ID: `sync: <device-name> (<device-id-short>)`

## Error Handling

- **Git not installed:** `sync init` returns `ErrGitNotFound` with install instructions. `threedoors doctor` checks for Git.
- **Remote unreachable:** `git ls-remote` fails during init → clear error. During push/pull → circuit breaker tracks.
- **Auth failure:** SSH key missing or HTTPS 401 → connection transitions to Error state with descriptive message.
- **Rebase conflict:** `git pull --rebase` fails → log conflict details to `GetConfigDirPath()/sync/conflicts.jsonl`, abort rebase, keep local version. Flag in sync status.
- **Concurrent sync:** Mutex prevents overlapping sync operations on same repo.

## Test Design Notes

### State Transition Sequences (for assertions)
- **sync init (success):** Disconnected → Connecting → Connected
- **sync init (remote unreachable):** Disconnected → Connecting → Error
- **sync push (success):** Connected → Syncing → Connected (LastSync updated)
- **sync push (network failure):** Connected → Syncing → Error (circuit breaker records failure)
- **sync push (circuit open):** Connected → remains Connected, returns `ErrCircuitOpen`
- **auto-sync (debounce):** file change → timer reset → 30s quiet → Connected → Syncing → Connected

### Injectable Clock Pattern
Follow `CircuitBreaker.now` pattern (`circuit_breaker.go:67`):
- `GitSyncTransport.now func() time.Time` — for timestamps in changesets/commits
- `Debouncer.timerFunc func(d time.Duration, f func()) Timer` — replaces `time.AfterFunc` so tests can fire instantly
- `Timer` interface with `Stop() bool` — matches `*time.Timer`

### Conflict Test Expected Behavior
1. Device A writes `tasks.yaml` with task title "A-title", commits+pushes
2. Device B writes `tasks.yaml` with task title "B-title", commits
3. Device B pulls with rebase → if rebase auto-resolves, B's version wins (it's the rebased-on-top commit)
4. If rebase fails (conflict markers), abort rebase, keep B's local version, log conflict to `conflicts.jsonl`
5. Device B pushes successfully (its version is now on remote)
6. Device A pulls → gets B's version

### Test Helpers (`internal/sync/testutil_test.go`)
- `createBareRepo(t *testing.T) string` — returns path to bare Git repo
- `cloneRepo(t *testing.T, bareDir string) string` — clones bare repo, returns work dir
- `commitFile(t *testing.T, repoDir, filename, content string)` — writes file, adds, commits
- `pushRepo(t *testing.T, repoDir string)` — pushes to origin
- All helpers use `t.Helper()` and `t.Cleanup()` per project standards

### File Watcher Edge Cases to Test
- Rapid-fire changes (10 writes in 100ms) → single debounced sync
- Watched file deleted then recreated → watcher recovers, triggers sync
- Watcher error recovery → logs error, attempts to re-watch
- Multiple synced files changed → single coalesced sync trigger

### CLI Test Pattern
Check existing CLI tests in `internal/cli/` — follow same approach (likely `cmd.SetArgs()` + `cmd.Execute()` with captured stdout/stderr)

### `GitExecutor` Interface
```go
type GitExecutor interface {
    Run(ctx context.Context, repoDir string, args ...string) (stdout string, err error)
}
```
Real implementation wraps `os/exec`. For testing "git not found" error paths, use a stub that returns `exec.ErrNotFound`. All happy-path tests use real Git.
