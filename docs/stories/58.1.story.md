## Story

As a multiclaude daemon,
I want to monitor the supervisor's JSONL transcript for context window utilization signals,
So that I can detect when the supervisor is approaching degradation and trigger a handover.

## Background

Claude Code doesn't expose a "tokens used" counter to the running agent. However, context window pressure is observable through external signals: JSONL transcript file size, context compression event count, and assistant message count. The daemon already runs a 5-minute refresh loop — adding transcript monitoring to this loop is the minimum viable approach.

**Party mode consensus (Session 1):** External daemon monitoring, not supervisor self-reporting. A degrading supervisor is the worst entity to trust for self-assessment.

**Threshold model (three-tier):**
- **Green (0-60%):** Below all thresholds. Normal operation.
- **Yellow (60-80%):** Compression count >= 3 OR JSONL > 5MB. Begin advisory logging, update rolling snapshot more frequently.
- **Red (80%+):** Compression count >= 6 OR JSONL > 10MB. Trigger handover at next natural seam.

**Clock type:** Hybrid — time floor (no handover before 30 min) + usage ceiling (force handover at N compression events regardless of time).

## Acceptance Criteria

**AC-1: Transcript discovery**
**Given** a supervisor agent is running
**When** the daemon refresh loop executes
**Then** the shift clock locates the supervisor's active JSONL transcript file at `~/.claude/projects/<project-hash>/<session-id>.jsonl`

**AC-2: Metrics collection**
**Given** a valid JSONL transcript file
**When** the shift clock runs
**Then** it collects three metrics: file size (bytes), compression event count (grep for compression markers), and assistant message count

**AC-3: Three-tier threshold evaluation**
**Given** collected metrics
**When** evaluated against thresholds
**Then** the shift clock classifies the supervisor session as Green, Yellow, or Red zone
**And** logs the zone classification and metric values

**AC-4: Time floor enforcement**
**Given** the supervisor session entered Red zone
**When** the session has been running for less than 30 minutes
**Then** the shift clock does NOT trigger handover
**And** logs an advisory that handover is pending time floor

**AC-5: Anti-oscillation guard**
**Given** a handover completed less than 30 minutes ago
**When** the shift clock evaluates thresholds
**Then** no new handover is triggered regardless of metrics

**AC-6: Natural seam detection**
**Given** the shift clock determines handover should trigger
**When** checking for a natural seam
**Then** it verifies: no `multiclaude work` commands in the last 60 seconds, no pending message acknowledgments, no active tool calls
**And** defers handover until a natural seam is found

**AC-7: Signal file output**
**Given** handover conditions are met (Red zone + time floor + natural seam)
**When** the shift clock decides to trigger
**Then** it writes a signal file at `~/.multiclaude/handover/<repo>/handover-requested` containing timestamp and metrics

## Technical Notes

- Implementation: Shell script integrated into daemon's existing refresh loop
- No Go code changes needed for v1
- Transcript path discovery: `~/.claude/projects/` directory, find active session by modification time
- Compression markers in JSONL: grep for `"type":"context_compression"` or similar Claude Code compression indicators
- The signal file is the handover trigger — Story 58.3 (Handover Orchestrator) reads it
