# Story 0.31: CI/Security Hardening — Secrets, Supply Chain & Reproducibility

## Story

As the ThreeDoors maintainer,
I want the CI pipeline hardened against secret exposure, supply chain attacks, and non-reproducible builds,
So that signing credentials are scoped to release jobs, third-party action risk is eliminated, and lint results are deterministic.

**Status:** In Review

---

## Background

A security audit (2026-03-08) identified four hardening opportunities in `.github/workflows/ci.yml`, filed as GitHub issues #244, #245, #246, and #248. Party mode triage (all agents) confirmed all four are valid, low-effort, high-value improvements that belong in a single story.

**Party Mode Artifact:** `_bmad-output/planning-artifacts/ci-security-hardening-triage-party-mode.md`
**GitHub Issues:** #244, #245, #246, #248

---

## Acceptance Criteria

### AC1: Pin golangci-lint version (Issue #248)

- [ ] `golangci-lint-action` step specifies an explicit `version:` parameter
- [ ] Version matches the currently installed local version or latest stable release
- [ ] CI lint job produces deterministic results across runs

### AC2: Pass secrets via env vars (Issue #246)

- [ ] All `run:` steps that use secrets reference them via `$ENV_VAR`, not `${{ secrets.X }}` interpolation
- [ ] Secrets are passed through the step's `env:` block
- [ ] No `${{ secrets.* }}` expressions remain in any `run:` block
- [ ] Steps that already use env vars correctly are left unchanged

### AC3: Replace softprops/action-gh-release with gh CLI (Issue #245)

- [ ] `softprops/action-gh-release` is removed from the workflow
- [ ] Replaced with a `run:` step using `gh release create`
- [ ] All release assets (binaries, .pkg, .dmg) are uploaded correctly
- [ ] Pre-release flag is preserved
- [ ] Release title format is preserved
- [ ] `GH_TOKEN` is passed via env var (consistent with AC2)

### AC4: Reference protected GitHub environment (Issue #244)

- [ ] `sign-and-notarize`, `release`, and `update-homebrew` jobs include `environment: release`
- [ ] Workflow changes are compatible with a "release" GitHub environment (to be created manually by repo owner)
- [ ] **Manual prerequisite documented:** Repo owner must create the "release" environment in GitHub Settings, configure branch protection (main only), and move the 10 signing/deployment secrets into it

## Technical Notes

### Implementation Order

1. #248 (pin lint version) — trivial, no risk
2. #246 (env vars) — mechanical refactor of existing pattern
3. #245 (replace 3rd party action) — must match exact release behavior
4. #244 (protected environment) — YAML change + manual Settings prerequisite

### File Changed

- `.github/workflows/ci.yml`

### Verification Plan

- CI must pass on the PR (lint, test, build jobs)
- Release-related changes (#245, #244) should ideally be verified with a test release on a non-main branch or manual inspection of the workflow syntax
- No application code changes; no unit tests needed

### Risks

- **#245:** The `gh release create` command must replicate exact `softprops/action-gh-release` behavior. Verify asset upload syntax and release notes format.
- **#244:** If the "release" environment isn't created in GitHub Settings before merging, release jobs will fail. Document this clearly.
