# Story 25.3: Bidirectional Sync & WAL Integration

**Status:** In Review (PR #336)
**Epic:** 25 — Todoist Integration
**Depends On:** Story 25.2

## User Story

As a ThreeDoors user who uses Todoist,
I want completing a task in ThreeDoors to mark it complete in Todoist,
So that my task status stays synchronized across both tools without manual updates.

## Acceptance Criteria

**Given** a configured Todoist provider with bidirectional sync
**When** a task is marked complete in ThreeDoors
**Then:**

- AC1: `MarkComplete(taskID)` calls `Client.CloseTask(ctx, taskID)` to complete the task via Todoist API
- AC2: On successful API call, task status updated locally to `complete`
- AC3: On API failure (network error, rate limit), change queued via `WALProvider` for later replay
- AC4: `WALProvider` wrapping: `core.NewWALProvider(todoistProvider)` integrates with existing offline queue
- AC5: Queued changes replayed automatically when connectivity is restored (sync scheduler integration)
- AC6: Rate limit errors (429) trigger exponential backoff with jitter before retry
- AC7: Circuit breaker integration: after 5 consecutive failures within 2 minutes, circuit trips to Open
- AC8: When circuit is Open, `MarkComplete` queues to WAL without attempting API call
- AC9: Unit tests for successful completion, API failure with WAL fallback, and rate limit handling

## Technical Notes

- Todoist task completion endpoint: `POST /rest/v1/tasks/{id}/close` — no request body needed
- Unlike Jira (which requires discovering valid transitions), Todoist completion is a simple POST
- WALProvider pattern is already established from Epic 21 — reuse existing infrastructure
- The sync scheduler from Epic 21 handles per-provider loops; Todoist gets its own loop with configurable `poll_interval` (default 60s)
- Circuit breaker thresholds: 5 failures / 2-minute window, probe at 30s expanding to 30m max
- Only `MarkComplete` is bidirectional for Phase 1; full CRUD (create, update, delete) deferred
