# Story 81.3: Workspace Refresh Discoverability

**Epic:** 81 — Git Coordination Fixes
**Status:** Not Started
**Priority:** P1
**Estimate:** S (< 1 hour)

## User Story

As a multiclaude operator working in the workspace window,
I want the workspace to be easily refreshable and the /refresh command to be well-documented,
So that I don't work with stale code when reviewing agent output or making manual changes.

## Context

The workspace worktree (where the human operator works) is created by multiclaude but never auto-refreshed by the daemon. The `/refresh` slash command exists and works, but:
- It's not mentioned in CLAUDE.md's operator workflow section
- New operators may not know it exists
- The workspace can drift significantly behind main during active multi-agent sessions

## Acceptance Criteria

- **AC1:** CLAUDE.md "Operator Workflow (multiclaude)" section mentions `/refresh` as the way to sync the workspace worktree
- **AC2:** A note is added explaining that the workspace worktree is NOT auto-refreshed by the daemon (unlike worker worktrees)
- **AC3:** The `/refresh` slash command documentation in `.claude/commands/` is verified to work correctly for the workspace context

## Technical Notes

- The `/refresh` command already exists and performs `git fetch origin main && git rebase origin/main`
- Workspace worktrees are at `/Users/skippy/.multiclaude/wts/<repo>/<workspace-name>/`
- The daemon refresh loop only targets worker worktrees, not workspace worktrees

## Out of Scope

- Automatic daemon refresh of workspace worktrees (that's Story 81.4)
- Changes to the /refresh command itself (it already works)
