# Story 38.4: Alpha Release Verification

**Epic:** 38 — Dual Homebrew Distribution
**Status:** Done (PR #295)
**Priority:** P2
**Estimate:** M (3-5 hours)

---

## User Story

As a repository maintainer, I want automated verification that the alpha Homebrew formula works after each push to main so that broken formulas are detected and flagged immediately.

## Context

`release-verify.yml` monitors the stable formula after GoReleaser pushes it, waiting for the tap repo's CI to pass. No equivalent exists for the alpha formula. This story adds alpha formula verification that mirrors the stable pattern with lighter-weight checks appropriate for the higher frequency of alpha releases.

Party mode artifact: [Homebrew Dual Publish Course Correction](../../_bmad-output/planning-artifacts/homebrew-dual-publish-course-correction.md) (Decision CC-5)

## Acceptance Criteria

### AC-1: Alpha Verification Workflow
- [x] A verification step or workflow monitors the alpha formula after CI pushes it to the tap
- [x] Verification is triggered only when the alpha formula was actually pushed (respects `ALPHA_TAP_ENABLED` toggle)
- [x] Verification looks for the alpha formula commit message pattern: `chore(formula): update threedoors-a`

### AC-2: Tap CI Monitoring
- [x] After detecting the alpha formula commit in `arcaven/homebrew-tap`, the verification waits for the tap repo's CI status
- [x] Uses the `HOMEBREW_TAP_TOKEN` for API access to the tap repo
- [x] Timeout is shorter than stable verification (alpha failures are less critical) — suggest 15 minutes max
- [x] Reports success or failure

### AC-3: Issue Creation on Failure
- [x] If tap CI fails for the alpha formula, a GitHub issue is auto-created in the ThreeDoors repo
- [x] Issue title distinguishes alpha from stable failures: "Alpha release verification failed: tap CI did not pass"
- [x] Issue body includes the tap commit SHA, CI link, and remediation steps
- [x] Issue is labeled `bug,release`

### AC-4: Graceful Handling of Missing Formula Push
- [x] If `ALPHA_TAP_ENABLED` is off and no formula was pushed, verification is skipped cleanly (not an error)
- [x] If the formula commit is not found within a reasonable timeout (shorter than stable — 3 minutes), verification reports this but does not block CI

## Technical Notes

- The verification can be implemented as either:
  - A new `workflow_run` trigger on `release-verify.yml` (extending the existing workflow)
  - Additional steps in the CI workflow's `release` job (simpler but couples verification to the release job)
  - Recommendation: extend `release-verify.yml` with a second job for alpha, triggered by CI workflow completion
- The tap repo (`arcaven/homebrew-tap`) runs its own CI on every push, which validates `brew audit`, `brew install`, and `brew test` — this is the primary quality gate
- `HOMEBREW_TAP_TOKEN` is already available as a secret
- Alpha formula pushes happen much more frequently than stable (every main push vs every tag), so timeouts should be shorter to avoid CI queue buildup

## Dependencies

- Story 38.1 (PR #273) — Alpha formula must exist in tap
- Story 38.2 — Toggle must exist so verification respects it
- `arcaven/homebrew-tap` CI — Must be configured to run on alpha formula commits

## Out of Scope

- Running `brew install` directly in this workflow (tap CI handles that)
- Weekly cron-based deep verification
- Modifying the tap repo's CI configuration
