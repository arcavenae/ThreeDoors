# Story 65.1: Core CLI Path Tests ‚Äî Bootstrap, Root, Doors, TaskPool Loading

## Status: Not Started

**Epic:** 65 ‚Äî CLI Test Coverage Hardening
**Priority:** P0
**Depends On:** None

## User Story

As a developer,
I want the critical CLI entry paths to have comprehensive test coverage,
So that regressions in application startup, command routing, and the primary doors command are caught before release.

## Context

The state-of-testing audit (TEA report, 2026-03-09) identified `internal/cli` at 34.8% coverage ‚Äî the only critical gap in the project. The core CLI paths (`bootstrap`, `root`, `doors`, `loadTaskPool`, `output`) are the primary user entry points. Untested paths here can break silently and affect every user.

## Acceptance Criteria

- AC1: `bootstrap()` tested with mock dependencies ‚Äî verify initialization sequence, config loading, provider factory wiring
- AC2: `Execute()` tested ‚Äî verify root command setup, subcommand registration, and error propagation
- AC3: `KnownSubcommands()` tested ‚Äî verify returns complete list of registered subcommands
- AC4: `NewDoorsCmd()` coverage increased from 16.7% to ‚â•70% ‚Äî test flag parsing, `loadTaskPool` integration, error paths
- AC5: `loadTaskPool()` tested with mock providers ‚Äî success path, missing config, invalid provider, fallback behavior
- AC6: `isJSONOutput()` and `isTerminal()` tested with table-driven tests covering all flag/env combinations
- AC7: Output formatting helpers in `output.go` have table-driven tests for all output modes (text, JSON, quiet)
- AC8: All tests pass with `go test -race ./internal/cli/...`
- AC9: No new dependencies added ‚Äî use stdlib `testing` and existing `testkit` factories

## Technical Notes

- Use `cobra.Command.ExecuteContext()` with test contexts for command execution tests
- Mock `TaskProvider` via interface ‚Äî don't need real file/adapter backends
- `bootstrap()` may need refactoring to accept injected dependencies for testability ‚Äî if so, keep refactoring minimal and focused on enabling tests
- Test `loadTaskPool` error paths: missing YAML, invalid adapter config, empty task list
- For `isTerminal()`, test with explicit flag override (don't depend on actual terminal state)
