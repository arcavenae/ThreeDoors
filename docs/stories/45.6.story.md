## Story

As a user running `threedoors connect <provider>` without flags in a terminal,
I want the interactive setup wizard to launch automatically,
So that I can configure a data source connection without memorizing flag syntax.

## Background

The connect wizard (`internal/tui/connect_wizard.go`, Story 44.1) is fully implemented and battle-tested with 35+ tests. It supports all 8 providers, OAuth device code flow, environment token detection, and provider auto-detection. However, the CLI `connect` command (`internal/cli/connect.go`, Story 45.1) returns a hard-coded stub error on line 101 when invoked without `--label`:

```go
return fmt.Errorf("interactive wizard not yet available (Story 44.1), use flags: --label, --token, etc")
```

The wizard already implements `tea.Model` (Init/Update/View). A thin Bubbletea wrapper that creates a `tea.Program` with just the `ConnectWizard` model (not the full ThreeDoors TUI) is all that's needed.

## Acceptance Criteria

**AC1: Interactive wizard launches from CLI**
**Given** `threedoors connect todoist` with no `--label` flag
**When** stdin is a terminal (TTY)
**Then** the interactive connect wizard launches in the terminal
**And** the wizard starts at Step 1 (provider selection) with "todoist" pre-selected

**AC2: Wizard completion persists the connection**
**Given** the wizard launched from CLI completes successfully (emits `ConnectWizardCompleteMsg`)
**When** the user finishes all wizard steps
**Then** the connection is created via `ConnectionService.Add()`
**And** credentials are stored
**And** a connection health test runs
**And** a success summary is printed to stdout
**And** the command exits 0

**AC3: Wizard cancellation exits cleanly**
**Given** the wizard launched from CLI
**When** the user presses Esc at any step (emits `ConnectWizardCancelMsg`)
**Then** the command exits 0 with no error
**And** no connection is created
**And** any in-progress OAuth flow is cancelled

**AC4: Non-TTY falls back to error with guidance**
**Given** `threedoors connect todoist` piped from a script (stdin is not a terminal)
**When** the command runs without `--label`
**Then** an error is returned explaining that `--label` and other flags are required in non-interactive mode
**And** exit code is non-zero

**AC5: JSON mode rejects interactive**
**Given** `threedoors connect todoist --json` without `--label`
**When** the command runs
**Then** a JSON error is returned (existing behavior preserved)
**And** the wizard does NOT launch

**AC6: All 8 providers accessible via CLI wizard**
**Given** `threedoors connect <provider>` for any of: todoist, github, jira, textfile, applenotes, obsidian, reminders, linear
**When** the wizard launches
**Then** the provider is available in the wizard's provider list
**And** providers already in `ValidArgs` but not in `knownProviderSpecs` (linear, obsidian, reminders, applenotes) work via the wizard path

**AC7: Provider pre-selection from CLI argument**
**Given** `threedoors connect github` (provider passed as arg)
**When** the wizard launches
**Then** the provider selection step pre-selects "github"
**And** the user can still change the selection if desired

## Technical Notes

### Approach: Standalone `tea.Program` wrapping `ConnectWizard` (Option A from research)

- Create a minimal Bubbletea program that wraps `ConnectWizard` as the root model
- The wrapper model handles `ConnectWizardCompleteMsg` (persist connection, print summary, quit) and `ConnectWizardCancelMsg` (quit cleanly)
- Use `tea.NewProgram(model, tea.WithAltScreen())` for the standalone program â€” same pattern as `cmd/threedoors/main.go:230`
- Terminal detection uses existing `isTerminal(os.Stdin.Fd())` from `internal/cli/interactive.go`
- `ConnectWizard` already implements `tea.Model` with Init/Update/View
- Reuse `bootstrapForConnect()` for `ConnectionService` and `ConnectionManager` setup
- Reuse `DefaultProviderSpecs()` from `internal/tui/connect_wizard.go` for full 8-provider coverage

### Pre-selection Design

The `ConnectWizard` currently starts at Step 1 (provider selection) with no pre-selection. Two approaches for pre-selecting the CLI-provided provider:

1. **Add `SetProvider(name string)` method** to `ConnectWizard` that skips Step 1 and jumps directly to Step 2 (config) â€” cleanest UX
2. **Pre-populate the form value** in Step 1 so the provider is highlighted but user can change â€” preserves full wizard flow

Option 1 is preferred for CLI UX (user already chose the provider in the command). Option 2 is acceptable if modifying the wizard is out of scope.
