## Story

As the supervisor agent,
I want a persistent research-supervisor agent that invokes the Gemini CLI via OAuth for web-grounded research, with a clear definition file, polling loop, request protocol, and authority matrix,
So that any agent can request research via messaging and receive structured findings without blocking.

## Background

This is a rearchitecture of the original 54.1. The research-supervisor now wraps the official Gemini CLI (`@google/gemini-cli`) using OAuth authentication instead of the Python-based `agent-deep-research` tool with a paid API key.

Key changes from the original design:
- **Execution layer:** `gemini -p "<query>" --output-format json` replaces `uv run agent-deep-research`
- **Authentication:** Google Account OAuth (free, cached tokens) replaces `GEMINI_DEEP_RESEARCH_API_KEY` (paid)
- **Polling loop:** Simplified — standard queries are synchronous (seconds to minutes), no async polling needed
- **Model selection:** Research-supervisor selects Pro (deep analysis, 50/day) vs Flash (quick lookups, 1,000/day)

The agent definition must specify: the 5-minute message-check loop, the structured request format, authority boundaries (information provider only), communication protocols, priority queue, and model selection heuristics.

## Acceptance Criteria

**Given** the agent definition file at `agents/research-supervisor.md`
**When** a developer reads the file
**Then** it follows the Responsibility+WHY format established in Story 51.2

**Given** the agent definition
**When** a persistent agent is spawned from it
**Then** it describes a 5-minute message-check loop that: checks for new requests, dispatches queued queries via `gemini -p`, and processes completed results

**Given** the request protocol section
**When** an agent wants to request research
**Then** the format `RESEARCH priority=<high|normal|low> depth=<quick|standard|deep> [context=<bundle-names>]: <question>` is clearly documented with examples

**Given** a `depth=quick` request
**When** the research-supervisor selects a model
**Then** it uses `gemini-2.5-flash` (1,000 RPD free tier)

**Given** a `depth=standard` or `depth=deep` request
**When** the research-supervisor selects a model
**Then** it uses `gemini-2.5-pro` (50 RPD free tier)

**Given** the authority matrix
**When** the research-supervisor processes results
**Then** it is explicitly restricted to: receive, queue, dispatch, store, summarize, and deliver — never decide, execute, create stories, or modify code

**Given** the communication protocols
**When** the research-supervisor completes a query, encounters an error, or has a budget alert
**Then** structured message formats are documented for each: `RESEARCH-RESULT`, `RESEARCH-ERROR`, `RESEARCH-BUDGET`

**Given** the ESCALATE section
**When** budget is exhausted, OAuth fails, or queries fail repeatedly
**Then** the definition specifies escalation to the supervisor

## Technical Notes

- This is a documentation/agent-definition change only. No Go code.
- Follow the Responsibility+WHY format from Story 51.2 (see `agents/merge-queue.md` for reference).
- The execution model is synchronous: `gemini -p` blocks until completion, then returns JSON. No polling loop for individual queries.
- The 5-minute loop is for checking multiclaude messages, not for polling Gemini API status.
- OAuth tokens are managed by the Gemini CLI itself — the agent definition does NOT need to handle token refresh.
- Prerequisite: Gemini CLI installed and authenticated (Story 54.2).
- After creating the definition, the agent must be spawned via `multiclaude agents spawn --name research-supervisor --class persistent --prompt-file agents/research-supervisor.md`.
- Add research-supervisor to the startup checklist in supervisor memory.
