## Story

As a multiclaude operator,
I want an external heartbeat mechanism that periodically wakes all persistent agents and triggers their polling loops,
So that agents don't go idle after completing initial startup work and continue autonomously monitoring the project.

## Background

All 6 persistent agents (merge-queue, pr-shepherd, arch-watchdog, envoy, project-watchdog, retrospector) go idle after completing their initial startup work. Claude agents have no internal timers — they only act when prompted. Even the retrospector, which has the most developed autonomy language (Story 51.11's "Your Rhythm" polling loop, Epic 62's checkpoint/identity/queue improvements), still goes idle because there is no external mechanism to trigger the next loop iteration.

### Root Cause

Claude's system prompt describes desired behavior ("every 15 minutes, poll X") but Claude cannot self-invoke on a schedule. After processing all startup work, the agent sits at an idle prompt with no incoming messages, so it stops acting.

### Research Findings

**multiclaude has no native cron/scheduling.** No `cron` subcommand exists.

**Claude Code has `CronCreate`/`CronList`/`CronDelete` tools** — session-scoped cron jobs that fire prompts into the REPL when idle:
- Session-only — lost when Claude exits
- Auto-expire after 3 days
- Only fire while REPL is idle (not mid-query)
- This is the most practical mechanism for heartbeats

**Agent definition audit — autonomy language:**

| Agent | Has Polling/Rhythm Section? | Has Startup Sequence? |
|-------|---|---|
| retrospector | Yes (detailed) | Yes (detailed) |
| arch-watchdog | Yes (rhythm + polling loop) | Yes (restart/recovery) |
| envoy | Yes (brief rhythm) | Partial |
| project-watchdog | Yes (polling loop) | Yes (restart/recovery) |
| merge-queue | **NO** | **NO** |
| pr-shepherd | **NO** | **NO** |

**Definition language issues:**
- merge-queue and pr-shepherd have ZERO autonomy language — they are written as purely reactive agents that wait for events
- Even agents with polling loops describe behavior but have no mechanism to execute it
- No agent has a "HEARTBEAT Response Protocol" for recognizing and acting on heartbeat messages

### Proposed Solution: Two Layers

**Layer 1 — Supervisor CronCreate heartbeats:**
The supervisor sets up `CronCreate` cron jobs during its startup checklist. Each cron fires a prompt that sends `multiclaude message send <agent> HEARTBEAT` to the target agent.

Proposed intervals:
- merge-queue, pr-shepherd: every 7 minutes (`*/7 * * * *`)
- envoy: every 11 minutes (`*/11 * * * *`)
- retrospector: every 13 minutes (`*/13 * * * *`)
- project-watchdog, arch-watchdog: every 23 minutes (`*/23 * * * *`)

Prime-number intervals prevent all agents from being poked simultaneously and avoid the :00/:30 minute marks.

**Layer 2 — Agent definition HEARTBEAT handler:**
Every persistent agent definition gets a "HEARTBEAT Response Protocol" section:
1. On receiving a message containing "HEARTBEAT": run full polling loop
2. Ack the HEARTBEAT message
3. Report any findings via normal channels

Agents currently missing polling loops (merge-queue, pr-shepherd) need polling loop sections added.

## Technical Notes

- CronCreate is session-scoped (3-day auto-expiry, lost on restart). The supervisor must re-create crons after every restart.
- System cron (`crontab`) was considered but rejected: it requires shell-level setup outside the repo, doesn't integrate with multiclaude messaging, and adds operational complexity. CronCreate is simpler and sufficient given the supervisor's restart cadence.
- Prime-number intervals (7, 11, 13, 23 minutes) prevent heartbeat storms where all agents poll simultaneously.
- HEARTBEAT messages are lightweight — the agent determines what work to do, not the heartbeat sender.
- The heartbeat doesn't replace message-driven work (agents should still respond to direct messages). It supplements by ensuring agents poll even when no messages arrive.

## Risks and Mitigations

| Risk | Mitigation |
|------|-----------|
| CronCreate lost on supervisor restart | Startup checklist includes cron setup; 3-day expiry is acceptable given daily restarts |
| HEARTBEAT message flood if interval too short | Prime-number intervals spread load; agents ack and return to idle quickly |
| Agent context fills faster with more polling | Heartbeat just triggers existing polling — no new context-heavy operations added |
| multiclaude messaging still unreliable | Retrospector already has file-based fallback (Story 62.1); other agents may need similar if messaging fails consistently |

## Out of Scope (Noted for Future)

- System-level cron/launchd for heartbeats (more durable but more complex)
- Agent self-restart on context exhaustion (requires multiclaude daemon support)
- File-based inbox fallback for all agents (currently only retrospector has this)
- multiclaude daemon-level agent poking (would require multiclaude code changes)
