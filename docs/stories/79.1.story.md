# Story 79.1: Core Daemon Infrastructure — `threedoors serve` Command

**Epic:** 79 — ThreeDoors Daemon/Server Mode
**Status:** Not Started
**Priority:** P1
**Estimate:** M (new command, socket listener, graceful shutdown)

## Context

ThreeDoors has no externally addressable API. The TUI runs as a foreground terminal process, and while `threedoors-mcp` provides an MCP server, it uses stdio/SSE transport designed for AI tool integration — not for hardware peripherals like Stream Deck.

A lightweight daemon mode (`threedoors serve`) will expose the TaskProvider and door selection logic over a Unix domain socket, enabling Stream Deck, Apple Shortcuts, widgets, and any future integration to interact with ThreeDoors programmatically.

This story establishes the daemon skeleton: CLI command, socket lifecycle, signal handling, and PID file management. The JSON-RPC protocol layer (79.2) and specific API methods (79.3, 79.4) build on this foundation.

**Research:** [Stream Deck Research](../../_bmad-output/planning-artifacts/streamdeck-research.md) — Section 2 (Integration Architecture), Section 6 (Recommended Tech Stack).

## Acceptance Criteria

- [ ] `threedoors serve` subcommand registered in CLI (Cobra)
- [ ] Daemon listens on Unix domain socket at `~/.threedoors/daemon.sock` by default
- [ ] `--socket` flag allows overriding the socket path
- [ ] `--port` flag enables optional localhost HTTP listener (default: disabled)
- [ ] PID file written to `~/.threedoors/daemon.pid` on startup
- [ ] Stale PID file detected and cleaned up (process not running)
- [ ] Graceful shutdown on SIGINT/SIGTERM: close listener, drain connections, remove socket + PID file
- [ ] Daemon reuses existing config loading and provider initialization from `cmd/threedoors/main.go`
- [ ] Concurrent client connections supported (one goroutine per connection)
- [ ] Startup logs socket path and PID to stderr
- [ ] `threedoors serve --check` returns exit 0 if daemon is running, exit 1 if not
- [ ] Tests: socket lifecycle, PID file management, concurrent connections, graceful shutdown

## Technical Notes

### Socket Location

Default: `~/.threedoors/daemon.sock` (inside the existing config directory). This avoids `/tmp` namespace collisions and respects XDG conventions. The `--socket` flag overrides for testing or custom setups.

### Provider Initialization

Extract the shared provider initialization logic from `cmd/threedoors/main.go` into a reusable function (e.g., `internal/core/bootstrap.go`) so both the TUI and daemon can use it without duplication. The MCP server (`cmd/threedoors-mcp/main.go`) has a similar but diverged copy — consider unifying all three.

### Connection Model

Each accepted connection gets its own goroutine. Use `sync.WaitGroup` for graceful shutdown drain. Set read/write deadlines on connections to prevent leaked goroutines from slow/dead clients.

### Existing Patterns

- `cmd/threedoors-mcp/main.go` — signal handling with `signal.NotifyContext`
- `internal/cli/root.go` — Cobra command registration pattern
- `internal/core/door_selector.go` — `SelectDoors()` already works outside TUI
