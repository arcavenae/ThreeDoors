## Story

As a user who skipped import during onboarding or wants to import additional tasks later,
I want to use `:import` to import tasks from a file,
So that the onboarding wizard's promise of "import later with :import" actually works.

## Background

Story 10.2 (PR #59) implemented the onboarding import step with full file parsing infrastructure (`core.ImportTasksFromFile` supporting plain text and Markdown). The onboarding view (line 450 of `onboarding_view.go`) tells users: "You can also import later with :import". However, the `:import` command was never added to the command system. Typing `:import` returns "Unknown command: 'import'. Type :help for available commands."

The import infrastructure is complete — `core.ImportTasksFromFile` handles path expansion, format detection (text/markdown), checkbox parsing, and returns an `ImportResult` with parsed tasks. The onboarding import step already demonstrates the full flow (path input → parse → preview → confirm). This story wires up that existing infrastructure as a standalone command.

## Acceptance Criteria

**AC-1: Command registered and discoverable**
**Given** the user types `:` in the command palette
**When** they type "imp"
**Then** autocomplete shows `:import` with description "Import tasks from a file"

**AC-2: Import prompts for file path**
**Given** the user executes `:import`
**When** the command is processed
**Then** a view appears with a text input prompting for a file path
**And** the view explains supported formats (plain text, Markdown checkboxes)

**AC-3: Import with inline argument**
**Given** the user executes `:import ~/tasks.txt`
**When** the command is processed
**Then** the file is parsed immediately without prompting for a path
**And** the preview is shown (or error if file not found)

**AC-4: Import preview before confirmation**
**Given** the user provides a valid file path
**When** tasks are successfully parsed
**Then** a preview shows: task count, format detected, first 5 tasks
**And** the user must confirm (Enter/y) or cancel (Esc/n) before tasks are added

**AC-5: Tasks added to pool on confirmation**
**Given** the user confirms the import preview
**When** the import is accepted
**Then** all parsed TODO tasks are added to the task pool
**And** a flash message reports "Imported N tasks from <filename>"

**AC-6: Error handling**
**Given** the user provides an invalid file path or the file contains no tasks
**When** the import is attempted
**Then** an error message is displayed (file not found, no tasks found, etc.)
**And** the user can try a different path or cancel

**AC-7: Escape cancels at any point**
**Given** the user is in the import flow (path input or preview)
**When** they press Esc
**Then** the import is cancelled and they return to the doors view

## Technical Design

### Command Registration

Add to `commandRegistry` in `search_view.go`:
```go
{"import", "Import tasks from a file"},
```

### Command Handling

Add `case "import":` in `executeCommand()` that returns a new message type:

```go
case "import":
    return func() tea.Msg {
        return ShowImportMsg{PrefilledPath: args}
    }
```

### Import View (`internal/tui/import_view.go`)

New standalone import view, reusing the pattern from onboarding but as a dedicated view:

- **State machine:** `pathInput` → `preview` → (confirm → flash) or (cancel → doors)
- **Path input:** `textinput.Model` with placeholder "Path to task file (e.g. ~/tasks.txt)..."
- **Preview:** Reuse same preview layout as `onboarding_view.go:viewImportPreview()`
- **Core call:** `core.ImportTasksFromFile(path)` — already handles path expansion, format detection, parsing

### Message Types (in `messages.go`)

```go
type ShowImportMsg struct {
    PrefilledPath string
}

type ImportConfirmedMsg struct {
    Tasks []*core.Task
    Source string
}
```

### Main Model Integration

In `main_model.go`:
- Handle `ShowImportMsg` → switch to import view
- Handle `ImportConfirmedMsg` → add tasks to pool, flash success message
