## User Story

As the architect,
I want to research and document the sync protocol, transport mechanism, and conflict resolution strategy for cross-computer task synchronization,
So that implementation stories have a clear, validated technical foundation.

- AC2: Transport mechanism evaluated and chosen from: git-based sync, cloud intermediary (S3/GCS), peer-to-peer, or hybrid

- AC4: Device identity mechanism designed: how devices register, identify, and authenticate to each other

## Technical Notes

### Key Questions to Answer

1. **Transport:** Git-based sync (`.threedoors/` already exists as a directory) vs cloud intermediary vs peer-to-peer?
   - Git-based: Natural for the project, uses existing tooling, but requires shared remote
   - Cloud: Simpler conflict handling, but adds external dependency
   - P2P: No server needed, but complex NAT traversal and discovery

2. **Conflict Resolution:** CRDTs vs LWW vs manual?
   - CRDTs: Theoretically correct, but complex to implement and debug
   - LWW with vector clocks: Simpler, proven in distributed systems, may lose data
   - Manual: Safest but worst UX

3. **Identity:** UUID per install? User account? Device fingerprint?
   - Must work without a central server if P2P chosen
   - Must be stable across app updates

4. **Scope:** Full task state or selective?
   - Task list and metadata: Must sync
   - Session logs: Large, append-only — may not need real-time sync
   - Config and themes: Sync once, rare changes
   - Analytics: Aggregate, don't sync raw data

5. **Existing Infrastructure Reuse:**
   - WAL (Epic 21): Currently single-machine — can it be extended for multi-machine?
   - SourceRef (Epic 43): Currently tracks provider origin — can it track device origin?
   - Circuit breaker (Epic 43): Applicable to sync transport failures
