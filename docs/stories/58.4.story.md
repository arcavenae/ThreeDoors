## Story

As an incoming supervisor agent,
I want to read a shift-state.yaml file on startup and assume operational control,
So that I can seamlessly continue managing workers and agents without losing context.

## Background

When the handover orchestrator (Story 58.3) spawns an incoming supervisor, it passes a task referencing the shift-state.yaml file. The incoming supervisor must:

1. Read shift-state.yaml for immediate operational context
2. Read MEMORY.md for persistent cross-session context
3. Read ROADMAP.md for scope and priority context
4. Run `multiclaude message list` for messages that arrived during transition
5. Ping each active worker for status verification
6. Signal "READY" to the daemon

**Party mode consensus (Session 2):** State file for immediate context, worker pings for verification. The combined approach gives immediate orientation plus fresh confirmation.

## Acceptance Criteria

**AC-1: State file detection**
**Given** the supervisor starts with a SHIFT_HANDOVER task
**When** parsing the task
**Then** it extracts the path to shift-state.yaml and reads it

**AC-2: Worker awareness**
**Given** the state file lists active workers
**When** the supervisor reads the file
**Then** it knows each worker's name, task, branch, and last known status

**AC-3: Worker ping verification**
**Given** active workers listed in the state file
**When** the supervisor starts
**Then** it sends a status ping to each active worker via `multiclaude message send <worker> "Status check — new supervisor online"`
**And** processes responses to update its mental model

**AC-4: Priority inheritance**
**Given** the state file contains a priorities section
**When** the supervisor reads it
**Then** it adopts those priorities for its initial decision-making

**AC-5: Pending decision awareness**
**Given** the state file contains pending decisions
**When** the supervisor reads them
**Then** it tracks unresolved decisions and follows up as needed

**AC-6: Message queue processing**
**Given** messages may have arrived during the transition window
**When** the supervisor runs `multiclaude message list`
**Then** it processes all unacknowledged messages before accepting new work

**AC-7: Ready signal**
**Given** the supervisor has completed its startup sequence
**When** state file is read, messages processed, and worker pings sent
**Then** it sends `multiclaude message send daemon "READY"` (or writes a ready signal file)
**And** begins normal supervisor operation

**AC-8: Graceful degradation without delta**
**Given** the state file contains only daemon-collected observable state (no supervisor delta)
**When** the supervisor reads it
**Then** it still operates correctly, with reduced context about pending decisions and priorities
**And** logs that it's running in "emergency handover" mode

## Technical Notes

- This story modifies the supervisor agent definition file (`agents/supervisor.md`) to add shift handover awareness
- The supervisor CLAUDE.md MEMORY.md startup checklist already exists — this adds a conditional branch: "if SHIFT_HANDOVER task, read state file first"
- Worker pings are non-blocking — the supervisor sends pings and continues startup while waiting for responses
- The "READY" signal mechanism should match what Story 58.3 expects to detect

**Supervisor startup sequence (shift handover mode):**
```
1. Detect SHIFT_HANDOVER task in initial prompt
2. Read shift-state.yaml
3. Read MEMORY.md (already part of normal startup)
4. Read ROADMAP.md (already part of normal startup)
5. Run multiclaude message list — process all pending
6. Ping active workers for status
7. Signal READY
8. Resume normal supervisor loop
```
