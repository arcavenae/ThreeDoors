# Story 63.3: ClickUp Bidirectional Sync & WAL Integration

## Status: In Review

**Epic:** 63 — ClickUp Integration
**Priority:** P2
**Story Points:** 5
**Depends On:** Story 63.2
**Blocks:** None
**FRs:** product-scope.md Phase 5 (ClickUp integration)

## User Story

As a ThreeDoors user who uses ClickUp,
I want task status changes in ThreeDoors to sync back to ClickUp (and vice versa),
So that both systems stay consistent without manual updates.

## Acceptance Criteria

- AC1: Completing a task in ThreeDoors calls `UpdateTaskStatus` to set ClickUp task status to the configured "done" status
- AC2: Blocking a task in ThreeDoors updates ClickUp task status to the configured "blocked" status (if mapped)
- AC3: Status changes in ClickUp are detected on next poll and reflected in ThreeDoors
- AC4: All sync operations are logged to the WAL (Write-Ahead Log) before execution
- AC5: WAL replay recovers incomplete sync operations after crash/restart
- AC6: Conflict resolution follows last-writer-wins strategy with field-level granularity (Epic 47 pattern)
- AC7: Circuit breaker trips after 3 consecutive ClickUp API failures, auto-resets after backoff period
- AC8: Sync errors are surfaced in the Sources dashboard (Epic 44) and status bar (Story 44.5)
- AC9: `SourceRef` back-link allows opening ClickUp task in browser from ThreeDoors detail view
- AC10: Sync operations respect ClickUp rate limits (100 req/min) — batch writes if needed

## Technical Notes

- Follow the bidirectional sync pattern from `internal/adapters/todoist/todoist_sync.go`
- WAL integration uses the existing `internal/tasks/wal.go` infrastructure from Epic 21
- Circuit breaker pattern from Epic 43 (`internal/tasks/circuit_breaker.go`)
- Conflict resolution from Epic 47 (`internal/tasks/conflict.go`)
- ClickUp webhook support is available but out of scope — poll-based sync only for MVP
- Status write-back requires mapping ThreeDoors statuses back to ClickUp status names
