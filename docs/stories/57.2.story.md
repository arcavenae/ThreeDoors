## Story

As a user,
I want ThreeDoors to automatically detect which LLM CLI tools are installed and use the best available one,
So that LLM features work with zero configuration on my machine.

## Background

Different users have different LLM CLI tools installed. Rather than requiring manual configuration, ThreeDoors should auto-discover available tools at startup using `exec.LookPath` and select the best available backend via a priority-ordered fallback chain.

**Priority order (S2-D5):** User-configured → claude CLI → gemini CLI → ollama CLI → Ollama HTTP API → Claude HTTP API → LLM features disabled (graceful degradation).

**Party mode decision (S2-D4):** Auto-discovery via `exec.LookPath`. Rejected: manual-only configuration.

## Acceptance Criteria

**AC-1: DiscoverBackend returns best available**
**Given** no explicit `backend` configured in config.yaml
**When** `DiscoverBackend(cfg)` is called
**Then** it checks CLI tools in priority order (claude, gemini, ollama)
**And** returns the first available CLI backend

**AC-2: User-configured backend takes priority**
**Given** `backend: "ollama-cli"` in config.yaml
**When** `DiscoverBackend(cfg)` is called
**Then** it returns the ollama CLI backend regardless of other available tools

**AC-3: HTTP backend fallback**
**Given** no CLI tools are installed
**When** `DiscoverBackend(cfg)` is called
**And** Ollama HTTP endpoint is reachable or ANTHROPIC_API_KEY is set
**Then** it falls back to the appropriate HTTP backend

**AC-4: Graceful degradation**
**Given** no CLI tools and no HTTP backends available
**When** `DiscoverBackend(cfg)` is called
**Then** it returns `ErrBackendUnavailable`
**And** LLM-dependent features are disabled without crashing the app

**AC-5: Discovery result logging**
**Given** auto-discovery runs at startup
**When** backends are checked
**Then** the selected backend and available alternatives are logged at INFO level
**And** unavailable backends are logged at DEBUG level
