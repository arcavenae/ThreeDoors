## Story

As a multiclaude system,
I want automatic periodic quota checks with window reset detection,
So that usage monitoring runs continuously without manual intervention.

## Background

The 5-hour window starts from the first message and resets automatically. Without periodic monitoring, warnings only fire when someone manually runs `/quota-status`. This story adds cron-based automation and window lifecycle tracking. R-004 recommends the hybrid daemon+cron approach (QT-2), but Phase 1 implements only the cron component (simpler, daemon integration is Phase 2).

## Acceptance Criteria

**AC1: Window boundary detection**
**Given** JSONL session files with timestamps
**When** the monitor runs
**Then** identifies the first message timestamp in the current window and tracks the window lifecycle

**AC2: Reset time calculation**
**Given** a detected window start
**When** calculated
**Then** displays: window start time, expected reset time (start + 5h), countdown to reset

**AC3: Cron-compatible invocation**
**Given** the monitoring script/command
**When** scheduled via `CronCreate`
**Then** runs successfully every 5-10 minutes, reads JSONL, evaluates thresholds, sends warnings if needed, records snapshots

**AC4: Warning deduplication**
**Given** multiple cron runs within the same threshold tier
**When** a warning was already sent for this tier in this window
**Then** does NOT send duplicate warnings (uses state file for dedup)

**AC5: Window reset "all clear" notification**
**Given** a window reset is detected (new window started after previous one expired)
**When** the monitor runs
**Then** sends an "all clear" message to supervisor: "Quota window reset. New window started at [time]. Budget: 100%."

**AC6: Cron setup documentation**
**Given** the monitoring script
**When** documented
**Then** includes recommended `CronCreate` invocation with interval (every 5 minutes) and the supervisor startup checklist addition

**AC7: Idempotent execution**
**Given** multiple rapid invocations
**When** run
**Then** produces correct results without duplicate warnings, duplicate snapshots, or state corruption

**AC8: State file management**
**Given** the dedup/window state
**When** stored
**Then** uses a state file at `~/.multiclaude/quota/state.json` with: last warning tier sent, window start timestamp, last check timestamp

## Dev Notes

- State file: `~/.multiclaude/quota/state.json` — create directory if needed
- Dedup logic: track `{window_start, last_tier_warned}`. If same window and same-or-lower tier, skip warning. If higher tier, warn again.
- Window detection heuristic: find the most recent JSONL entry older than 5h. Everything after that is the current window.
- The cron should be added to the supervisor startup checklist (MEMORY.md) alongside existing heartbeat crons
- Recommended cron: `CronCreate("*/5 * * * *", "multiclaude message send supervisor QUOTA_CHECK")`
- Alternative: the cron runs a script directly instead of going through the supervisor
- This is the Phase 1 foundation for Phase 2's daemon-native integration
