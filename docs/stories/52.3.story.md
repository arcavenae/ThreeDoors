## Story

As the envoy agent,
I want detailed specifications for each Layer 1 gate,
So that I can apply deterministic checks consistently without unnecessary AI reasoning.

## Background

Layer 1 gates are the first line of defense — fast, mechanical checks that resolve issues without AI deliberation. The current operations guide has duplicate detection and spam handling documented, but the other gates (already-fixed, previously-decided) lack formal specifications. This story creates comprehensive specifications for all four gates, including detection criteria, thresholds, and response templates.

Party mode consensus (Round 1.4, Round 5) established:
- Spam: envoy MAY close but MUST notify supervisor immediately
- Duplicates: flag, NEVER auto-close (even "obvious" duplicates can be subtly different)
- The envoy should err on the side of detecting too many links (false positives are low-cost)

## Acceptance Criteria

**Given** the spam detection gate specification
**When** the envoy encounters an issue
**Then** the spec defines concrete detection criteria:
- Body length threshold (e.g., < 10 characters)
- Known spam patterns (URL patterns, cryptocurrency keywords)
- Gibberish detection heuristic (e.g., no recognizable English words in title)
**And** the spec defines the response: close issue, post brief comment, notify supervisor

**Given** the duplicate detection gate specification
**When** the envoy encounters an issue
**Then** the spec defines matching criteria:
- Exact title match (case-insensitive)
- Fuzzy title match with keyword overlap threshold
- Symptom keyword extraction and matching against open + recently-resolved issues
- GitHub closed issue search as fallback (`gh issue list --state closed --limit 50`)
**And** the spec defines the response: flag with comment linking original, add to tracker, do NOT close

**Given** the already-fixed detection gate specification
**When** the envoy encounters an issue
**Then** the spec defines cross-reference criteria:
- Search PRs merged in last 30 days for issue number references
- Match component/file mentions in issue body against recently changed files
- Strong link patterns (`Fixes #N`, `Closes #N`) vs weak links
**And** the spec defines the response: comment linking fix PR, suggest verification, recommend closure to supervisor

**Given** the previously-decided detection gate specification
**When** the envoy encounters an issue
**Then** the spec defines lookup criteria:
- Search BOARD.md Decided section for matching keywords/topics
- Search BOARD.md Pending Recommendations for related in-progress work
- Match against SOUL.md misalignment patterns (from `docs/issue-tracker.md` alignment reference)
**And** the spec defines the response: cite the specific decision/recommendation with link, explain status

**Given** all gate specifications
**When** a gate resolves an issue
**Then** there is a documented response template for each outcome (closed, flagged, linked, cited)
