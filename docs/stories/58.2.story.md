## Story

As a multiclaude daemon,
I want to maintain a rolling snapshot of system state from external sources,
So that incoming supervisors have immediate operational context without relying on a potentially degraded outgoing supervisor.

## Background

Instead of relying on the outgoing supervisor to serialize all state at handover time, the daemon maintains a rolling snapshot updated every 5 minutes as part of the existing refresh loop. This snapshot pulls from external sources only — `multiclaude worker list`, `gh pr list`, `multiclaude message list`, `tmux list-windows` — and does NOT require supervisor cooperation for the base snapshot.

**Party mode consensus (Session 3):** Daemon-maintained rolling snapshot. Minimizes cognitive load on a degraded supervisor — it only contributes what it uniquely knows (the "delta" in Story 58.3).

**Size constraint:** Maximum 10KB. Alert if exceeded.

## Acceptance Criteria

**AC-1: Snapshot generation**
**Given** the daemon refresh loop executes
**When** snapshot generation runs
**Then** it produces a YAML file at `~/.multiclaude/handover/<repo>/shift-state.yaml`

**AC-2: Worker state collection**
**Given** active workers exist
**When** snapshot runs
**Then** the `workers.active` section contains each worker's name, task description, branch, PR number (if any), and dispatch timestamp
**And** the `workers.recently_completed` section contains workers that completed in the last 30 minutes

**AC-3: Persistent agent state collection**
**Given** persistent agents are running
**When** snapshot runs
**Then** the `persistent_agents` section lists each agent's name and status (from tmux session list)

**AC-4: Open PR collection**
**Given** open PRs exist on the repository
**When** snapshot runs
**Then** the `open_prs` section lists each PR's number, title, and CI status

**AC-5: Schema version**
**Given** a generated snapshot file
**When** parsed
**Then** it contains `version: 1` and `timestamp` in UTC ISO-8601 format

**AC-6: Size constraint**
**Given** a generated snapshot
**When** the file exceeds 10KB
**Then** the daemon logs a warning with the actual size

**AC-7: Atomic writes**
**Given** snapshot generation
**When** writing to disk
**Then** the write is atomic (write to .tmp, rename) to prevent corruption if read during write

**AC-8: Supervisor delta section preserved**
**Given** an existing shift-state.yaml with supervisor delta content
**When** the daemon updates the observable state sections
**Then** the supervisor delta sections (pending_decisions, priorities, blockers, warnings, issue_triage) are NOT overwritten

## Technical Notes

- Shell script integrated into daemon's existing refresh loop
- Data sources: `multiclaude worker list`, `multiclaude message list`, `gh pr list --json number,title,statusCheckRollup`, `tmux list-windows -t mc-<repo>`
- YAML output via heredoc or simple echo — no YAML library needed for v1
- The supervisor delta (Section 2 of the schema) is written by the outgoing supervisor during handover (Story 58.3), not by this script
- Directory: `~/.multiclaude/handover/<repo>/` — create if not exists

**Schema (observable state only — Section 1):**
```yaml
version: 1
timestamp: "2026-03-11T14:30:00Z"
workers:
  active:
    - name: "bold-eagle"
      task: "Implement story 42.3"
      branch: "work/bold-eagle"
      pr: null
      dispatched_at: "2026-03-11T13:45:00Z"
  recently_completed: []
persistent_agents:
  - name: "merge-queue"
    status: "active"
open_prs:
  - number: 565
    title: "feat: task pool analytics"
    ci: "passing"
```

- Generate versioned YAML with all collected state
- Preserve existing supervisor delta sections if present

- Check file size after write, warn if > 10KB
