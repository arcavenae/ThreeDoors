# Story 0.29: Envoy Operations Guide & Integration Documentation

## Story

As the ThreeDoors development team,
I want an envoy operations guide documenting patrol workflows, cross-agent protocols, staleness thresholds, and communication patterns,
So that the envoy agent can be spawned and operate consistently according to the team's consensus rules.

**Status:** Done (PR #259)

---

## Background

Story 0.28 creates the envoy's primary state file (`docs/issue-tracker.md`). This story creates the operational documentation the envoy needs to execute its patrol cycles, communicate with other agents, and handle edge cases like duplicate detection, direction-conflicting requests, and staleness escalation.

**Party Mode Artifact:** `_bmad-output/planning-artifacts/envoy-rules-of-behavior-party-mode.md`
**Agent Definition:** `agents/envoy.md`

**Key Decisions from Party Mode:**
- Poll-based awareness (GitHub API), not message-dependent
- Proactive notifications via `multiclaude message send` for time-sensitive items only
- Tracker file as awareness backbone — crash-safe, stateless between patrols
- Three-tier authority model affecting routing speed, not triage quality
- Owner override recognized implicitly from comments
- Staleness thresholds: 14d no update, 30d no story, 21d PR not merged

## Acceptance Criteria

### Operations Guide

- **AC1:** `docs/envoy-operations.md` exists with the following sections:
  - Patrol Cycle Workflow
  - Cross-Agent Communication Protocols
  - Triage Authority Matrix
  - Staleness Detection & Escalation
  - Reporter Communication Guidelines
  - Duplicate Detection Process
  - Direction Alignment Handling

### Patrol Cycle Documentation

- **AC2:** Patrol cycle workflow is documented step-by-step:
  1. Read tracker file and authority tiers
  2. Fetch current state from GitHub (open issues, open PRs, recent merges)
  3. Compare against tracker to detect deltas
  4. Process new issues (acknowledge, triage)
  5. Process merged PRs (cross-check against open issues)
  6. Check staleness thresholds
  7. Update tracker file with patrol watermark
- **AC3:** PR-to-issue linkage detection patterns are documented: `Fixes #N`, `Closes #N`, `Relates to #N`, `Refs #N`, bare `#N`

### Cross-Agent Protocols

- **AC4:** Communication protocols documented for each agent:
  - **Supervisor:** Triage results, scope decisions, stale issue alerts, decline notifications
  - **Merge-queue:** Issue-PR linkage for merge cross-checks
  - **PR-shepherd:** Only when triage-related PR needs rebasing
  - **Workers:** No direct communication (context flows through story files)
- **AC5:** Message templates provided for common notifications (e.g., triage complete, escalation needed, stale issue alert)

### Authority & Triage

- **AC6:** Three-tier routing rules documented in a table format:
  - Tier 1 (Owner): Skip misalignment check, highest priority, never decline direction requests
  - Tier 2 (Contributor): Enhanced priority, lower escalation threshold
  - Tier 3 (Community): Standard triage, full SOUL.md alignment checks
- **AC7:** Owner override process documented (owner comment → reverse decline → reopen triage → notify supervisor)
- **AC8:** Spam handling documented (envoy MAY close + MUST notify supervisor immediately)

### Staleness & Escalation

- **AC9:** Staleness thresholds documented:
  - No envoy update in 14 days → check in and update status
  - No linked story after 30 days → escalate to supervisor
  - PR open but not merged after 21 days → flag to supervisor and pr-shepherd
- **AC10:** Escalation message templates provided for each threshold

### Reporter Communication

- **AC11:** Milestone update templates documented for all 5 stages:
  - Acknowledgment, Triage complete, Story created, PR opened, Fix merged
- **AC12:** Tone guidelines documented: warm professional, grateful, clear, honest about timelines
- **AC13:** Clarifying question guidelines documented: frame helpfully, one round max
