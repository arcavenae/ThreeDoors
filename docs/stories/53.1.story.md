## Story

As a developer on a different machine,
I want clear documentation on how to SSH into a multiclaude session and interact with agents,
So that I can collaborate with the supervisor and monitor agent activity without any code changes.

## Background

multiclaude's architecture already supports remote access via SSH — tmux sessions, CLI commands, and file-based messages all work transparently over SSH. However, there is no documentation explaining how to set this up, what commands are safe to run remotely, or best practices for avoiding interference with running agents.

The feasibility analysis (Tier 1) confirmed that SSH + tmux + CLI is the highest-feasibility, zero-development approach. This story creates the guide that makes it accessible.

## Acceptance Criteria

**Given** a developer with SSH access to the multiclaude host machine
**When** they read the remote collaboration guide
**Then** they can connect to the tmux session in read-only mode and observe agent activity

**Given** the remote collaboration guide
**When** a developer follows the "send a task" instructions
**Then** they can dispatch a worker task via `ssh host 'multiclaude worker create "..."'` and verify it started

**Given** the remote collaboration guide
**When** a developer follows the "message passing" instructions
**Then** they can send a message to the supervisor, poll for replies, and acknowledge messages

**Given** the remote collaboration guide
**When** a developer reads the "safety" section
**Then** they understand which commands are read-only vs state-modifying, and the risks of typing into agent tmux windows

**Given** the remote collaboration guide
**When** a developer reads the "GitHub Issues fallback" section
**Then** they understand how to submit tasks via GitHub Issues when SSH access is unavailable, using structured labels

## Technical Notes

- This is a documentation-only story. No Go code changes.
- Guide lives at `docs/remote-collaboration.md`.
- Reference the CLI command table from the feasibility analysis (Section 4).
- Include SSH command examples for: status check, message send, worker dispatch, tmux attach (read-only), log tailing.
- Document the `remote-task` GitHub Issues convention as a fallback for async/no-SSH scenarios.
- Warn explicitly against typing into agent tmux windows (raw keystrokes corrupt Claude Code input stream).
- Reference `mosh` as a resilience option for unstable connections.
