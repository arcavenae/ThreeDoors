# Story 3.5.3: Config.yaml Schema & Migration Spike

**Status:** done

## User Story

As a developer,
I want a spike on config.yaml schema design and migration path,
So that Epic 7's config-driven provider selection has a validated foundation.

## Acceptance Criteria

1. **Given** the current scattered configuration (hardcoded paths, text files)
   **When** the spike is complete
   **Then** `docs/spikes/config-schema.md` documents: proposed config.yaml schema, provider section design, migration path from current config

2. **Given** an existing user without config.yaml
   **When** they upgrade to the new version
   **Then** zero-friction upgrade verified: defaults to current behavior (text file provider)

3. **Given** the spike deliverables
   **When** reviewed
   **Then** sample config.yaml drafted with commented provider examples

4. **Given** the spike analysis
   **When** reviewed
   **Then** breaking changes identified with mitigation strategies

## Tasks / Subtasks

- [ ] Task 1: Audit current configuration landscape
  - [ ] 1.1: Document all config-related files and their responsibilities
  - [ ] 1.2: Map all YAML sections currently in config.yaml (provider, providers, llm, values, onboarding_complete, calendar)

- [ ] Task 3: Design migration path
  - [ ] 3.1: Document upgrade path from no-config to new schema
  - [ ] 3.2: Document upgrade path from legacy flat config to new schema

  - [ ] 3.4: Identify breaking changes and mitigation strategies

- [ ] Task 4: Create spike document
  - [ ] 4.1: Write `docs/spikes/config-schema.md` with all findings

  - [ ] 5.1: Test that missing config.yaml defaults to textfile provider
  - [ ] 5.2: Test that legacy flat config loads correctly

  - [ ] 5.4: Test that schema version field is handled (future-proofing)

## Dev Notes

### Key Finding: Config Infrastructure Already Exists

The codebase already has extensive config.yaml support:

- `internal/core/provider_config.go` — `ProviderConfig` struct with legacy and multi-provider modes
- `internal/core/registry.go` — `Registry` for adapter factory registration
- `internal/core/config_paths.go` — Centralized path management (`~/.threedoors/`)
- `internal/core/onboarding.go` — `OnboardingConfig` reads/writes to config.yaml
- `internal/core/values_config.go` — `ValuesConfig` reads/writes values section
- `internal/calendar/config.go` — Calendar configuration section
- `internal/intelligence/llm/backend.go` — LLM configuration section
- `cmd/threedoors/main.go` — Bootstrap reads config, resolves providers

### Current Config Sections in config.yaml

| Section | Owner | Status |
|---------|-------|--------|
| `provider` | ProviderConfig | Legacy flat mode |
| `note_title` | ProviderConfig | Legacy Apple Notes |
| `providers` | ProviderConfig | Multi-provider list |
| `llm` | LLM Config | Task decomposition |
| `values` | ValuesConfig | User goals |
| `onboarding_complete` | OnboardingConfig | First-run state |
| `calendar` | CalendarConfig | Calendar sources |

### What This Spike Must Resolve for Epic 7

Epic 7 needs:
- Story 7.1: Adapter registry with runtime discovery (already built!)
- Story 7.2: Config-driven provider selection (already built!)
- Story 7.3: Adapter developer guide & contract tests

The spike must document what exists, identify gaps, and propose the canonical schema.
