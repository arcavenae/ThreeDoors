## Story

As a multiclaude daemon,
I want to handle the case where the outgoing supervisor is unresponsive during handover,
So that the system never gets permanently stuck due to a broken supervisor.

## Background

The nightmare scenario: supervisor so degraded it can't even cooperate with handover. It can't write its delta, can't acknowledge messages, can't signal completion. The system must recover.

**Party mode consensus (Session 5):** Never let a broken supervisor block the system. If it can't hand over gracefully, hand over ungracefully. Some context loss is better than a permanently stuck system.

**Emergency protocol:**
1. Daemon sends HANDOVER_REQUESTED
2. 120-second timeout expires with no response
3. Daemon force-kills outgoing supervisor
4. Daemon spawns incoming with emergency flag
5. Incoming does full worker audit
6. Incoming reports discrepancies to user

## Acceptance Criteria

**AC-1: Timeout detection**
**Given** the orchestrator sent HANDOVER_REQUESTED to the outgoing supervisor
**When** 120 seconds pass with no HANDOVER_COMPLETE response
**Then** the emergency protocol activates

**AC-2: Force-kill**
**Given** the emergency protocol is active
**When** terminating the outgoing supervisor
**Then** the daemon force-kills the supervisor's tmux session and Claude process
**And** logs "EMERGENCY_HANDOVER: outgoing supervisor unresponsive, force-killed after 120s timeout"

**AC-3: Emergency flag on incoming**
**Given** the incoming supervisor is spawned after emergency protocol
**When** constructing the spawn task
**Then** the task includes an emergency flag: "EMERGENCY_SHIFT_HANDOVER: Previous supervisor was unresponsive. Using daemon snapshot only (no supervisor delta). Verify all worker states manually."

**AC-4: Full worker audit**
**Given** the incoming supervisor starts in emergency mode
**When** it runs its startup sequence
**Then** it pings EVERY worker (not just those in the state file)
**And** cross-references worker responses with the daemon snapshot
**And** checks every open PR via `gh pr list`
**And** reconciles the message queue

**AC-5: Discrepancy reporting**
**Given** the incoming supervisor completes its audit
**When** discrepancies are found between snapshot and reality
**Then** it reports them to the user via `multiclaude message send` or logs
**And** lists: missing workers, unknown workers, PR status mismatches

**AC-6: Incoming spawn failure retry**
**Given** the incoming supervisor fails to spawn (API error, tmux issue)
**When** the first spawn attempt fails
**Then** the daemon retries once after 30 seconds
**And** if the second attempt fails, alerts the user and does NOT leave the system without a supervisor

## Technical Notes

- Extends the orchestrator from Story 58.3 (the timeout path that was stubbed)
- Force-kill: `tmux kill-window -t mc-<repo>:supervisor` or `kill -9 <pid>`
- The emergency flag is part of the `--task` parameter in the spawn command
- The incoming supervisor's emergency mode behavior is an extension of Story 58.4's agent definition updates
- User alerting: for v1, write to a file or send a multiclaude message to a known location the user checks
