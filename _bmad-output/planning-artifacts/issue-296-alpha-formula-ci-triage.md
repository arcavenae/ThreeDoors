# Issue #296 Triage: Alpha Homebrew Formula CI Failures

**Date:** 2026-03-09
**Issue:** [#296](https://github.com/arcaven/ThreeDoors/issues/296)
**Severity:** P1 — CI signal degraded on every push to homebrew-tap
**Epic:** 38 (Dual Homebrew Distribution)

---

## Problem Statement

The `threedoors-a.rb` alpha formula template generated by ThreeDoors CI uses `on_arm`/`on_intel`/`on_linux` blocks with `url`/`sha256`, which Homebrew's DSL explicitly disallows. This causes `brew audit --strict` to fail with 7 offenses on every push to `arcaven/homebrew-tap`. The stable formula (`threedoors.rb`) was already fixed to use the correct `if Hardware::CPU.arm?` pattern, but the alpha formula template was not updated to match.

## Options Evaluated

### Option A: Fix the alpha formula template in ci.yml to match stable pattern (ADOPTED)

Replace `on_arm`/`on_intel`/`on_linux` blocks in the alpha formula template (in ThreeDoors `ci.yml`) with `if OS.mac? && Hardware::CPU.arm?` / `elsif OS.mac?` / `elsif OS.linux?` conditionals.

**Rationale:**
- Directly fixes the root cause — the template in ci.yml generates the formula pushed to homebrew-tap
- Minimal change — only the formula template glob in ci.yml needs updating
- Proven pattern — the stable formula already uses this exact approach and passes CI
- The alpha formula includes Linux support, so needs `OS.mac?` guards (unlike the macOS-only stable formula)
- Next push to main will auto-generate a fixed formula and push it to homebrew-tap

### Option B: Add Linux conditional support with OS.mac? guards (INCORPORATED into A)

Use `if OS.mac? && Hardware::CPU.arm?` instead of just `if Hardware::CPU.arm?` to properly support the three-way OS+arch split the alpha formula needs (macOS ARM, macOS Intel, Linux AMD64).

**Rationale:** This is not a separate option — it's inherent to fixing the alpha formula correctly. The stable formula is macOS-only so it only needs `Hardware::CPU.arm?`, but the alpha formula includes Linux binaries and must use `OS.mac?` guards. Incorporated into Option A.

### Option C: Extract a shared template function (REJECTED)

Create a shared Ruby function or heredoc template that both stable and alpha formulas use.

**Why rejected:**
- GoReleaser controls the stable formula — ThreeDoors CI doesn't generate `threedoors.rb` (D-066)
- The two formulas have fundamentally different structures (GoReleaser-managed vs CI-generated)
- Prior decision D-028/X-028 explicitly rejected shared formula templates
- Adding abstraction for two instances violates YAGNI
- A little copying is better than a little dependency (Go Proverb, applies here too)

### Option D: Fix directly in homebrew-tap (REJECTED)

Edit `threedoors-a.rb` directly in the homebrew-tap repository.

**Why rejected:**
- The formula gets overwritten on every push to main in ThreeDoors
- Would be a temporary fix until the next CI push overwrites it
- Root cause is in ThreeDoors ci.yml, not in homebrew-tap

## Decisions Summary

| ID | Decision | Rationale |
|----|----------|-----------|
| Adopted | Fix alpha formula template in ci.yml using `if OS.mac? && Hardware::CPU.arm?` pattern | Matches stable formula fix; minimal change; addresses root cause |
| Rejected | Shared template function | GoReleaser controls stable; prior decision X-028; YAGNI |
| Rejected | Direct fix in homebrew-tap | Would be overwritten on next CI push |

## Impact

- Unblocks CI signal for `arcaven/homebrew-tap` — currently every push fails
- No functional change to formula behavior — `brew install arcaven/tap/threedoors-a` likely already works despite style violations
- Implementation is small (~20 lines of template change in ci.yml)
