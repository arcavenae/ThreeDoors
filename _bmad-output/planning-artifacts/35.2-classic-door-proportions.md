# Story 35.2 Planning Artifact — Classic Theme Door-Like Proportions

## Adopted Approach: Row-by-row rendering using DoorAnatomy

**Rationale:** Use the `DoorAnatomy` model from Story 35.1 to calculate structural row positions, then render the Classic theme as a proper door frame with:
- Rounded border top/bottom (existing Classic style)
- Panel divider at `PanelDivider` row using `├─────────────┤`
- Doorknob `●` at `HandleRow` on the right side
- Threshold line `▔▔▔▔▔` as the final row below the bottom border
- Content placed starting at `ContentStart`
- Compact fallback (height=0 or height < MinHeight): use existing Lipgloss-based card rendering

**Implementation:** Extract a `classicRender` function (like Modern's `modernRender`), build output row-by-row using `strings.Builder` and `fmt.Fprintf`. Each row is one of: top border, content, panel divider, handle row, blank, bottom border, threshold.

**Testing:** Add height-aware golden tests (h16, h24, selected/unselected). Update existing golden tests to use height=0 so compact mode tests remain valid.

## Rejected Options

### Option A: Lipgloss-only approach (extend existing style)
Use Lipgloss `Border`, `Padding`, and styled content blocks to approximate door proportions.
**Rejected because:** Lipgloss borders don't support mid-door dividers, handle placement at specific rows, or threshold lines. Would require significant hacks with nested styles that would be fragile and hard to test.

### Option B: Template-based rendering
Define door layout as a multi-line string template with placeholders.
**Rejected because:** Templates are rigid — they don't adapt well to variable heights and content wrapping. The row-by-row builder approach (as used by Modern/Sci-Fi/Shoji themes) is the established pattern in this codebase.

### Option C: Render content in Lipgloss, then post-process lines
Render using existing Lipgloss card style, then string-replace specific lines.
**Rejected because:** Fragile — depends on exact ANSI escape sequences in Lipgloss output. Any Lipgloss version change could break it. Direct rendering is more reliable.
