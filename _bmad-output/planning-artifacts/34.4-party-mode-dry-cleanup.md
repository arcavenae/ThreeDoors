# Party Mode Artifact: Story 34.4 — Retroactive Story DRY & Spec Alignment

## Adopted Approach: Batch Cleanup by Violation Category

**Rationale:** Process all story files in passes by violation type (pre-PR checklists, Quality Gates, coding standards, design patterns, testing standards). This is most efficient because each violation type has a recognizable pattern that can be systematically found and replaced.

### Replacement Patterns

1. **Pre-PR Submission Checklists** → Remove entirely. These are fully covered by `/pre-pr` and CLAUDE.md.
2. **Quality Gate items duplicating CLAUDE.md** → Replace with "All standard checks pass (see CLAUDE.md)" + keep only story-specific QG items.
3. **Coding standards reminders** → Remove. Cross-reference CLAUDE.md only if context is needed.
4. **Design pattern reminders** → Remove. CLAUDE.md is the single source.
5. **Testing standards reminders** → Remove. CLAUDE.md covers all testing standards.

### Scope Guard

- Never remove story-specific content (unique ACs, architecture decisions, file lists)
- Never change story status or metadata
- Preserve any story-specific QG items (e.g., "golden file tests regenerated")
- Leave Epic 0 stories as-is (they're retroactive backfills with minimal structure)

## Rejected Options

### Option A: Per-File Manual Review
**Why rejected:** With 60+ files, manual per-file review is too slow and error-prone. The violation patterns are consistent enough for systematic batch processing.

### Option B: Automated sed/awk Script
**Why rejected:** Story files have enough structural variation that a regex-based script would risk removing story-specific content. Human-guided batch processing with pattern matching is safer.

### Option C: Only Fix New Stories Going Forward
**Why rejected:** NFR-DX6 requires specs to be living documentation. Leaving 60+ files with duplicated content undermines the DRY principle and makes the codebase inconsistent. The cleanup is necessary to establish the new standard retroactively.

### Option D: Full Spec-vs-Code Alignment Audit
**Why rejected:** While NFR-DX6 calls for specs reflecting code reality, a full deep audit of every completed story against actual code is massive scope. Story 34.4 establishes the DRY pattern and removes obvious duplication. Deep per-story alignment is an ongoing practice, not a one-time batch job.
