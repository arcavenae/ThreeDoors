# Story 35.6: Golden File Test Regeneration and Accessibility Validation — Planning Artifact

**Date:** 2026-03-08
**Story:** 35.6

## Adopted Approach

**Unified cross-theme golden file test + accessibility validation suite**

Add a single new test file `internal/tui/themes/golden_door_test.go` that:

1. **Cross-theme golden files (AC1-AC2):** Table-driven tests across all 4 themes × 3 heights (min, 16, 24) × 3 widths (min, 80, 120) × 2 states (selected, unselected) = 72 golden files. Uses existing `golden.RequireEqual` pattern from charmbracelet/x/exp/golden.

2. **Monochrome accessibility (AC3):** Renders in ASCII color profile and verifies structural door signifiers (panel divider chars, handle chars, threshold chars) are present via string matching — no color dependency.

3. **Compact fallback (AC4-AC5):** Verifies height=0 and height < MinHeight render without door elements, and that task content text is present in output.

4. **Screen reader text extraction (AC6):** Strips all non-letter/non-digit/non-space characters and verifies task text appears in reading order without decorative interference.

**Rationale:** Centralizing cross-theme door tests in one file avoids duplication across 4 separate test files. Individual theme tests already cover theme-specific proportions; this story is about cross-cutting coverage.

## Rejected Options

### Option A: Extend each per-theme door test file
- Would add width variants to classic_door_test.go, modern_door_test.go, etc.
- **Rejected because:** Duplicates the same test matrix 4 times; harder to maintain; the existing per-theme tests already cover structural behavior. Story 35.6 is specifically about cross-theme validation.

### Option B: Property-based/fuzz testing for golden files
- Generate random widths/heights and verify invariants (consistent width, structural chars present)
- **Rejected because:** Golden files are snapshot-based by design — they verify exact visual regression. Property-based tests complement but don't replace golden file snapshots. The existing `TestVisualWidthConsistency` already covers property-style checks.

### Option C: Separate accessibility test file
- Create `accessibility_test.go` separate from golden file tests
- **Rejected because:** The accessibility tests are closely related to the golden file generation (both render at same dimensions). Keeping them together in one file reduces context-switching and makes the test matrix clear.
