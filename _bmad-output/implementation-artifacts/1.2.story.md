# Story 1.2: Display Three Doors from a Task File

Status: ready-for-dev

## Story

**As a** developer,
**I want** the application to read tasks from a text file and display three of them as "doors",
**so that** I can see the core interface of the application.

## Acceptance Criteria

1. On startup, the application reads tasks from `~/.threedoors/tasks.txt`.
2. If `tasks.txt` does not exist, it is created with a default set of 3-5 sample tasks.
3. A `FileManager` component is created in `internal/tasks/file_manager.go` to handle file reading and creation.
4. A `Task` model is defined in `internal/tasks/task.go` to represent a task (for now, just text content).
5. The main application view (`internal/tui/main_model.go`) is updated to display three randomly selected tasks from the file.
6. The display shows three randomly selected tasks, distributed horizontally from left to right, without "Door X" labels.
7. The application responds to 'a' or 'left arrow' for the left door, 'w' or 'up arrow' for the middle/center door, and 'd' or 'right arrow' for the right door.
8. Pressing 's' or 'down arrow' re-rolls the doors, presenting a new set of three tasks.
9. Initially, or after re-rolling, no door is selected/active.
10. The application responds to the following keystrokes for task management (functionality to be implemented in future stories):
    - 'c': Mark selected task as complete.
    - 'b': Mark selected task as blocked.
    - 'i': Mark selected task as in progress.
    - 'e': Expand selected task (into more tasks).
    - 'f': Fork selected task (clone/split).
    - 'p': Procrastinate/avoid selected task.
11. The application still quits on 'q' or 'ctrl+c'.

### Data Models & Persistence

**For Story 1.2, use SIMPLIFIED Task model** — NOT the full architecture model:
```go
// internal/tasks/task.go
type Task struct {
    Text string
}
```

The full Task model with UUID, status, notes, timestamps, YAML storage is for future stories (1.3+). Story 1.2 uses plain text file with one task per line. [Source: `docs/prd/epic-details.md`, Story 1.2 AC#4]

**File format for Story 1.2:**
- `~/.threedoors/tasks.txt` — plain text, UTF-8 encoded, one task per line
- NOT `tasks.yaml` (that's the future format from architecture docs)
- The `FileManager` reads lines from the text file, returns `[]Task`
- Lines are trimmed of leading/trailing whitespace
- Empty lines and blank lines are skipped
- Lines starting with `#` are treated as comments and skipped
- No maximum line length enforced (task text is displayed truncated in the door UI if too long)

**Default sample tasks (when file doesn't exist):**
```

Fix login page bug
Refactor database queries
Write unit tests for API
```

### TaskLoader Interface & FileManager Design

**TaskLoader interface (defined in `internal/tasks/` package):**
```go
// internal/tasks/task_loader.go (or in file_manager.go)

// TaskLoader defines the interface for loading tasks.
// FileManager implements this. Tests can use stubs.
type TaskLoader interface {
    LoadTasks() ([]Task, error)
}
```

**FileManager constructor — accepts configurable base directory for testability:**
```go
// NewFileManager creates a FileManager with the given base directory.
// Use "~/.threedoors" for production, t.TempDir() for tests.
// FileManager implements the TaskLoader interface.
func NewFileManager(baseDir string) *FileManager
```

**Key methods:**
- `LoadTasks() ([]Task, error)` — reads tasks.txt, creates dir/file with defaults if missing
- Uses atomic write pattern for default file creation (write .tmp → sync → rename)

// SelectRandomDoors picks up to `count` random tasks from allTasks.

### Bubbletea Model Specification

**Model struct (white-box — tests in `package tui` have access to unexported fields):**
```go
type Model struct {
    quitting    bool
    tasks       []tasks.Task  // all loaded tasks
    doors       []tasks.Task  // currently displayed doors (up to 3)
    selectedIdx int           // -1 = none selected, 0-2 = door index
    width       int           // terminal width (default 80)
    height      int           // terminal height (default 24)
    err         error         // non-nil if task loading failed
    taskLoader  tasks.TaskLoader
}
```

**Constructors:**
```go
// NewModel creates Model with default FileManager (~/.threedoors).
// Used by cmd/threedoors/main.go. NO change to main.go signature needed.
func NewModel() Model

// NewModelWithTasks creates Model pre-loaded with tasks (for testing).
// Bypasses FileManager entirely. Sets selectedIdx = -1, default dimensions.
func NewModelWithTasks(taskList []tasks.Task) Model
```

**`NewModel()` internally creates its own FileManager:**
```go
func NewModel() Model {
    homeDir, _ := os.UserHomeDir()
    baseDir := filepath.Join(homeDir, ".threedoors")
    return Model{
        selectedIdx: -1,
        width:       80,
        height:      24,
        taskLoader:  tasks.NewFileManager(baseDir),
    }
}
```

### Bubbletea Message Types (MVU Pattern)

**Custom message types:**

```go
// tasksLoadedMsg is sent when tasks are successfully loaded from file
type tasksLoadedMsg struct {
    tasks []tasks.Task
}

// tasksLoadErrorMsg is sent when task loading fails
type tasksLoadErrorMsg struct {
    err error
}
```

**Init() → tea.Cmd flow:**
```go
func (m Model) Init() tea.Cmd {
    return func() tea.Msg {
        loadedTasks, err := m.taskLoader.LoadTasks()
        if err != nil {
            return tasksLoadErrorMsg{err: err}
        }
        return tasksLoadedMsg{tasks: loadedTasks}
    }
}
```

**Update() must handle these messages:**
- `tasksLoadedMsg`: Store tasks in model, call `SelectRandomDoors` for initial 3 doors
- `tasksLoadErrorMsg`: Store error in model for display in View()
- `tea.WindowSizeMsg`: Store terminal width/height for responsive layout

### Technical Requirements

**Tech Stack** [Source: `docs/architecture/tech-stack.md`]:
- Go 1.25.0 (as in go.mod)
- `charmbracelet/bubbletea` v1.2.4
- `charmbracelet/lipgloss` v1.0.0
- No new dependencies needed for this story

**Door rendering with Lipgloss (v1.0.0 API):**
- Use `lipgloss.NewStyle()` constructor (v1.0.0 API)
- Each door occupies ~1/3 terminal width
- Use `lipgloss.JoinHorizontal(lipgloss.Top, door1, door2, door3)` to arrange doors side by side
- **Default terminal dimensions** before first `tea.WindowSizeMsg`: assume 80 columns, 24 rows as fallback
- **Minimum usable width**: 60 columns. Below this, doors still render but may be narrow
- Door box styling:
  - **Unselected door**: `lipgloss.RoundedBorder()`, border color `"240"` (gray), padding 1
  - **Selected door**: `lipgloss.RoundedBorder()`, border color `"212"` (bright magenta/pink), bold border, padding 1
  - Task text inside door: centered, truncated with "..." if exceeds box width minus padding
- No door initially selected (all gray borders)

**Error display in View():**
- If `tasksLoadErrorMsg` was received, View() shows: `"Error loading tasks: <error message>\n\nPress q to quit\n"`
- Normal view only shown when tasks loaded successfully

**Bubbletea patterns to follow:**
- Use `tea.WindowSizeMsg` to handle terminal resizing — store `width` and `height` in Model
- `Init()` returns a `tea.Cmd` that loads tasks (NOT nil like Story 1.1)
- Model returns `(tea.Model, tea.Cmd)` from Update — never mutate in place

### Edge Cases

1. **Fewer than 3 tasks**: If tasks.txt has only 1 or 2 tasks, display that many doors (1 or 2), centered horizontally. Do NOT show empty placeholder doors.
2. **Zero tasks / empty file**: Display message: `"No tasks found. Add tasks to ~/.threedoors/tasks.txt\n\nPress q to quit\n"`
3. **Re-roll with 3 or fewer total tasks**: Re-roll still works but may show the same tasks (since there are no alternatives). This is acceptable.
4. **Re-roll duplicate avoidance**: When more than 3 tasks exist, re-roll SHOULD try to show different tasks than currently displayed (best effort, not guaranteed).
5. **Task management keys ('c','b','i','e','f','p') with no door selected**: Silently ignore. No error message, no hint. These are no-ops in this story anyway.
